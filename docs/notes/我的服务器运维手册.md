### 降配判断

停用 8091 后，机器可以降配，但不建议降到 1 GiB 内存。

| 规格 | 判断 |
|------|------|
| 2C / 1G / 30G | 不建议。Docker + NewAPI + Postgres + Redis + adapters 余量太小，容易 OOM；30G 盘也刚好贴近当前占用。 |
| 2C / 2G / 40G | 最低可用，适合极限省钱；需要接受偶发流量时余量偏紧。 |
| 2C / 4G / 50G | 推荐。价格仍低，内存和磁盘余量更稳。 |
| 4C / 8G / 70G | 当前规格，停用 8091 后偏富余。 |

迁移前必须先备份数据库和配置。不要直接重装旧服务器。

### 查询渠道状态

```bash
# 所有渠道概览
docker exec $(docker ps -q -f name=postgres) psql -U newapi -d new-api -c "SELECT id, name, status, type, base_url, \"group\", substring(models, 1, 200) AS models_preview FROM channels ORDER BY id;"
```

status: `1`=启用, `2`=禁用, `3`=自动禁用

```bash
# 查单个渠道完整配置（替换 id）
docker exec $(docker ps -q -f name=postgres) psql -U newapi -d new-api -c "SELECT id, name, status, auto_ban, model_mapping, setting FROM channels WHERE id = 16;"
```

```bash
# 查模型属于哪些渠道（替换模型名）
docker exec $(docker ps -q -f name=postgres) psql -U newapi -d new-api -c "SELECT id, name, status FROM channels WHERE models LIKE '%gpt-image%' ORDER BY id;"
```

### 渠道内部配置说明

`setting` 是 JSON 字段，关键字段：

| 字段 | 说明 |
|------|------|
| `auto_ban` | `1`=错误率高时 NewAPI 自动禁用该渠道 |
| `pass_through_body_enabled` | 是否透传请求体给上游 |
| `proxy` | 代理地址（空=直连） |
| `model_mapping` | 模型名映射（空=透传，不重命名） |

```bash
# 查某渠道最近任务（替换 channel_id）
```


### 迁移前完整备份

迁移逻辑：旧服务器导出数据库和配置 → 复制到新服务器 → 新服务器恢复 → 测试通过后切域名。不要把“新服务器”理解成自动继承旧服务器；两台机器是独立电脑，必须手动搬关键数据。

```bash
ls -lh /root/jiucai-migration-backup-*.tar.gz
```

### Web 端 CORS 最小影响修复（优先用 Nginx，不重启 NewAPI）

适用场景：`https://jiucaihezi.com` 的静态 Web 端调用 `https://api.jiucaihezi.studio/v1/chat/completions` 报 `Failed to fetch`，且 `OPTIONS` 预检缺少 `Access-Control-Allow-Origin`。

优先策略：

```text
先用 Nginx 层补 CORS，然后 nginx reload。
不要为了单纯 CORS 先替换 NewAPI 二进制。
```

原因：

- Nginx reload 基本不中断现有服务。
- NewAPI 是核心网关，替换或重启会短暂影响后台、聊天、创作接口。
- 这只是 Web 跨域授权问题，不需要先动数据库或渠道配置。

操作流程：

```bash
ssh root@47.82.86.196

# 1. 备份当前 Nginx 配置
mkdir -p /etc/nginx/backups
cp /etc/nginx/sites-enabled/api.jiucaihezi.studio.conf \
  /etc/nginx/backups/api.jiucaihezi.studio.conf.$(date +%Y%m%d-%H%M%S)

# 2. 编辑 api.jiucaihezi.studio 的 /v1/ 或 /v1/chat/completions 代理 location
nano /etc/nginx/sites-enabled/api.jiucaihezi.studio.conf
```

在代理到 NewAPI 的 `/v1/` location 内加入以下 CORS 配置；如果配置里有精确的 `/v1/chat/completions` location，也要加到那个 location 内：

```nginx
set $jc_cors_origin "";
if ($http_origin ~* "^https://(www\.)?jiucaihezi\.com$") {
    set $jc_cors_origin $http_origin;
}

add_header Access-Control-Allow-Origin $jc_cors_origin always;
add_header Access-Control-Allow-Credentials "true" always;
add_header Access-Control-Allow-Methods "GET, POST, PUT, DELETE, OPTIONS" always;
add_header Access-Control-Allow-Headers "Authorization, Content-Type, x-api-key, anthropic-version, anthropic-beta" always;
add_header Vary "Origin" always;

if ($request_method = OPTIONS) {
    return 204;
}
```

保存后验证并重载：

```bash
nginx -t && nginx -s reload
```

验证 CORS：

```bash
curl -i -X OPTIONS https://api.jiucaihezi.studio/v1/chat/completions \
  -H 'Origin: https://jiucaihezi.com' \
  -H 'Access-Control-Request-Method: POST' \
  -H 'Access-Control-Request-Headers: authorization,content-type,x-api-key'
```

应看到：

```text
HTTP/2 204
Access-Control-Allow-Origin: https://jiucaihezi.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE, OPTIONS
Access-Control-Allow-Headers: Authorization, Content-Type, x-api-key, anthropic-version, anthropic-beta
```

重要备注：

```text
Nginx CORS 是最小影响线上修复，不代表以后不升级 NewAPI。
后续 NewAPI 官方版本升级、通道适配升级、或需要把源码改动合入网关时，仍然要走 NewAPI 正式升级/重启流程。
```

### NewAPI 生产热修红线（2026-06-10 事故复盘）

NewAPI 是 `api.jiucaihezi.studio` 的核心网关：聊天、创作面板、管理后台、支付相关路由都会受影响。生产环境不得把 NewAPI 当普通辅助服务在线试错。

已确认的当前事实：

- `/root/new-api-new/` 才是当前服务器 NewAPI 运行目录 / 源码目录。
- `/root/new-api-new/docker-compose.yml` 的 `new-api` 服务使用 `calciumion/new-api:latest` 镜像，当前没有 `build:` 配置；执行 `docker compose build new-api` 会显示 `No services to build`，不会编译源码。
- 服务器默认没有 `go` 命令；源码编译需要额外 Go 环境。
- 当前源码 `main.go` 同时 embed `web/default/dist` 与 `web/classic/dist`；若要从源码编译，必须先构建这两套前端 dist。
- 生产热修失败时，不要反复 `docker cp` 或 `docker restart`；先用上面的 `--force-recreate new-api` 回滚到镜像版本。

禁止事项：

```text
禁止在没有新二进制校验的情况下 docker cp /new-api。
禁止在生产服务器上边试边编译 NewAPI。
禁止把 /opt/new-api 默认当作当前源码根目录。
禁止在没有回滚命令和验证窗口时替换 new-api 容器二进制。
```

推荐流程：

```text
1. 先保证 production 正常运行。
2. 本地或 staging 编译并测试 NewAPI 二进制。
3. 确认可回滚：docker compose up -d --force-recreate new-api。
4. 低峰期上传已验证二进制。
5. 替换后 10 秒内验证 /login、/v1/models、核心创作接口。
6. 失败立即 force-recreate 回官方镜像。
```

## 5. 已知问题 & 解决方案

### NewAPI `progress` 字段解析 bug（历史问题）

**现象**: Seedance 视频提交返回 500，日志报 `cannot unmarshal object into Go struct field responseTask.progress of type int`

**原因**: Seedance 上游 `progress` 返回 `{"message": "..."}` 对象，NewAPI 期望 int

**当前策略**: 创作面板主链路不再依赖 Seedance 直连代理；模型是否展示和是否允许执行由 `/api/creation/models` 结合 NewAPI 渠道状态决定。`/api/seedance/` 保留为历史/备用代理，若后续重新启用 Seedance，优先在 NewAPI/RH 适配链路里闭环。

### NewAPI 模型名自动转换

**现象**: 请求 `veo3.1-fast` 时，NewAPI 内部转换为 `veo_3_1-fast`（点号变下划线），导致找不到渠道

**解决**: 渠道 models 列表中两种写法都要加上，或用模型重定向

### 渠道自动禁用

**现象**: 渠道连续报错后被自动禁用 (status=3)

**解决**: 先查看 NewAPI 渠道错误日志和上游余额/密钥，再执行 `UPDATE channels SET status = 1 WHERE id = <id>;`。创作面板会通过 `/api/creation/models` 把自动禁用的模型标记为不可用。

### 创作面板点击后转圈消失

**现象**: 用户点击生成，任务短暂转圈后没有结果，也没有明确失败反馈。

**排查顺序**:

```bash
# 1. 看创作可用性服务是否正常
curl https://api.jiucaihezi.studio/api/creation/models

# 2. 看 NewAPI 是否返回模型
curl -H "Authorization: Bearer <用户 NewAPI Token>" \
  https://api.jiucaihezi.studio/v1/models

# 3. 看 rh-adapter / creation-models 日志
cd /opt/rh-adapter && docker compose logs -n 100 rh-adapter
journalctl -u creation-models -n 100 --no-pager

# 4. 看 NewAPI 渠道状态
docker exec $(docker ps -q -f name=postgres) psql -U newapi -d new-api -c 'SELECT id,name,status,models FROM channels ORDER BY id;'
```

**当前前端行为**: 不可用模型应在创作面板显示不可用状态；运行中任务如果失败，会回写创作画廊失败卡片，避免“无声失败”。

## 7. 分组 & 计费

- **AutoGroup**: `["0.5","1","1.2","1.5","2"]` — Token group 为空时自动匹配
- **GroupRatio**: `{"0.5":0.5, "1":1, "1.2":1.2, "1.5":1.5, "2":2, "9":9, "川普特供":8}`
- 计费公式: `费用 = 分组倍率 × 模型倍率 × (输入tokens × 基准单价 + 输出tokens × 基准单价 × 补全倍率)`
- 模型如无 `ModelRatio` 或 `ModelPrice` 配置，`/v1/models` 接口不会返回该模型

## 9. RunningHub 适配器 (rh-adapter) — 2026-06-03 更新

### 架构

```
提交计费:
Studio → NewAPI /v1/* → http://rh-adapter:8789 → RH 原生 API

轮询:
Studio → Nginx /rh/tasks/{task_id} → http://172.17.0.1:8789/tasks/{task_id}
```

适配器是 Python FastAPI Docker 服务，将标准 OpenAI 格式请求翻译为 RunningHub 原生 API 调用。POST 提交走 NewAPI 渠道并产生一次计费；GET 轮询走 Nginx `/rh/tasks/` 直连 adapter，避免重复计费。NewAPI 负责用户鉴权和计费，rh-adapter 只在 Docker/内网可达，不额外增加 adapter 鉴权。

### 文件位置

| 部署配置产物 | `scripts/rh-deploy/` |
| 日志 | `docker compose logs -f rh-adapter` |

### 环境变量

```

```

不要在 adapter 上新增独立密钥。隔离方式是 Docker/内网绑定 + Nginx 只公开 `/rh/tasks/` 轮询路径。
