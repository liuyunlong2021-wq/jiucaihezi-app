# Autoresearch 能力复现清单

> 参考源：Karpathy `autoresearch` 仓库，尤其是 `program.md` 中的文件系统工作流、实验循环、日志和评估约束。

## 一句话结论

`autoresearch` 的核心不是“自动研究”四个字，而是把 agent 的工作空间压缩成一套可审计的文件协议：固定说明书、固定数据入口、固定产物、固定评估、固定日志。韭菜盒子的知识库要复现的不是训练脚本本身，而是这套“raw → structured working memory → measurable result → log”的闭环。

## 能力拆解

| 编号 | Autoresearch 能力 | 韭菜盒子对应能力 | 当前状态 | 复现要求 |
| --- | --- | --- | --- | --- |
| AR-01 | 固定工作目录协议 | 每个知识库固定 `CLAUDE.md / raw / wiki / _reports` | 部分复现 | 创建、导入、整理时自动补齐目录 |
| AR-02 | `program.md` 作为任务说明书 | `CLAUDE.md` 作为知识库整理和检索说明书 | 已复现基础 | 后续整理必须读取它 |
| AR-03 | 原始数据只读入口 | `raw/` 存对话记录、上传资料、用户自建素材 | 部分复现 | 对话进入 `raw/对话记录`；用户空白处新建文件夹默认进入 raw |
| AR-04 | 唯一结构化工作面 | `wiki/` 是模型检索的唯一结构化知识区 | 部分复现 | 检索优先 wiki，raw 只作未整理原料 |
| AR-05 | 实验/整理有固定循环 | “整理”按钮执行 raw → wiki 编译 | 部分复现 | 从第二列知识库右键可直接整理 |
| AR-06 | 结果可度量 | 整理报告写明 raw 数、创建数、更新数、跳过数 | 本轮复现 | 每次整理生成报告 |
| AR-07 | 日志可追溯 | `_reports/整理记录` 保存每次整理过程 | 本轮复现 | 报告文件留在知识库树中 |
| AR-08 | 失败不污染主结果 | JSON 解析失败或 API 失败时降级，不乱写 wiki | 部分复现 | 保留 fallback，并在报告中说明 |
| AR-09 | 小步迭代 | 每轮只整理未 indexed 的 raw | 已复现基础 | 保持 raw 标记，不重复整理 |
| AR-10 | 自检/体检 | 知识索引、知识体检、冲突报告 | 部分复现 | 后续补冲突检测和新增栏目确认 |
| AR-11 | 可迁移工作空间 | `.jcvault` 导入/导出知识库 | 本轮先复现导入 | 后续补导出知识库 |
| AR-12 | 人类可理解文件树 | 第二列知识库显示真实层级树 | 部分复现 | 文件夹打开必须按 `folderId` 展示子项 |

## 本轮复现范围

本轮先复现底座能力，保证后续的智能整理可以稳定跑：

1. 第二列知识库空白右键菜单增加：
   - `新建文件夹`
   - `导入知识库`
2. 新建文件夹规则：
   - 在某个知识库内空白处新建时，默认创建到 `raw/` 下。
   - 在已有文件夹内新建时，创建为当前文件夹子文件夹。
3. 导入知识库规则：
   - 支持 `.jcvault` 知识库包。
   - 兼容 Web 备份 `.jcbackup` 中的知识库和知识文件导入，但不导入 API Key、主题、字号等设置。
4. 整理报告规则：
   - 每次 raw → wiki 整理后，写入 `_reports/整理记录/整理报告_时间.md`。
   - 报告记录读取 raw 数、创建 wiki 项、更新 wiki 项。
5. 自检：
   - 新增工具测试覆盖 `.jcvault` 解析、Web 备份知识库抽取、冲突 ID 重映射。
   - 构建通过后，再按本清单标记本轮完成项。

## 后续复现顺序

1. `整理预览层`：raw → 候选知识 → 用户确认 → wiki。
2. `新增知识栏目建议`：发现重复主题时建议新增 wiki 子目录。
3. `冲突报告`：同一角色/设定/文风出现互斥信息时写入 `_reports/冲突报告`。
4. `知识库导出`：生成 `.jcvault`，供 Web 端导入。
5. `检索体检`：模拟用户问题，检查 wiki 命中率和上下文预算。
