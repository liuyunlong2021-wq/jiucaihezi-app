# 本地能力与长文输出 SDD 设计

## 目标

让韭菜盒子形成清晰的三足能力模型：

- 知识库负责上下文：回答“依据什么资料”。
- Skill负责指挥：回答“按什么专业方法干活”。
- 本地能力负责手脚：回答“如何读文件、写文件、格式转换、导出结果”。

用户仍然只需要理解“选Skill、选知识库、直接提要求”。本地能力默认开启，作为底层能力存在，不再要求用户理解 Gateway、Provider 或复杂工具配置。

## 对话组合

桌面端支持 8 种组合，工具不再强制绑定Skill：

| 组合 | 产品行为 |
| --- | --- |
| 直接聊 | 普通对话和轻写作。 |
| 直接聊 + 本地能力 | 默认工具执行器处理文件读取、格式转换、Office 导出。 |
| 知识库 | 基于绑定知识库回答，不调用工具。 |
| 知识库 + 本地能力 | 基于资料生成内容，并可导出文件。 |
| Skill | Skill按专业方法回答。 |
| Skill + 本地能力 | Skill调度工具完成 Office、文件、浏览器等任务。 |
| Skill + 知识库 | Skill基于资料写报告、总结、分析。 |
| Skill + 知识库 + 本地能力 | 完整链路：资料、指挥、执行三者协作。 |

## 工具安全边界

本地能力默认开启，但工具按风险分层：

- safe：自动可用，如文档读取、知识图谱查询。
- write：可生成文件，如 Word/PDF/PPT/Excel 导出。
- approval：需要确认或后续审批，如浏览器自动操作、命令执行、定时任务。

第一阶段实现只改变“是否向模型暴露工具”的策略，不自动执行未注册的危险本地命令。已有 `executeToolCall` 仍是最终执行边界。

## 默认工具执行器

用户未选择Skill时，如果本地能力开启，系统内部使用隐藏的默认工具执行器：

- 不改变 UI，不显示一个新Skill。
- 给模型开放安全和办公类工具。
- 消息仍归属普通助手。
- 工具仓库继续显示哪些工具被调用。

用户选择Skill时，工具由Skill调度；未选择Skill时，工具由默认执行器调度。

## 长文输出链路

长文能力不是“单次无限 token”，而是用户感知上的无限长：

1. 理解任务：判断是否为长文、报告、方案、脚本或文件生成。
2. 搭结构：先形成大纲、章节目标、输出格式。
3. 分段生成：按章节生成，不靠一次性硬撑。
4. 上下文账本：维护已写摘要、术语、风格、未完成章节。
5. 断点保存：流式内容持续保留，断线可继续写。
6. 校验重组：检查覆盖、重复、跑题和格式。
7. 导出：由工具生成 Markdown、Word、PDF、PPT、Excel、SRT。

第一阶段不实现完整 LongForm Engine，只为工具策略与后续长文引擎预留稳定入口。

## 第一阶段实现范围

- 增加“本地能力”开关，默认开启。
- 工具策略从“必须有Skill”改为“本地能力开启即可用工具”。
- 没选Skill时也能调用 Office/文档类工具。
- 工具仓库展示本地能力状态。
- 保留设置页简洁，不恢复 Gateway/Provider 面板。

## 非目标

- 不改 App 图标。
- 不恢复复杂 Gateway 设置。
- 不让用户直接配置工具白名单。
- 不一次性实现完整 2 万字长文引擎。
- 不改变知识库选择规则：选了才用，不选不用。
