# Knowledge Vault 2.0 Design

## Goal

Knowledge Vault 2.0 turns uploaded material into a real local wiki, not just a folder placeholder. The desktop app preserves original material, converts every supported source into Markdown, asks the model to propose a domain-specific wiki structure, then creates usable Markdown wiki pages that can be retrieved, cited, edited, checked, and improved over time.

Core product rule stays unchanged:

- Users still choose a `搭子` and a `知识库` independently.
- If no `知识库` is selected, conversation content is not written to `raw/`.
- If a `知识库` is selected, conversation content enters that vault's `raw/对话记录/`.
- `搭子` decides how to work, `知识库` supplies facts and context, local tools execute actions, and the LLM reasons and writes.

## Design Principles

1. `raw/` is the preserved source layer. Original files and converted Markdown stay there and are never destructively rewritten.
2. `wiki/` is the retrieval layer. It contains concise, structured Markdown pages designed for LLM lookup and human editing.
3. `CLAUDE.md` is the vault rulebook. It is generated by local code from metadata and folder rules, not returned as a large model-generated JSON string.
4. The model proposes architecture and content units. The app writes real files.
5. Strongly structured sources, such as toolbooks, can generate a rich first wiki immediately.
6. Weakly structured sources, such as chat logs and mixed notes, start with a skeleton plus high-confidence pages and improve through use.
7. Every organize pass writes a report so the app does not repeatedly re-read the same raw material without progress.

## Target File Tree

```text
知识库名/
  CLAUDE.md
  raw/
    对话记录/
    上传资料/
    原始文件/
    转换后的MD/
  wiki/
    index.md
    overview.md
    hot.md
    log.md
    meta/
      dashboard.md
      health.md
    <domain folders>/
      <wiki pages>.md
  _reports/
    整理记录/
    健康检查/
    冲突报告/
    覆盖率报告/
  _templates/
    entity.md
    concept.md
    source.md
    question.md
```

Domain folders are not fixed. A novel vault can create `角色/`, `世界观/`, `道具/`, and `情节/`. A business vault can create `客户/`, `竞品/`, `流程/`, and `决策/`. A toolbook can create `基础概念/`, `操作流程/`, `方法模型/`, `对比分析/`, `模板/`, and `FAQ/`.

## Creation Flow

1. User uploads files or describes the vault goal.
2. Local tools convert supported files into Markdown.
3. Original files are preserved under `raw/原始文件/`; converted Markdown is written under `raw/转换后的MD/`.
4. The app scans the corpus: file types, headings, tables, entities, concepts, repeated topics, workflows, and source anchors.
5. The LLM receives the scan and proposes 2-3 wiki architecture directions.
6. User chooses one direction.
7. Local code creates the fixed vault scaffold.
8. The app generates the first wiki:
   - `index.md`: master catalog.
   - `overview.md`: executive summary.
   - `hot.md`: recent context cache.
   - `log.md`: append-only operation log.
   - 8-15 initial high-confidence pages for structured material.
9. The app writes `_reports/整理记录/首次创建报告.md`.

## First Wiki Definition

The first wiki is not a perfect final knowledge base and not a mechanical chapter split. It is the first usable retrieval surface.

For a 30,000-word toolbook, the first wiki should usually include:

```text
wiki/
  index.md
  overview.md
  hot.md
  log.md
  基础概念/
    账号定位.md
    目标用户.md
  操作流程/
    账号冷启动流程.md
    选题流程.md
    数据复盘流程.md
  方法模型/
    标题公式.md
    封面设计原则.md
  对比分析/
    图文笔记_vs_视频笔记.md
  模板/
    笔记标题模板.md
  FAQ/
    为什么笔记没有流量.md
```

Each page should contain a summary, key points, applicable scenarios, related pages, and source references. The full original text remains in `raw/转换后的MD/`.

## Runtime Retrieval Flow

```text
用户输入
-> 搭子指令
-> 近期对话上下文
-> wiki/hot.md
-> wiki/index.md and relevant wiki pages
-> raw fallback when wiki misses
-> local tool results when needed
-> LLM output
-> writeback candidate
-> wiki/log.md and organize report
```

Wiki is always searched before raw. Raw is the fallback. If both miss, the model may answer from its own capability, but the generated insight should be marked as a candidate wiki update.

## Health Check

Health check needs UI support, but not a complex dashboard. The second-column knowledge tree should expose `健康检查` on the vault context menu and show a simple result:

- 未整理资料
- 缺失引用
- 冲突内容
- 建议新增栏目

The detailed report is written to `_reports/健康检查/健康检查_时间.md`.

## Ability Mapping

### Karpathy LLM Wiki / Autoresearch

- Fixed workspace protocol maps to `CLAUDE.md / raw / wiki / _reports / _templates`.
- Program instructions map to vault-local `CLAUDE.md`.
- Raw data remains preserved.
- Wiki is the structured working memory.
- Every organize pass is measurable and logged.

### Understand-Anything

- `project-scanner` maps to local corpus scanner.
- `file-analyzer` maps to Markdown source analyzer.
- `tour-builder` maps to wiki architecture and guided index generation.
- `graph-reviewer` maps to link/source/coverage health check.
- `domain-analyzer` maps to business workflow and process extraction.
- `article-analyzer` maps to entity, claim, and implicit relation extraction from wiki pages.

### claude-obsidian

- Fixed scaffold maps directly to `index.md`, `log.md`, `hot.md`, `overview.md`, `_templates/`, and `CLAUDE.md`.
- Resume/check behavior maps to organize reports and health checks.
- Filing current conversation maps to `raw/对话记录/`.
- Autonomous research loop maps to later local research and source filing tools.

## Phase Plan

### Phase 1: Vault Protocol Foundation

- Add a local scaffold builder for the 2.0 file protocol.
- Stop asking the LLM to generate a huge `CLAUDE.md` string.
- Make `VaultWizard`, `vaultStore`, and desktop `vaultFs` share the same scaffold.
- Create `index.md`, `overview.md`, `hot.md`, `log.md`, `meta/`, `_templates/`, and report folders for every new vault.
- Accept small structured seed-page specs and let local code turn them into Markdown pages.

### Phase 2: Local Markdown Ingestion

- Preserve original files.
- Convert supported Word, PDF, Excel, PPT, Markdown, text, subtitle, audio transcript, and browser content into Markdown.
- Store source metadata and hashes.
- Record conversion reports.

### Phase 3: First Wiki Generation

- Scan converted Markdown.
- Let the LLM propose wiki architecture directions.
- Let user choose one.
- Generate first 8-15 high-confidence wiki pages through controlled page-level writes.

### Phase 4: Progressive Organize

- Replace raw-to-wiki large JSON actions with small candidate updates.
- Review candidate additions, conflicts, and new folder suggestions.
- Write reports and mark processed raw ranges.

### Phase 5: Retrieval and Writeback

- Implement wiki-first retrieval with raw fallback.
- Attach source references.
- Convert valuable conversation input/output into wiki update candidates.

### Phase 6: Health Check and Export

- Add second-column health check UI.
- Detect orphan pages, broken links, missing source references, duplicate pages, stale hot cache, unprocessed raw, and conflicts.
- Export `.jcvault` for Web/Desktop migration.

## Phase 1 Acceptance Criteria

- New vaults always include the 2.0 scaffold.
- `CLAUDE.md` is generated locally from vault metadata and folder rules.
- `VaultWizard` prompts no longer request `claudeMd`.
- Uploaded material mode can pass small seed-page specs that local code writes as Markdown pages.
- Desktop disk scaffold and IndexedDB scaffold match.
- Existing templates still work.
