#### 5.7.2 端口聚合规则（**已修正缺陷 1**）

**核心原则**：**绝不允许静默丢弃任何外部输入**。

**prompt-flow 端口聚合规则（最终版）**：

当 Group 折叠时，系统会扫描所有从**外部**进入 Group 的 `prompt-flow` 连线：

- 如果只有 1 条外部 prompt-flow 连线 → 折叠后暴露 **1 个** `left-prompt` 端口。
- 如果有 ≥2 条外部 prompt-flow 连线 → 折叠后**同时暴露 N 个独立的 `left-prompt` 端口**（按连线顺序编号：Prompt-1、Prompt-2…），**不允许合并或丢弃**。
- 额外提供一个可选的「合并所有外部 Prompt」虚拟端口（用户可在 Group 内部放置一个显式的 Text Merge 节点来使用）。

**context-inject 端口**：
- 折叠后暴露一个或多个 `left-context` 端口（按类型分组显示摘要，例如「+3 知识库」），hover 可展开完整列表。
- 内部所有 Context Provider 的引用关系必须在折叠态可见（至少以计数形式）。

**media-reference 端口**：
- 按媒体类型分别聚合（图片参考、视频参考），不丢弃任何参考。

**source 端口**：
- 折叠后暴露 Group 内部所有有对外输出的 source 端口类型。

**验收标准**：Group 内 3 个 TextNode 各自接收不同外部 prompt，折叠后必须能看到 3 个独立的 prompt 输入端口，不能只剩 1 个。

#### 5.7.3 Context 作用域隔离规则（同 v4.0 加强版）

- Group 内部的 Vault/Skill/ToolNode **默认只对 Group 内部 LLM 生效**。
- 如需让外部 LLM 看到，必须在 Group 边界显式连出 `context-inject` 端口。
- 执行时 Engine 以子 scope 处理 Group 内部上下文。

### 5.8 默认起始模板（**v5.1 修正：五节点标准工作流**）

新建画布时，默认放置以下 5 个预置连线节点：

```
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ 📝 需求   │ → │ 🧠 AI大脑 │ → │ 📝 输出   │ → │ 🖼️ 生成   │ → │ 🖼️ 结果   │
│ 写你的想法 │   │ 执行思考  │   │ 检查编辑  │   │ 文生图    │   │ 预览保存  │
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
```

**为什么是 5 个节点而不是 4 个？**

LLM 的输出必须落到一个文本节点，用户**检查、编辑、确认**后再流入媒体生成。这符合产品「纯手动显式控制」原则——用户永远在链条中有一道审查关卡。

**用户操作流：**
1. 在「📝需求」写需求（如「生成一组产品图，正面+侧面+45度」）
2. （可选）拖知识库/Skill 连线到「🧠AI大脑」
3. （可选）拖工具集连线到「🧠AI大脑」
4. 点「🧠AI大脑」的 ▶ → 生成内容流入「📝输出」
5. 在「📝输出」里检查、修改 AI 生成的内容
6. 点「🖼️生成」的 ▶ → 产出流入「🖼️结果」

### 5.9 LLM 节点的标准上下文注入流（**v5.1 修正**）

LLM 节点有三个输入端，**同时注入、同等重要**：

```
📁知识库 ──┐
🧩Skill  ──┤  (context-inject 紫色虚线)
           ├──→ 🧠AI大脑
📝用户文本 ─┘  (prompt-flow 蓝色实线)
           
🔧工具集 ────→ 🧠AI大脑  (context-inject，可选)
              ↑ 工具集连线后仅暴露 tool definitions，
                LLM 根据 prompt 自行判断是否调用，
                不做强制调用。
                逻辑对齐 useChat.ts 的 tool loop。
```

**关键原则（与 CLAUDE.md 对齐）：**
- 用户文本（prompt-flow）是**最高业务决策信号**（P5）
- 知识库召回结果是 **user-side evidence**，不进 system role（P3）
- 工具集连线 ≠ 强制调用，LLM 自行判断（P4）
- 执行时统一通过 `ConversationContextEngine.build()` 构建上下文（P7）

### 5.10 工具的参与方式（**v5.1 明确**）

工具集（ToolNode）连线到 LLM 节点后：

- LLM 执行时，工具集内的开关工具被翻译为 OpenAI tool definitions
- LLM 根据当前 prompt 自行决定是否调用
- 行为与当前对话区 `useChat.ts` 的 tool loop 完全一致
- **不做强制调用**。连线工具集 = 「你可以用这些工具」，不是「你必须用」
- 一个 LLM 可以连线多个 ToolNode（取并集）

### 7.4 脏检查传播规则（**v5.0 明确定义**）

**传播策略**：**传递闭包（transitive closure）**

- 当一个节点的内容或配置发生变化时：
  - 标记该节点为脏
  - 沿所有 **prompt-flow** 和 **media-output** 方向，递归标记所有可达的后继节点为脏
- Context Provider 节点（vault/skill/toolset）内容变化时：
  - 标记所有直接连到它的 LLM 节点为脏（通过 context-inject 边）
  - 继续沿 prompt-flow 传播
- 环路检测：拓扑排序前检测到环路 → 跳过该环路中的脏标记 + 弹出 toast 警告「检测到环路，部分节点无法自动标记脏」

用户可随时手动「清除所有脏标记」或「仅清除选中子图」。

## 十二、验收标准（v5.1 更新）

- 新建画布 → 五节点起始模板（📝需求 → 🧠AI大脑 → 📝输出 → 🖼️生成 → 🖼️结果）
- LLM 节点同时接受知识库（紫色虚线）+ Skill（紫色虚线）+ 用户文本（蓝色实线）三路输入
- LLM 节点执行时上下文组装正确：知识库→user evidence，Skill→system prompt（通过 skillApplicability），用户文本→最高优先级
- ToolNode 连线后 LLM 自行判断是否调工具（不做强制调用），对齐 useChat.ts tool loop
- Group 内多个外部 prompt 输入，折叠后全部保留为独立端口，无数据丢失
- 14×14 矩阵 + 边类型自动推断规则在代码中 1:1 实现
- 脏检查能正确传递闭包（含 Context Provider 变化场景）
