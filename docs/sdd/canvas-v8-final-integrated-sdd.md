# 画布系统 V8 最终设计 SDD（v4.0 整合版）

> 版本: v4.0  
> 日期: 2026-06-03  
> 状态: 待评审（可直接进入 Phase 0）  
> 基于: DeepSeek v3.0 + Grok 深度批判 + 修正明细 v1 + 项目 CLAUDE.md 宪法  
> 原则：手动显式 > 核心对象可视化 > 手感润滑 > 可组合性 > 工程可落地（9-10 周）

---

## 零、产品宪法（不可违反）

所有设计必须严格遵守 `CLAUDE.md`：

- P1. 纯手动显式控制（默认无任何自动编排）
- P3. Knowledge **只能**作为 user-side evidence/context，绝不进入 system role
- P4. Tool 默认关闭，用户显式开启
- P7. ConversationContextEngine 是唯一上下文构建入口
- P10. 画布与对话区共享底层 Store/Engine，但选择完全独立

任何与以上冲突的设计直接否决。

## 一、核心目标与范围

### 1.1 本次 V8 要解决的根本问题

1. 知识库、Skill、工具在画布中**不可见、不可组合**
2. 节点 UI 混乱、缩放手感差、执行控制弱
3. 复杂度失控（42 节点平铺，无有效分组机制）
4. 画布与对话区完全割裂
5. T8 遗留节点导致维护地狱

### 1.2 最终节点数量（精确）

**激活节点：14 个**

- Context Provider（选择器，无执行按钮）：3 个（`vault`、`skill`、`toolset`）
- 核心原语：2 个（`text`、`llm`）
- 生成：3 个（`imageGen`、`videoGen`、`audioGen`）
- 结果：3 个（`imageResult`、`videoResult`、`audioResult`）
- 编排：3 个（`loop`、`textSplit`、`group`）

**废弃节点：32 个**（全部标记 `@deprecated`，提供清晰降级路径，不再新画布中出现）

## 二、设计原则（最终版）

1. **手动显式至上**（默认空白或极简四节点模板）
2. **核心对象第一公民**：知识库/Skill/工具必须有独立可视形态（选择器）
3. **节点语义差异优先** + 轻量共享 Frame 保底一致性
4. **手感是 P0 基础设施**（Phase 0 必须先验证）
5. **可组合性 > 数量精简**（保留必要编排能力）
6. **用 Group 管理复杂度**（Group 必须真正可用）
7. **执行可观测 + 可中断**（至少支持停止 + 子图 + 媒体缓存）
8. **渐进式披露 + 强预设**
9. **画布与对话区共享源、独立选择**
10. **Context Provider = 纯选择器**（无 ▶，连线仅标记引用关系，真实召回由 LLM 执行时通过 Engine 完成）

## 四、通用交互原语（Frame + 手感）

所有节点共享轻量 Frame：
- 左 3px 角色色条
- 头部：图标 + 标题 + 折叠按钮 + 删除
- 底部执行条（Context Provider 和 Result 节点无 ▶■）
- 右下角统一缩放手柄 ▦

**Phase 0 必须实现的手感硬约束**：
- 拖拽/缩放期间全局 `.is-interacting` 冻结策略（pointer-events: none + transition: none + 内容渲染暂停）
- Resize 使用原生 pointer + RAF 直接改 DOM，结束时才批量写入 Pinia
- 所有节点支持 collapsed（折叠态只保留头部 + 端口 + 状态，内容区 `display:none`）
- 最小 240×100px，双击头部恢复默认尺寸

## 五、关键节点详细规格

### 5.1 Context Provider 节点（选择器）

**核心原则**：
- 永远没有 ▶ 按钮
- 连线仅**标记引用关系**
- 真正召回/注入发生在 LLM 节点执行时，由 `ConversationContextEngine` 完成

**VaultNode 示例**：
- 可选择具体页面 + 钉选
- 右侧 source Handle 只能连到 LLM 的 `left-context` 端口
- 多个 LLM 可共享同一个 VaultNode

相同规则适用于 SkillNode（双击可只读预览 SKILL.md）和 ToolNode（可命名复用）。

### 5.2 TextNode（性能优先）

**折叠态**：Markdown 紧凑预览（marked 安全渲染），高度随内容智能增长（有上限）。

**展开态**：
- 原地展开完整 Tiptap（复用 EditorPanel 核心扩展）
- 支持 [[wiki-link]]、代码高亮、KaTeX、Mermaid、表格等

**5 条硬性能约束**（必须在 Phase 0 验证）：
1. **同时最多只有一个节点处于全功能 Tiptap 编辑态**（blur 或点击其他节点立即降级）
2. 降级时快速销毁 Tiptap 实例，保留纯 Markdown 预览
3. Tiptap 实例采用延迟初始化（真正进入编辑态才创建）
4. 重型扩展（Mermaid、WikiLink 完整功能）按需动态加载
5. 30 节点画布（至少 4 个 TextNode 展开过）在 MacBook Air M1 上展开/折叠/拖拽时，**主线程 Jank < 200ms**（需有自动化基准）

拖入 .md/.txt 自动导入。▶ 按钮 = AI 润色/续写（轻量调用，不走完整 LLM 节点）。

### 5.3 LLM 节点（Tab 渐进披露）

采用 Tab 设计，默认仅显示摘要行，大幅降低信息密度：

Tab 列表（点击切换）：
- 📋 摘要（模型 + 温度 + 已生效上下文数量 + 上游预览）
- 📁 知识库（连线注入列表 + 节点内选择 + 去重后生效列表）
- 🧩 Skill（同上 + 适用性提示）
- 🔧 工具（连线 ToolNode + 节点内开关 → 并集）
- ⚙️ 高级（max_tokens、top_p、systemPrompt 等）

**上下文优先级（严格）**：
1. 连线注入的 Context Provider（最高，显式意图）
2. LLM 节点内部下拉选择
3. 画布 Context 栏默认值（最低）

**合并规则**：
- Vault/Skill：自动去重
- Tool：取并集

端口：
- `left-prompt`（prompt-flow）
- `left-context`（context-inject，接收 vault/skill/toolset）
- `right-text`（输出）

执行时必须通过 `ConversationContextEngine.build()`，Knowledge 严格只进入 user 侧证据。

### 5.4 媒体生成节点（以 videoGen 为例）

主要参数常驻（比例、分辨率、时长），参考帧区大面积展示，高级参数折叠。

左端口：`left-prompt` + `left-image`（media-reference）
右端口：`right-media`（media-output）

所有生成节点必须实现完整的异步状态机（见执行章节）。

### 5.5 结果节点

统一画廊模板 + 右键菜单（保存到知识库、设为参考、上传素材库等）。

### 5.6 编排节点（loop / textSplit）

保持原有语义，重点做好端口数量动态生成和状态可视化。

### 5.7 Group / Subgraph（本版最加强化部分）

**这是 V8 能否真正管理复杂度的关键，必须做到可用。**

#### 5.7.1 两种状态

- **展开态**：普通容器，可随意拖入节点、连线
- **折叠态**：只显示标题 + 颜色 + 聚合后的输入输出端口 + 子图执行按钮

#### 5.7.2 端口聚合规则（必须实现）

| 内部端口类型 | 折叠后对外暴露 | 聚合策略 |
|--------------|----------------|----------|
| prompt-flow target | 单个 `left-prompt` | 取“主入口”文本（用户可标记）或第一个 |
| context-inject target | 多个或合并 | 显示“+3 知识库”等摘要，hover 展开列表 |
| media-reference target | 按类型聚合 | 图片参考 / 视频参考 分开展示 |
| 所有 source | 对应类型 source | 数量 = 内部有连出的最大分支数 |

**Context 作用域隔离规则**：
- Group 内部的 VaultNode/SkillNode 默认**只对 Group 内 LLM 生效**
- 如需“泄露”到外部，必须在 Group 边界显式连出 context 端口
- 执行时 Group 内部上下文由 Engine 以子 scope 方式隔离构建

#### 5.7.3 独立执行语义

- 右键 Group → “仅执行此子图”
- 执行时内部所有节点状态独立显示，外部节点不受影响
- 结果可通过 Group 对外暴露的 source 端口流出

#### 5.7.4 模板导出机制

- Group 可导出为“可复用模板”
- 模板中的 Context Provider 变成**占位符**（用户拖入模板后需要重新绑定具体 vault/skill）
- 模板支持版本和描述

#### 5.7.5 视觉与交互

- 折叠态有明显“容器”边框和标题栏
- 展开/折叠有平滑动画（性能允许时）
- 内部连线在折叠态完全隐藏

**验收**：用户能在一个 25 节点画布里，用 2-3 个 Group 把逻辑清晰地分成“素材准备 / 多版本生成 / 后处理”三个子图，并能独立执行其中一个。

## 六、边系统与完整验证矩阵

### 6.1 4 种边类型（语义收敛）

- `prompt-flow`（蓝色实线）
- `context-inject`（浅紫虚线）
- `media-reference`（绿色虚线）
- `media-output`（橙色实线）

### 6.2 14×14 完整连接验证矩阵（核心规则）

（矩阵已按最终 14 个激活节点完整定义，以下为关键节选，完整版见实现时 `isValidConnection`）

**核心原则**：
- Context Provider（vault/skill/toolset）**只能**连到 LLM 的 `left-context`
- 结果节点只能连到对应类型的生成节点作为参考
- Group 在折叠态的验证规则与展开态不同（运行时动态判断）
- textSplit 可连 LLM 和所有生成节点
- loop 可作为中继连多种类型

**Group 特殊规则**：
- 展开态：内部连线完全自由（按普通规则）
- 折叠态：只允许连入聚合后的端口，非法连线在拖拽时就拒绝

非法连线必须给出清晰中文提示，例如：“知识库选择器只能连接到 AI 大脑节点”。

## 七、执行模型（V8 务实增强版）

### 7.1 基础能力（必须交付）

- 每个可执行节点独立 AbortController（支持 ▶ / ■）
- 支持“执行选中子图”（含 Group）
- 全局停止

### 7.2 媒体输入签名缓存（新增重大价值）

- 对 imageGen / videoGen / audioGen 的输入（prompt + 参考图 URL 列表 + 主要参数）计算 SHA256
- 相同签名 + 模型 + 版本 且结果未过期 → **秒级复用**，并在节点上显示“缓存命中”徽章
- 用户可手动“强制重新生成”

### 7.3 异步任务状态机（所有生成节点统一）

`idle → submitting → polling → success / cancelled / error`

状态必须实时反映在节点上（进度、剩余时间估算、失败原因）。

### 7.4 基础脏检查（V8 最低要求）

- TextNode / LLM 节点内容变化时，标记下游所有 prompt-flow 连接的节点为“脏”
- 脏节点执行前给出明显视觉提示
- 完整增量缓存 + 轨迹可视化 留到 V9

## 八、视觉层级与执行时动态反馈（新增第八章）

### 8.1 静态 6 级视觉权重（从高到低）

1. 当前正在执行的节点 + 其直接连线（最强呼吸/高亮）
2. 选中节点 + 相关连线
3. 所有 Context Provider 节点（浅紫 + 虚线，永远保持一定可见性）
4. 普通已完成节点
5. 折叠态节点
6. 背景网格与非相关元素（自动降暗）

### 8.2 执行时动态强化规则（性能硬约束）

- 只有当画布节点总数 ≤ 15 时，才启用连线呼吸动画
- 节点数 > 15 时，动态效果自动降级为“仅高亮当前执行路径的连线 + 状态色变化”
- 所有动画必须使用 `transform` / `opacity`，并在 `.is-interacting` 期间完全禁用
- Token 贡献提示仅在 hover 对应 context 连线时出现（轻量）

目标：既让用户清楚看到“知识库/Skill 在这次执行中实际发挥了作用”，又不破坏手感。

## 九、画布与对话区的关系（独立选择）

- 共享：`vaultStore`、`agentStore`、`ConversationContextEngine`
- 完全独立：画布 Context 栏选择的默认值、每个 LLM 节点的选择，**绝不影响**对话区当前会话
- 反之亦然
- 画布执行产生的文本/媒体，**默认不**自动写入对话历史（用户可手动“发送到对话区”）

## 十、迁移策略（务实且用户友好）

### 10.1 打开旧画布时的行为

1. 自动备份原 `.jccanvas` 文件（带时间戳）
2. 弹出**迁移向导弹窗**（非阻塞，可稍后处理）
3. 对每个废弃节点提供三种选项：
   - **一键智能升级**（推荐）：按映射表自动替换为新节点 + 尽量保留参数
   - **逐个处理**
   - **保留为只读警告节点**（黄色角标，点击可查看“此节点已废弃，建议替换为 XXX”）

强烈不推荐长期“保留旧版可执行”选项，建议设置 3 个月后旧节点彻底只读。

## 十三、风险与缓解（第十三章）

1. **Group 复杂度超预期** → Phase 2 前必须有可点击原型 + 端口聚合规则表评审
2. **TextNode 性能不达标** → Phase 0 必须跑通 30 节点基准，达不到就改用更轻量的 Markdown + 简单富文本方案
3. **动态视觉动画掉帧** → 严格按节点数量分级策略，动画默认关闭，用户可手动开启
4. **迁移用户怨声载道** → 必须有“一键智能升级”作为主路径 + 清晰的“为什么废弃”文案
5. **VueFlow 在重 Handle 场景下的性能墙** → Phase 0 必须做 Handle 数量压测，必要时提前准备简化边渲染方案
6. **LLM 节点 Tab 依然太重** → 默认只显示“摘要 + 关键已选上下文”两行，其余全部折叠
7. **底部推荐功能被滥用** → 必须两段式确认（先生成推荐，用户二次点“确认创建”），并记录使用数据

## 十四、与之前版本的关键差异

- 比原 DeepSeek v3.0 更完整：Group 有实质规则、视觉层级有性能约束、迁移 UX 清晰、TextNode 约束更平衡
- 比 Grok 版更务实：没有 12-15 周幻想、没有 popover 编辑、执行引擎增强但不一步登天、节点数精确 14 个
- 最大新增价值：真正可用的 Group + 媒体签名缓存 + 安全的动态视觉反馈 + 可落地的 9.5 周计划
