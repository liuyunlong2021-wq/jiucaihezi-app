# 创作面板多画布与直接改图 SDD

> **状态**: 设计已确认，待实施
> **日期**: 2026-07-12
> **范围**: 桌面端 Tauri 创作面板
> **核心决策**: 文件树是画布唯一真源；创作面板直接复用现有媒体任务链路；OpenCode 不参与媒体生成和改图。

## 1. 背景

当前创作面板只有一个 `default` 画布。用户设计到一半后若要处理另一项任务，只能清空或覆盖当前内容，无法像切换项目文件一样暂停和恢复工作。

同时，当前方案曾计划通过 OpenCode、`JC-瞬间创作` 和 Canvas Bridge 完成改图。该链路重复了创作面板已经具备的模型选择、任务提交、轮询、下载和任务恢复能力，增加了不必要的进程边界与异步任务归属问题。

本方案将两件事分开：

- 创作面板负责媒体生成、选中图片改图和画布编辑。
- 文件树负责画布文件的创建、打开、重命名、复制和删除。
- OpenCode 未来只用于 Wiki 知识关系整理，不进入本期改图链路。

## 2. 目标

### 2.1 多画布

一个项目可以拥有多个独立画布。用户可以随时切换，之前画布上的图片、文字、箭头、画笔、编号、编组、位置和层级都能完整恢复。

### 2.2 直接改图

用户选中画布中的一张图片，在创作面板输入提示词并选择模型后，直接使用现有 `mediaTaskStore` 和媒体生成 API 提交改图。成功结果可以替换原图或在旁边新增一版。

### 2.3 本地文件真源

画布列表和画布内容全部来自项目目录中的文件。数据库和 localStorage 不维护第二份画布列表；删除应用缓存不会丢失画布。

### 2.4 职责边界

| 现有模块 | 本期职责 |
|---|---|
| `ProjectFileTree` | 展示和管理 `.jccanvas` 文件，点击后打开画布 |
| `canvasPersistence` | 扫描、读取、标准化和原子写入画布文件 |
| `canvasStore` | 保存当前画布路径和纯 JSON 场景，串行化保存与任务写回 |
| `CreationPanel` | 呈现 Leafer 场景、画布工具和媒体参数，不持有后台任务生命周期 |
| `mediaTaskStore` | 沿用现有提交、轮询、恢复和本地化流程，成功后调用画布写回 |

不新增第二套媒体执行器、画布索引数据库或通用命令总线。

## 3. 非目标

- 不让 OpenCode、Skill、MCP 或 Python 脚本参与本期媒体生成和改图。
- 不建设通用 Canvas Bridge。
- 不实现多人协作、云端同步和画布冲突合并。
- 不实现 Web 端多画布；Web 端没有桌面项目文件树。
- 不实现画布文件夹、画布专属媒体目录或媒体自动清理。
- 不实现画布缩略图缓存；文件树第一期使用画布图标和文件名。
- 不把 Wiki 知识节点混入媒体画布 schema；知识画布另行设计。

## 4. 用户体验

### 4.1 新建画布

用户可以从两个位置执行同一个命令：

1. 文件树 `jc-canvas` 目录右键“新建画布”。
2. 创作面板标题栏点击“＋”。

两个入口都在磁盘创建新的 `.jccanvas` 文件，并立即在创作面板打开。默认文件名依次为：

```text
未命名画布.jccanvas
未命名画布 2.jccanvas
未命名画布 3.jccanvas
```

### 4.2 切换画布

用户点击文件树中的 `.jccanvas` 文件：

1. 完成当前画布待保存内容的落盘。
2. 保存成功后清理当前 Leafer 场景和撤销栈。
3. 读取目标文件并恢复完整场景。
4. 创作面板标题显示目标文件名。

保存失败时不切换，并向用户显示错误，避免当前编辑内容丢失。

### 4.3 管理画布

`.jccanvas` 文件在文件树支持：

- 打开
- 重命名
- 复制
- 删除
- 在 Finder/资源管理器中显示

重命名直接重命名磁盘文件，标题随文件名变化。复制画布时生成新的 `canvasId`，场景内容和媒体引用保持不变。

删除画布只删除 `.jccanvas` 文件，不删除任何 `jc-media` 素材。删除当前画布后打开同目录中最近修改的另一张画布；没有剩余画布时创建新的空白画布。

### 4.4 直接改图

首期只支持选中一张图片：

```text
选中图片
  → 创作面板自动把图片作为参考图
  → 输入提示词并选择支持图生图的模型
  → 选择“替换原图”或“新增一版”
  → 提交现有 mediaTaskStore
  → 结果下载到 jc-media
  → 写回提交任务时绑定的画布
```

默认写回方式为“替换原图”。“新增一版”保留原图，并将结果放在原图右侧 24px。

未选中图片时，创作面板保持现有文生图行为，生成结果加入当前画布。

## 5. 文件结构

```text
{projectDir}/
├── jc-media/
│   ├── images/
│   ├── videos/
│   └── audio/
└── jc-canvas/
    ├── 海报方案一.jccanvas
    ├── 海报方案二.jccanvas
    └── 紧急海报.jccanvas
```

- `jc-media` 保存媒体字节。
- `jc-canvas` 保存画布场景文件。
- 画布文件只引用项目内媒体路径，不保存 Base64、Blob URL 或媒体字节。
- 画布文件名是用户看到的画布名称，不在 JSON 内重复保存 `name`。

## 6. 画布文件契约

`.jccanvas` 是 UTF-8 JSON 文件，使用专用扩展名，避免与 Obsidian JSON Canvas 的 `.canvas` 格式混淆。

```ts
interface CanvasDocumentV2 {
  version: 2
  canvasId: string
  updatedAt: number
  viewport: {
    x: number
    y: number
    zoom: number
  }
  scene: LeaferNodeJson[]
  assets: Record<string, CanvasAsset>
}

interface CanvasAsset {
  id: string
  kind: 'image'
  path: string
  source: 'creation' | 'drop' | 'paste' | 'image-edit'
  model?: string
  prompt?: string
  parentAssetId?: string
  createdAt: number
}
```

### 6.1 ID 规则

- 新建画布使用 `crypto.randomUUID()` 生成 `canvasId`。
- 每个可编辑根节点拥有稳定 node ID。
- node ID 只要求在所属画布内唯一。
- 复制画布时只更换 `canvasId`，节点 ID 保持不变。

### 6.2 图片序列化

Leafer 运行时可以使用 data URL 或解析后的本地显示 URL，但保存前必须经过标准化：

1. 图片节点通过 `assetId` 关联 `assets`。
2. `assets.path` 只保存相对项目路径，例如 `jc-media/images/abc.png`。
3. `scene` 中不得出现 `data:`、`blob:` 或图片 Base64。
4. 恢复画布时，再将 `assets.path` 解析为 Leafer 可显示的运行时 URL。

从剪贴板或项目外部拖入的图片，必须先复制到 `jc-media/images/`，再加入画布。

### 6.3 不持久化的状态

以下状态属于临时 UI 状态，不写入 `.jccanvas`：

- 当前选中的节点
- 当前工具
- 右键菜单状态
- 撤销/重做内存快照
- 正在打开的参数弹窗

应用可以使用现有本地偏好记录“项目最后打开的画布路径”。该值只是导航缓存，丢失后通过扫描 `jc-canvas` 恢复，不是画布真源。

## 7. 保存与切换

### 7.1 自动保存

- 场景发生有效变化后 500ms debounce 保存。
- 平移或缩放结束后同样 debounce 保存，保证重新打开时恢复工作视角。
- 切换画布、关闭创作面板和应用退出前立即 flush。
- 同一画布同时只能有一个保存 Promise；后续保存复用或排队，不能并发覆盖。
- 内容和视角都没有变化时不写文件。

### 7.2 原子写入

保存采用同目录临时文件后 rename：

```text
海报方案一.jccanvas.tmp
  → 写入并关闭
  → rename 为 海报方案一.jccanvas
```

第一期只有当前应用写画布，不需要 lock、revision、轮询或冲突合并。原子写入用于防止崩溃和断电造成半个 JSON 文件。

### 7.3 打开规则

进入创作面板时按以下顺序选择画布：

1. 打开该项目最后使用且仍存在的画布。
2. 否则打开 `jc-canvas` 中最近修改的画布。
3. 目录为空时创建 `未命名画布.jccanvas`。

切换项目后必须重新扫描新项目的 `jc-canvas`，不得沿用上一个项目的当前画布。

## 8. 媒体任务与画布绑定

现有媒体任务是异步执行。用户提交任务后可能立即切换画布，因此任务必须记录提交时的写回目标。

```ts
interface CanvasTaskTarget {
  canvasId: string
  canvasPath: string
  operation: 'append' | 'replace' | 'add-version'
  targetNodeId?: string
  targetAssetId?: string
  expectedAssetPath?: string
}
```

`MediaTask` 增加可选 `canvasTarget`。创建任务时写入并随现有任务数据一起持久化。任务完成事件只提供 `taskId`，消费者从任务记录读取 `canvasTarget`，不得使用“当前打开的画布”推断写回位置。

任务成功并完成本地媒体保存后，`mediaTaskStore` 调用画布持久化模块的单一结果写入函数。该函数负责判断目标是否为当前画布并完成写回；不能依赖 `CreationPanel` 的事件监听器正好处于挂载状态。组件只负责显示当前画布，不承担后台任务生命周期。

### 8.1 写回当前画布

如果任务目标仍是当前打开画布：

- 通过 `canvasStore` 修改当前纯 JSON 场景，由 `CreationPanel` 同步到 Leafer。
- 更新资产引用。
- 进入当前撤销栈。
- 触发正常自动保存。

### 8.2 写回后台画布

如果用户已经切换到其他画布：

- 读取任务绑定的 `.jccanvas`。
- 找到目标节点并修改文档。
- 原子写回该文件。
- 不切换用户当前画布。
- 在任务历史显示“结果已写入：画布名称”。

### 8.3 目标失效

任务完成前可能发生以下情况：

- 目标画布被删除。
- 目标图片被删除。
- 目标图片当前路径与 `expectedAssetPath` 不一致，说明它已被用户或另一项任务替换。

处理规则：

- 结果媒体仍保存在 `jc-media`，不能丢弃。
- 不猜测新的替换目标。
- 将任务标记为“生成成功，未写入画布”。
- 用户可以从任务历史执行“加入当前画布”。

首期不实现多任务对同一节点的自动合并。后完成的任务若发现目标 asset 已变化，按目标失效处理。

## 9. 创作面板交互

### 9.1 标题栏

创作面板标题栏显示：

```text
创作面板 · 海报方案一                         ＋
```

- 名称来自当前 `.jccanvas` 文件名。
- 点击名称在文件树中定位当前画布。
- `＋` 创建新画布文件。
- 不在标题栏建设第二套画布列表。

### 9.2 选中图片

选中单张图片后：

- 仅在当前模型支持参考图时，将其加入参考输入。
- 参数区提供“替换原图 / 新增一版”二选一控制。
- 切换为不支持图生图的模型时，明确取消改图模式并保留选中状态。

多选图片首期不进入改图模式，保持普通选择操作。

### 9.3 新生成媒体

- 图片生成结果按任务 `canvasTarget` 加入对应画布。
- 视频和音频继续落入 `jc-media` 和任务历史，首期不作为 Leafer 画布节点自动加入。
- 任务失败不修改画布。

## 10. 文件树行为

`ProjectFileTree` 将 `.jccanvas` 识别为画布文件：

- 使用画布图标。
- 单击在创作面板打开，不进入文本编辑器。
- 重命名限制为当前项目 `jc-canvas` 内，保留 `.jccanvas` 扩展名。
- 新建、复制、删除完成后刷新现有文件树，不维护额外列表缓存。

用户手动在 Finder 中复制或重命名 `.jccanvas` 后，刷新文件树即可看到结果。若复制文件保留了相同 `canvasId`，应用首次打开副本时为副本生成新 `canvasId` 并保存，避免任务写回歧义。

## 11. 旧数据迁移

当前旧画布位于：

```text
{projectDir}/.jiucaihezi/canvas/default.json
```

首次打开新版本时：

1. 若 `jc-canvas` 已有画布，不执行迁移。
2. 若旧 `default.json` 存在，转换为 `jc-canvas/未命名画布.jccanvas`。
3. 将旧 `layers[]` 和 `annotations[]` 转换为完整 `scene[]`。
4. 成功写入并验证可读取后，将旧文件保留为 `default.json.v1.bak`。
5. 若没有旧文件，创建空白画布。

迁移失败时保留原文件并显示错误，不创建看似成功的空画布覆盖用户预期。

## 12. 错误处理

| 场景 | 处理 |
|---|---|
| 画布 JSON 损坏 | 不覆盖原文件；提示无法打开，并允许在文件树中定位 |
| 自动保存失败 | 保留内存场景；显示持续错误；切换前再次保存 |
| 项目目录只读 | 禁止新建和保存，画布保持只读显示 |
| 引用媒体不存在 | 保留节点和路径，显示缺失占位，不删除节点 |
| 后台任务目标失效 | 媒体保留，任务历史提供“加入当前画布” |
| 删除最后一张画布 | 删除确认后创建新的空白画布 |

## 13. 实施阶段

### Phase 0：可靠场景文档

- 实现 `CanvasDocumentV2` 和 scene serializer。
- Base64/data/blob URL 转项目媒体引用。
- 完整保存和恢复所有现有 Leafer 元素。
- 原子保存、自动保存去重和旧数据迁移。

验收：重启后图片、文字、箭头、画笔、编号、编组、位置、层级和视角一致；画布 JSON 不含媒体字节。

### Phase 1：文件树多画布

- 创建并扫描 `jc-canvas`。
- `.jccanvas` 文件打开、新建、重命名、复制和删除。
- 创作面板显示当前画布名称和新建快捷按钮。
- 切换项目和切换画布前可靠 flush。

验收：三个海报画布可以独立编辑并任意切换，关闭应用后继续上次内容。

### Phase 2：创作面板直接改图

- 选中图片作为现有媒体任务参考输入。
- 增加替换/新增一版选择。
- `MediaTask` 持久化 `canvasTarget`。
- 当前画布和后台画布按任务目标安全写回。

验收：在画布一提交改图后切换到画布二，任务完成仍只修改画布一；失败或目标失效不破坏任何画布。

### Phase 3：独立规划 Wiki 画布

当 Wiki 关系图的节点类型、交互和输出格式明确后，单独编写 SDD。届时只评估知识关系导入所需的最小接口，不复用媒体改图链路，也不预设 MCP。

| 删除画布 | 不删除任何 `jc-media` 文件 |
