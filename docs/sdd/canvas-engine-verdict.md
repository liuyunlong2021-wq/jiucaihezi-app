# 画布引擎最终裁决

> **日期**: 2026-07-11
> **背景**: 上一次画布（VueFlow 节点工作流）选错原点，浪费 1.5 个月。这次必须彻底确认。

---

## 一、需求全景

经过对黑曜石 Skill 合集（json-canvas、obsidian-markdown、obsidian-bases）的分析，画布需要覆盖**两类完全不同的场景**：

### A. 图片编辑画布（像素级操作）

```
创作面板产出图片 → 画布上编辑
  ├── 标注修图（箭头/画笔/文字圈选 → AI 按标注修改）
  ├── 去背景（选中 → 调 API → 透明背景）
  ├── 扩图（拖拽画框 → outpainting 补全）
  ├── 文字改写（OCR → 编辑 → 保留风格替换）
  ├── 图层分离（AI 分割 → 独立图层 → 重排/PSD 导出）
  └── Skill 模板（小红书封面/电商组图/Logo/宣传册）
```

### B. 知识图谱画布（JSON Canvas 规范）

```
Obsidian .canvas 文件 ← 我们的画布渲染/编辑
  ├── text 节点（Markdown 内容卡片）
  ├── file 节点（图片/PDF 嵌入）
  ├── link 节点（URL）
  ├── group 节点（容器 + 标签 + 背景色）
  └── edge（节点间连线，带箭头 + 标签 + 颜色）
用途：Wiki [[双链]] 关系图、剧本角色关系、情节分支、项目看板
```

## 二、上一次为什么失败

| | 旧画布（VueFlow） | 实际需要 |
|---|---|---|
| 引擎类型 | 节点工作流编辑器 | 图形编辑引擎 |
| 核心操作 | 连线节点 → 执行 | 拖拽图片 → 标注 → AI 编辑 |
| 类比 | ComfyUI / n8n | Figma / Canva / Obsidian Canvas |
| 失败原因 | **类型错配**——用流程图工具做图片编辑 |

**教训**：引擎的「类型」比「功能列表」更重要。

## 四、逐场景分析

### 场景 A：图片编辑

| 需求 | LeaferJS | Fabric.js | Konva.js |
|------|----------|-----------|----------|
| 图片拖入/粘贴 | ✅ | ✅ | ✅ |
| 选中/拖拽/缩放/旋转 | ✅ Editor 开箱即用 | ⚠️ 需手写控制器 | ⚠️ 需手写 |
| 框选吸附标尺 | ✅ Editor | ❌ 需手写 | ❌ 需手写 |
| 箭头/画笔标注 | ✅ Arrow + Pen 元素 | ⚠️ 需手写笔刷 | ⚠️ 需手写 |
| 文字标注叠加 | ✅ Text 元素 | ✅ Text | ✅ Text |
| 图层树管理 | ✅ 场景树 | ✅ 对象树 | ✅ Layer |
| 去背景 API 调用 | N/A（我们的 API） | N/A | N/A |
| OCR 文字识别 | N/A（GPT-4V） | N/A | N/A |
| **预估开发量** | **3-5 天** | **8-12 天** | **8-12 天** |

LeaferJS 的 Editor 插件直接省掉旋转/缩放/多选/框选/吸附/标尺的手写——这是 Fabric.js 上最耗时的部分。

### 场景 B：JSON Canvas 知识图谱

| 需求 | LeaferJS | Fabric.js | Konva.js |
|------|----------|-----------|----------|
| text 节点（Markdown） | ✅ Text（富文本） | ⚠️ 纯文本 | ⚠️ 纯文本 |
| file 节点（图片） | ✅ Image | ✅ Image | ✅ Image |
| group 节点（容器） | ✅ Group | ✅ Group | ✅ Group |
| edge（箭头连线） | ✅ Arrow + 端点 | ⚠️ 需手写 | ✅ Arrow |
| 自动布局 | 需 dagre/elkjs | 需 dagre/elkjs | 需 dagre/elkjs |
| 拖拽节点重排 | ✅ draggable | ✅ draggable | ✅ draggable |
| `.canvas` 文件读写 | 需手写 parser | 需手写 parser | 需手写 parser |
| **预估开发量** | **2-3 天** | **3-5 天** | **3-5 天** |

JSON Canvas 渲染是**纯自定义代码**（解析 JSON → 创建图形元素 → 关联事件），三个引擎都需要手写。LeaferJS 的优势在于 Arrow 原生支持端点箭头、Group 支持标签和背景色。

### 场景 C：黑曜石 Wiki 联动

```
用户说「可视化 [[角色A]] 的关系」
  → Agent (json-canvas Skill) 读取 vault 中的 .md 文件
  → 生成 .canvas JSON（text 节点 + file 节点 + edge）
  → 我们的画布渲染 .canvas
  → 用户在画布上拖拽调整布局、添加新节点、修改连线
  → 保存回 .canvas 文件
  → Obsidian 打开 vault，画布同步更新
```

这个流程与引擎选择**几乎无关**——任何引擎都需要同样的 JSON ↔ 图形映射逻辑。关键是引擎能否方便地创建带样式和交互的节点。

## 五、风险评估

### LeaferJS 的真实风险

| 风险 | 真实程度 | 缓解 |

| 社区小（6 vs Fabric 320） | 🔴 **真实** | 代码质量高，TS 100%，5 年持续维护 |

| 问题 | 严重程度 |
|------|---------|
| 没有内置 Editor | 🔴 手写旋转/缩放/多选/框选/吸附/标尺 → 至少 2 周 |
| 没有内置 Viewport | 🔴 手写无限画布滚轮缩放+拖拽 → 至少 1 周 |
| API 是 v5 时代设计 | 🟡 历史包袱重，v6 正在重构 |
| 英文文档 | 🟡 对中文团队有额外学习成本 |

## 六、最终裁决

### 🏆 选择 LeaferJS

**理由不是「它最安全」，而是「它最对」。**

Fabric.js 是更安全的选择（31k stars, 12 年历史），但它是「通用 Canvas 库」——你需要自己在上面搭建编辑器。上次失败是因为选了一个「类型不对」的工具（VueFlow），这次如果选 Fabric.js，虽然类型对了（图形引擎），但需要手写大量编辑基础设施——这是另一种偏差：类型对了，但「起跑线离终点太远」。

LeaferJS 的定位和我们需要的产品**完全重合**：
- 「AI 无限画布引擎」← 正是我们要做的
- 「图形编辑 + AI 协同」← 正是我们的工作流
- 「像 DOM 一样操作图形」← 降低开发门槛
- Editor + Viewport 插件 ← 省掉基础设施的几周开发

### Fallback 策略

如果 LeaferJS 在开发中遇到无法解决的 Bug：

```
LeaferJS 的 API 设计（场景树 + 元素 + 事件）与 Fabric.js 高度相似
→ 迁移成本可控（主要是元素创建 API 的语法差异）
→ 核心业务逻辑（标注收集、API 调用、JSON Canvas 解析）不需要改
```

### 一句话

**上次失败是因为用节点编辑器做图片编辑。这次 LeaferJS 是做图片编辑的图形引擎——类型对了。JSON Canvas 知识图谱是任何引擎都需要手写的上层逻辑，LeaferJS 不拖后腿。**
