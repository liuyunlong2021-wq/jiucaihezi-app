# SDD: 创作面板 2.0 能用版 UI 子方案

> **状态**: 草案
> **日期**: 2026-06-16
> **分支**: `codex/rh-creation-ui`
> **范围**: 基于现有 `CreationPanel` 的多渠道 UI 优化，当前重点覆盖 RunningHub 渠道与直连 NewAPI / T8 / 火山 / 特朗普渠道；只服务本次“能用版”创作面板升级，不包含 Canvas
> **目标**: 让用户在当前创作面板里清楚知道“我在用什么渠道、我填了什么参数、最终会如何提交、成果在哪里看”

---

## 1. 背景

当前创作面板已经具备一套相对稳定的上下结构：上方媒体资产 / 画廊，中部参数条，下方参考图 / 提示词 / 生成按钮。但渠道相关信息不够明确，用户无法快速确认：

- 当前模型走的是直连 NewAPI / T8 / 火山 / 特朗普，RunningHub 官方 API，还是 RunningHub 工作流
- 比例、分辨率、参考图等参数是否真的会提交
- 任务进度、失败原因、结果来源是否可追踪

用户主要是小白，因此 UI 目标不是“更强”，而是“更清楚、更少认知负担、更少出错”。

本阶段必须明确：直连 NewAPI / T8 / 火山 / 特朗普渠道不经过 `rh-adapter`；只有 RunningHub 渠道才和 `rh-adapter` 有关系。

---

## 2. 第一阶段目标

### 2.1 用户目标

用户进入创作面板后，只需要完成以下事情：

1. 选任务类型
2. 选模型
3. 上传参考图（如果需要）
4. 填提示词
5. 选比例和分辨率
6. 点生成
7. 在上方媒体资产 / 画廊查看结果和任务状态

### 2.2 产品目标

- 保留现有创作面板的基本布局
- 不把界面改复杂
- 让 RunningHub 与直连 NewAPI / T8 / 火山 / 特朗普渠道和参数状态可见
- 让成果和任务状态更清楚
- 让失败原因更直白

---

## 3. 范围

### 3.1 只做

- 现有 `CreationPanel` 的图片 / 视频 / 音频 / 工作流 UI 提示优化
- 当前重点覆盖 RunningHub 渠道与直连 NewAPI / T8 / 火山 / 特朗普渠道
- 参数区的渠道标识、模式标识、提交检查
- 上方媒体资产 / 画廊里的任务状态、结果摘要、来源标识
- 小白可读文案优化

### 3.2 不做

- 不做对话到创作的联动
- 不做分镜流水线
- 不做脚本面板
- 不做大规模布局重构
- 不做 Canvas 画布
- 不做品牌包装
- 不做工作流市场化能力

---

## 4. 设计原则

- **小白优先**: 所有文案尽量用“图片 / 视频 / 参考图 / 比例 / 分辨率 / 生成 / 结果”这类直白词
- **少步骤**: 用户不应被迫理解 RH endpoint、webappId、nodeInfoList、NewAPI channel type 等术语
- **当前版本优先**: 保留现有面板结构，只增强可理解性
- **结果可见**: 生成前、生成中、生成后都能看到状态
- **渠道透明**: 每次生成都能看出当前走的是 NewAPI 直连 / T8 / 火山 / 特朗普、RH 官方 API，还是 RH 工作流

---

## 5. UI 结构

### 5.1 总体布局

第一阶段沿用当前创作面板的上下结构，不改成左右分栏：

```text
┌──────────────────────────────────────────────┐
│ 顶部：媒体资产 / 正在运行任务 / 结果画廊        │
├──────────────────────────────────────────────┤
│ 中部：参数条（任务 / 模型 / 渠道 / 比例 / 分辨率）│
├──────────────────────────────────────────────┤
│ 底部：参考图 / 提示词 / 提交检查 / 生成按钮      │
└──────────────────────────────────────────────┘
```

不要引入新的左右分栏，不做复杂多栏工作台。用户仍然按现在的路径操作：先看上方资产和结果，再调中间参数，最后在底部输入并生成。

### 5.2 顶部媒体资产 / 画廊

顶部继续承担“成果在哪看”的职责，按优先级展示：

1. 正在运行的任务卡片（置顶）
2. 本次生成结果
3. 历史画廊 / 媒体资产

每个结果卡片要显示：

- 模型名
- 渠道
- 比例
- 分辨率
- 时间

### 5.3 中部参数条

参数条保持现在的轻量形态，优先展示用户真正要确认的信息：

1. 任务类型
2. 模型
3. 渠道
4. 模式
5. 比例
6. 分辨率

### 5.4 底部输入区

底部保留当前输入生成区，避免小白在多个区域之间来回找入口：

1. 参考图 / 素材输入
2. 提示词
3. 提交检查
4. 生成按钮

---

## 6. 关键 UI 文案

### 6.1 参数区

建议显示为：

```text
任务：图片
模型：GPT Image 2
渠道：T8 直连
模式：图生图
比例：16:9
分辨率：2k
```

或：

```text
任务：图片
模型：GPT2.0
渠道：RH 官方 API
模式：图生图
比例：16:9
分辨率：2k
```

### 6.2 提交检查

在生成按钮上方增加一条轻量摘要：

```text
T8 直连 · 图生图 · 参考图 1 张 · size=2048x1152
```

或：

```text
RH 官方 API · 图生图 · 参考图 1 张 · aspectRatio=16:9 · resolution=2k
```

这条信息只做确认，不做教育。

### 6.3 结果卡片

结果卡片底部显示：

```text
GPT Image 2 · T8 直连 · 16:9 · 2k
GPT2 · RH API · 16:9 · 2k
```

失败时显示：

```text
GPT Image 2 · T8 直连 · 失败
GPT2 · RH API · 失败
比例参数无效 / 上传失败 / 任务超时
```

---

## 7. 渠道表现

### 7.1 直连 NewAPI / T8 / 火山 / 特朗普

当模型是直连 NewAPI / T8 / 火山 / 特朗普渠道时，UI 显示：

```text
渠道：T8 直连 / 火山直连 / 特朗普直连 / NewAPI 直连
模式：文生图 / 图生图 / 文生视频 / 图生视频
```

这类模型不经过 `rh-adapter`。

### 7.2 RunningHub 标准 API

当模型是标准 RH API 时，UI 显示：

```text
渠道：RH 官方 API
模式：图生图 / 图生视频 / 文生图 / 文生视频
```

这类模型由 NewAPI 的 RunningHub 渠道转发到 `rh-adapter`。

### 7.3 RunningHub 工作流

当模型是 RH 工作流时，UI 显示：

```text
渠道：RH 工作流
模式：节点参数
```

工作流阶段只做最小表单，不做复杂节点编辑器。这类模型同样由 NewAPI 的 RunningHub 渠道转发到 `rh-adapter`。

---

## 8. 用户体验要求

- 不让用户猜“到底提交了什么”
- 不让用户猜“图为什么头朝下”
- 不让用户猜“这个模型走的是哪个渠道”
- 不让用户误以为所有 NewAPI 模型都走 RH
- 不让用户面对技术名词堆叠
- 默认信息要足够少，但关键状态必须可见

---

## 9. 非目标

- 不做创作 studio 级重构
- 不做任务编排器
- 不做分镜编辑器
- 不做多模型联动
- 不做 Canvas 画布接入
- 不把直连 NewAPI / T8 / 火山 / 特朗普模型接入 rh-adapter

---

## 10. 验收标准

- 用户能在 3 秒内看懂当前模型走什么渠道，并能区分直连 NewAPI / T8 / 火山 / 特朗普与 RunningHub
- 用户能在生成前看到模型、比例、分辨率和参考图数量
- 用户能在上方媒体资产 / 画廊直接看见当前任务和结果摘要
- UI 没有明显复杂化
- 保留当前创作面板的工作台感，不破坏现有习惯

---
