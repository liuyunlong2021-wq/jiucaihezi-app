## 零、产品宪法（一切设计服从于此）

以下原则来自 `CLAUDE.md`，是产品不可妥协的边界。**任何违反这些的 SDD 提案都应被否决。**

```
P1. 纯手动显式控制：用户手动选 Skill / Knowledge / Tool / Model，AI 不自动决定任何东西。
P2. Skill 就是官方 Anthropic Skill（SKILL.md），不发明私有格式。
P3. Knowledge 独立于 Skill，进入 LLM 时只能作为 user-side evidence/context，不进 system role。
P4. Tool 默认关闭，用户显式开启后才暴露给 LLM。
P5. 当前用户输入是最高业务决策信号，优先级高于已选 Skill/Knowledge/Tool。
P6. 对话记忆 ≠ 知识库。知识库只接受用户手动添加，AI 不自动写入。
P7. ConversationContextEngine 是唯一上下文构建入口。不绕过它。
P8. Superpower/帮我配置只输出建议，用户确认后才进入执行。
P9. Workflow 不做独立模块；固定工作流内嵌在 SKILL.md 中。
P10. 画布和对话区共享同一个 Engine、vaultStore、agentStore。
```

## 一、问题诊断

### 1.1 根因分析

当前画布「乱」不是节点太多，而是**三个结构性问题**：

| 问题 | 表现 | 根因 |
|------|------|------|
| **产品灵魂不可见** | 画布里看不到知识库、Skill、工具 | 这些核心对象只在对话区可用，画布完全没有 |
| **节点无层级** | 42 个节点平铺，用户不知道从哪开始 | 没有语义分组、没有角色色、没有默认/高级区分 |
| **UI 无一致性** | LLM 极简 vs 图片重写 vs 视频臃肿 | 每个节点自己写样式，没有共享骨架 |
| **执行不完整** | 大部分图像处理节点是占位实现、无法停止 | T8 迁入遗留 + 无 AbortController |
| **画布-对话割裂** | 画布选的知识库/Skill 与对话区无关 | 两个独立系统各自为政 |

Grok SDD 的 Context Provider 节点方向正确，但有一个致命错误：**它把 VaultNode/SkillNode/ToolNode 设计成了模糊的「可连线实体」，没有明确它们是配置节点还是执行节点。**

### 1.2 核心修正

```
Grok 版:  VaultNode/SkillNode/ToolNode 可连线到 LLM，"连线即注入"
          ↓ 模糊：什么时候注入？谁执行召回？跟下拉选冲突了怎么办？

修正版:  Context Provider 节点是 SELECTOR（选择器），不是 EXECUTOR（执行器）
         它们没有 ▶ 按钮。
         它们的作用 = 在画布上可视化"这个 LLM 节点使用了哪些知识库/Skill/工具"
         连线 = 标记引用关系
         真正的召回/加载在执行时由 LLM 节点调用 Engine 完成
```

## 二、设计原则（Grok 8 条 + 修正 2 条）

以下原则取自 Grok SDD，经过审查和修正：

1. **手动显式控制至上**（保留，与 P1 一致）
2. **知识库、Skill、工具是第一视觉公民**（保留核心意图，但明确它们是选择器不是执行器）
3. **节点语义差异 > 视觉统一**（保留，但需要一个轻量共享 Frame 保底一致性）
4. **手感是 P0 基础设施**（保留）
5. **可组合性优于节点数量精简**（保留）
6. **复杂度用分组管理**（保留）
7. **执行必须可观测、可中断**（保留）
8. **渐进式披露 + 强预设**（保留）

新增两条：

9. **画布与对话区共享上下文源，但独立选择**  
   知识库列表、Skill 列表、工具列表来自同一个 Engine/vaultStore/agentStore。  
   但画布的 Context 选择和对话区的 Context 选择是独立的。  
   画布 Context 栏设定「画布默认值」，每个 LLM 节点可以覆盖。

10. **Context Provider 节点 = 选择器，非执行器**  
    VaultNode / SkillNode / ToolNode 没有 ▶ 按钮。  
    它们的作用是「标记引用关系」，让用户在画布上看到数据流。  
    真正的检索/注入在执行 LLM 节点时发生。

## 三、核心修正：Context Provider 节点的正确定义

这是与 Grok SDD 最大的分歧点，必须单列一节。

### 3.1 VaultNode（知识库选择器）

```
┌──────────────────────────────┐
│ 📁 产品文案库           [✕]  │ ← 浅紫色左边条（context 角色色）
├──────────────────────────────┤
│ 页面: [全部页面 ▼]          │
│ 钉选: 产品定位.md, FAQ.md   │
│                              │
│ 状态: 待 LLM 执行时召回      │  ← 这不是执行状态，是说明
└──────────────────────────────┘
    ↗ 右侧 source Handle: 可连到 LLM 节点的 left-context 端口
```

- **没有 ▶ 按钮**。VaultNode 不执行任何操作。
- 连线到 LLM 节点 = 声明「当这个 LLM 执行时，使用此知识库做召回」。
- 多个 LLM 节点可以连到同一个 VaultNode（共享引用）。
- 一个 LLM 节点可以连多个 VaultNode（多知识库联合检索）。
- 真正的召回由 LLM 节点在执行时调用 `ConversationContextEngine.build()` 完成。

### 3.2 SkillNode（Skill 选择器）

```
┌──────────────────────────────┐
│ 🧩 品牌设计指南        [✕]  │ ← 浅紫色左边条
├──────────────────────────────┤
│ 版本: v2.3                  │
│ 类型: 内置 Skill             │
│                              │
│ [双击查看 SKILL.md 内容]     │
│ 状态: 待 LLM 执行时注入      │
└──────────────────────────────┘
```

- **没有 ▶ 按钮**。
- 连线到 LLM 节点 = 声明「当这个 LLM 执行时，加载此 SKILL.md 为 system prompt」。
- 双击可预览 SKILL.md 内容（只读）。
- 执行时通过 `skillApplicability.ts` 做适用性判断。

### 3.3 ToolNode（工具开关集合）

```
┌──────────────────────────────┐
│ 🔧 工具集：日常创作     [✕]  │ ← 浅紫色左边条
├──────────────────────────────┤
│ [☑ 浏览器] [☐ Office]       │
│ [☐ 本地项目] [☐ 搜索]       │
│                              │
│ 状态: 待 LLM 执行时暴露      │
└──────────────────────────────┘
```

- **没有 ▶ 按钮**。
- 连线到 LLM 节点 = 声明「当这个 LLM 执行时，暴露这些工具定义」。
- ToolNode 可以命名和复用。

### 3.4 LLM 节点：接受两类上下文来源

LLM 节点仍然保留内部下拉选择器（兼容简单场景），同时增加 `left-context` 端口接受 Context Provider 连线。

**优先级规则（高→低）：**

```
1. 连线注入的 Context Provider（最高，显式连线 = 显式意图）
2. LLM 节点内部的下拉选择
3. 画布 Context 栏的默认值（最低，仅当 1 和 2 都没有时生效）
```

**去重规则：** 同一个知识库既被 VaultNode 连线引用、又在 LLM 下拉里选了 → 只注入一次。
**Tool 合并规则：** 连线 ToolNode 的工具 + LLM 节点内开关的工具 → 取并集。

### 4.2 Group 节点端口聚合规则（关键）

Group 折叠后对外暴露的端口由其**内部节点的未连接端口**自动聚合：

- **输入端口 (target)**：Group 内部所有节点的左侧 target Handle，如果**在 Group 内部没有连线来源**，则提升为 Group 的外部输入端口
- **输出端口 (source)**：Group 内部所有节点的右侧 source Handle，如果**在 Group 内部没有连线目标**，则提升为 Group 的外部输出端口
- 每个提升的端口保留原节点的 Handle ID 和边类型约束
- Group 展开时端口取消聚合，直接显示内部节点；折叠时恢复聚合

**示例：** 一个 Group 内含 text→llm→imageGen→imageResult，其中 text 无输入→提升为 Group 输入端口，imageResult 无输出→不提升（result 无 source Handle），llm 的 context 端口也未连接→提升为 Group 的 context-inject 输入。

### 4.3 Group 内部作用域规则

- Group 内部的 Context Provider（VaultNode/SkillNode/ToolNode）**只对同一 Group 内的 LLM 节点生效**
- Group 内部的 LLM 节点**不继承**画布 Context 栏的默认值（隔离原则）
- 外部 Context Provider 可以通过连线穿越 Group 边界（通过聚合端口）影响内部 LLM
- 导出为模板时，Context Provider 节点转为**占位符**（保留类型和名称，但具体选择清空），导入后由用户重新选择

## 五、节点详细规格

### 5.1 通用 Frame（所有节点共享的轻量骨架）

```
┌─────────────────────────────────┐
│ ● 角色色条(3px左边)             │  ← 6种角色色

├─────────────────────────────────┤
│                                 │
│    [各节点自己的语义内容区]       │  ← 身体 (slot，不强求一致)
│                                 │
├─────────────────────────────────┤
│ ⬅ 2输入  ⬆ 状态  [▶] [■] [✕]  │  ← 底部执行条（Context Provider 和 Result 无 ▶■）
└─────────────────────────────────┘
```

### 5.2 缩放机制（Phase 0 必须实现）

```
右下角手柄 ▦:
  pointerdown → 全局 body 加 .is-resizing
  pointermove → RAF 直接改 DOM width/height（不触发 Pinia 更新）
  pointerup   → 写入 node.data.width/height → 移除 .is-resizing

.is-resizing 下:
  - 所有节点 pointer-events: none
  - transition: none
  - 内容区冻结渲染
  - 松手后批量 flush
```

最小尺寸: 240×100px。双击头部恢复默认尺寸。所有节点数据新增 `width`, `height`, `collapsed` 字段。

### 5.3 Collapsed 机制

所有节点默认展开。用户点击头部 `[▼折叠]` → 节点收缩为仅显示头部 + 端口 + 状态：

```
┌──────────────────────┐
│ ● 🧠 AI大脑  [▲展开] │  ← 折叠态只保留头部
│ ⬅ 2入 →1出  idle     │
└──────────────────────┘
```

折叠态下内容区 `display:none`，不做任何渲染。保证大画布（30+ 节点）性能。

### 5.4 text 节点

```
┌──────────────────────────────────┐
│ ● 📝 文本标题              [▼][✕]│ 蓝色条
├──────────────────────────────────┤
│                                  │
│  [展开态] Tiptap 富文本编辑器     │  ← 复用 EditorPanel 的 Tiptap 配置
│  支持: H1-H4 / 粗斜体 / 列表     │     含 [[wiki-link]]、代码高亮、
│        引用 / 代码块 / 链接       │     KaTeX、Mermaid、表格
│        图片 / 表格 / 任务列表     │
│        [[双向链接]]              │
│                                  │
│  拖入 .md/.txt → 自动导入        │
│                                  │
├──────────────────────────────────┤
│ 1.2K字符 ≈450tokens  [✨润色] [▶] [✕] │  ← ▶ = AI续写/润色
└──────────────────────────────────┘
    ↗ 内容超过当前高度 → 自动 node.height += 行高
    ↗ 右下角拖拽 → 手动调宽高
```

**编辑模式：原地展开 Tiptap，不弹 popover。** 折叠态显示 Markdown 预览（marked 渲染）。

**性能约束（必须实现）：**
1. **单编辑态原则**：同一时刻画布上只有一个 TextNode 处于「展开编辑态」。其余全部 collapsed（Markdown 预览）。
2. **焦点离开降级**：用户点击画布空白区或切换到其他节点 → 当前编辑的 TextNode 自动 collapsed，保存内容。
3. **Tiptap 延迟初始化**：collapsed 态不创建 Tiptap 实例。仅在双击标题或点击「展开」时才 `new Editor()`。折叠时 `editor.destroy()`。
4. **重扩展按需加载**：Mermaid、KaTeX 等重扩展仅在展开态注册。collapsed 预览用 marked（<50KB）渲染。
5. **30 节点画布基准**：同时存在 4 个 TextNode（1 展开 + 3 collapsed）+ 其余 26 个节点时，展开/折叠切换 <200ms。

### 5.10 Group / Subgraph 完整设计（V8 最重要的新能力）

#### 5.10.1 折叠态

```
┌─────────────────────────────────────────┐
│ ● 📦 第1幕：角色建立    [▲展开] [▶组内] [✕]│ 琥珀条

│ 上次执行: 3分钟前 · ✓成功                 │
├─────────────────────────────────────────┤
│ [▶执行组内] [导出模板] [重命名]      [✕]  │
└─────────────────────────────────────────┘
```

#### 5.10.2 展开态

展开后显示内部所有节点，Group 变为半透明虚线框背景。组内节点可正常编辑和连线。

#### 5.10.3 端口聚合规则

| 规则 | 说明 |
|------|------|
| **输入聚合** | 内部所有节点的 target Handle，如果**在 Group 内部没有连线来源**，提升为 Group 外部输入端口 |
| **输出聚合** | 内部所有节点的 source Handle，如果**在 Group 内部没有连线目标**，提升为 Group 外部输出端口 |
| **端口命名** | 提升的端口保留原节点 Handle ID + 边类型。例: `llm1/right-text` (prompt-flow) |
| **动态更新** | Group 内部连线变化时，端口聚合自动刷新 |

#### 5.10.4 Context 作用域

```
Group 内部的 VaultNode/SkillNode/ToolNode:
  ✓ 对同一 Group 内的 LLM 节点生效
  ✗ 对外部 LLM 节点不可见（隔离原则）
  ✗ 不继承画布 Context 栏默认值

外部的 Context Provider 连线穿越 Group 边界:
  ✓ 可以通过聚合端口影响内部 LLM（显式连线优先）
  
Group 内部的 LLM 节点:
  - 不能通过内部下拉选择外部 Group 的 Context Provider
  - 但可以被外部 Context Provider 连线注入
```

#### 5.10.5 导出为模板

- Group → 「导出模板」→ 保存为 `.jcgrouptemplate` JSON 文件
- 模板内 Context Provider 节点 → 转为占位符（保留类型+名称，清空具体选择）
- 模板内 LLM 节点 → 模型选择保留，温度等参数保留
- 导入模板时 → 弹出「替换占位符」对话框，用户选择实际的 Vault/Skill/Toolset
- 模板存入节点库「③编排 → 我的模板」区

#### 5.10.6 独立执行

- 点击 Group 的「▶执行组内」→ 拓扑排序仅限 Group 内部节点
- 外部连线注入的输入作为「组执行输入」
- 执行状态显示在 Group 折叠态的底部状态栏

## 十、执行引擎

### 10.1 基础能力（V8 必须）

保留现有 `canvasExecutor.ts` 拓扑排序。核心场景是线性/简单 DAG。

**V8 新增三项：**

#### 10.1.1 AbortController 停止

```ts
const controllers = new Map<string, AbortController>()

async function runNode(nodeId: string) {
  const ac = new AbortController()
  controllers.set(nodeId, ac)
  try { await executeNode(nodeId, ac.signal) }
  catch (e) {
    if (ac.signal.aborted) setStatus('stopped')
    else setStatus('error')
  } finally { controllers.delete(nodeId) }
}
function stopNode(nodeId: string) { controllers.get(nodeId)?.abort() }
function stopAll() { controllers.forEach(ac => ac.abort()); controllers.clear() }
```

#### 10.1.2 子图执行

选中 Group 或多个节点 → 右键「执行选中」→ 拓扑排序仅限选中节点。外部连线注入的值作为输入。

#### 10.1.3 输入签名缓存（轻量版）

不追求 ComfyUI 级别的全节点缓存。只对以下情况做缓存：

- **媒体生成节点** (imageGen/videoGen/audioGen)：缓存「模型+prompt+参考图URL」的 SHA256 签名 → 相同输入直接复用上次 taskId 的结果 URL（不重新提交任务）
- **LLM 节点**：不做缓存（LLM 输出天然非确定性）
- **缓存有效期**：同一画布会话内有效，关闭后清空
- **缓存命中视觉**：节点执行时底部显示「⚡缓存命中，跳过生成」

#### 10.1.4 异步任务状态机

```
idle → submitting → polling → success
                    ↓ (stop)
                 cancelled
                    ↓ (error)
                   error → [重试]
```

媒体生成节点（imageGen/videoGen/audioGen）的异步轮询统一走此状态机。`canvasMediaRuntime.ts` 重构以适配。

### 10.2 增量执行（V9+）

以下能力 V8 不做，但在数据模型中预留字段（`lastExecutionInputHash`）：
- 全节点输入哈希 + 脏检查
- 增量重跑（只跑受影响下游）
- 执行轨迹可视化（连线上显示数据流向动画）

原因：当前用户规模和使用场景下，完整增量执行系统的 ROI 不足以支撑其 8-12 周的开发成本。

## 十三、已知风险与缓解措施

| 风险 | 影响 | 缓解 |
|------|------|------|
| VueFlow 大量自定义 Handle 时边渲染开销 | 30+ 节点画布拖拽掉帧 | collapsed 态不渲染 Handle；拖拽冻结时暂停边重计算 |
| Pinia 频繁更新连锁重渲染 | 缩放/拖拽时 flickering | 缩放过程用临时 ref，松手后批量 commit Pinia |
| 底部输入栏被滥用为「自动编排」 | 违背 P1 纯手动原则 | 两段式交互：AI 给建议 → 用户显式点「确认创建」；默认不自动执行 |
| 旧画布迁移数据丢失 | 老用户升级后画布坏了 | 迁移前自动备份原 JSON 到 `~/.jiucaihezi/data/backups/`；迁移向导支持「保留旧版(只读)」 |
| 向后兼容字段缺失 | 旧 `.jccanvas` 加载失败 | 缺失字段全部给默认值 + 静默降级；数据结构差异走迁移映射表 |
| Group 作用域混乱 | 用户搞不清知识库被谁用了 | Group 折叠态底部显示「内部生效: 📁产品文案库 🧩品牌指南」 |
| TextNode Tiptap 内存泄漏 | 长时间使用后内存飙升 | `editor.destroy()` on collapsed；全局最多保留 1 个 Editor 实例 |

## 十四、验收标准

- [ ] 打开画布 → 四节点起始模板 + 浮动提示卡
- [ ] 节点库 → 上下文区置顶（浅紫背景），核心展开，编排折叠
- [ ] VaultNode/SkillNode/ToolNode 无 ▶ 按钮
- [ ] VaultNode 连线到 LLM → LLM 底部显示「☑产品文案库」
- [ ] LLM 节点 Tab 切换正常，默认 📋摘要 Tab 紧凑
- [ ] LLM 节点同时连线 Skill + 下拉选 Skill → 自动去重
- [ ] LLM 节点连线 ToolNode + 内部开关 → 工具取并集
- [ ] 缩放 50 次 → 尺寸持久化，刷新恢复
- [ ] 30 节点画布缩放拖拽 50fps+
- [ ] TextNode 5000 字 → 自动增高不溢出；同时仅 1 个展开态
- [ ] TextNode 展开 → 折叠 → 再展开，无内存泄漏（Tiptap 正确销毁重建）
- [ ] 点 LLM ▶ → running → 点 ■ → stopped
- [ ] VaultNode 连线到 imageGen → 拒绝 + toast「知识库只能连 LLM」
- [ ] Group 折叠 → 端口自动聚合；展开 → 恢复内部节点
- [ ] Group 内部 VaultNode 不影响外部 LLM
- [ ] Group 导出模板 → 导入 → Context Provider 为占位符
- [ ] LLM 执行时 context-inject 边高亮 + 完成后短暂显示 token 贡献
- [ ] 媒体节点相同输入第二次执行 → 「⚡缓存命中」
- [ ] 旧画布打开 → 迁移向导弹出 → 一键升级后节点可用
- [ ] 画布 Context 栏切换 Skill → 对话区 Skill 不受影响
- [ ] 底部输入「生成一组产品图」→ AI 给建议 → 用户需点「确认创建」后才创建（两步，不能自动）
