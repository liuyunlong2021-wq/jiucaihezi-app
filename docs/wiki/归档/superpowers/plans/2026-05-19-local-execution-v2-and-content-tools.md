# 本地执行层 v2 与内容工具基础版

日期：2026-05-19

## 目标

让韭菜盒子 Studio 从“能调用少量本地工具”升级为“能更接近 Codex 的本地执行闭环”，同时提前补上普通用户强依赖的文档、音频、视频处理入口。

## 本轮交付范围

### 1. 开发执行层 v2

- 增加项目类型识别：识别 Node/Vue/Tauri/Rust/Python 等项目，并给出推荐检查命令。
- 增加项目文本搜索：在已选择源码项目内搜索关键词，自动跳过大目录。
- 增加批量读取文件：减少模型一轮一轮读取文件的成本。
- 增加精准替换：用 exact old/new 文本替换文件片段，避免整文件写入。
- 增加 diff 查看：优先读取 Git diff，便于模型和用户知道改了什么。

### 2. 本地内容与媒体工具基础版

- 增加上传附件文本提取工具：模型可以显式读取当前对话上传的文本/文档提取结果。
- 增加媒体元信息识别：音频/视频上传后先本地提取文件名、类型、大小、时长、视频尺寸等摘要。
- 增加媒体处理规划工具：遇到转码、压缩、抽音频、加字幕等需求时，先返回可执行处理计划和能力缺口，不伪造已完成结果。
- 工具仓库新增文档/音频/视频相关本地卡片。

## 非本轮范围

- 不开放任意 shell。
- 不做全盘文件访问。
- 不直接接入无确认的 ffmpeg 转码。
- 不改当前 App 图标。
- 不恢复复杂 Gateway/Provider 面板。

## 安全边界

- 源码项目工具只允许访问用户在工具仓库选择的项目目录内文件。
- 路径必须是项目内相对路径。
- 命令仍保持白名单和无 shell 执行。
- 文件替换必须命中 exact old_text；没有命中不写入。
- 媒体工具本轮只做本地元信息和计划，不伪造转码/转写能力。

## 验证

- 先写失败测试，再实现。
- 运行聚焦工具测试。
- 运行 `pnpm build`。
- 运行 `cargo check`。
