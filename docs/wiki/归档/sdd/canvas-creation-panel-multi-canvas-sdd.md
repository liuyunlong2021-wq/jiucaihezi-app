# 创作面板多画布与直接改图 SDD

> **状态**: 设计已确认，待实施
> **日期**: 2026-07-12
> **范围**: 桌面端 Tauri 创作面板
> **核心决策**: 文件树是画布唯一真源；创作面板直接复用现有媒体任务链路；OpenCode 不参与媒体生成和改图。

## 1. 背景

### 1.1 LeaferJS 对照来源

本画布不自行发明底层编辑行为。每次新增或修复画布能力前，先从以下仓库找到对应的官方 API、示例或实现，再做 Vue/Tauri 适配：

| 来源 | 地址 | 定位 |
|---|---|---|
| **LeaferJS 主仓库** | https://github.com/leaferjs/leafer | 核心渲染、节点、事件的事实源 |
| **Leafer UI** | https://github.com/leaferjs/leafer-ui | UI 节点、事件和画布交互 |
| **Leafer Draw** | https://github.com/leaferjs/leafer-draw | 箭头、画笔等绘制能力 |
| **Leafer Game** | https://github.com/leaferjs/leafer-game | 复杂交互场景的官方示例，按需参考 |
| **Leafer Editor** | https://github.com/leaferjs/leafer-editor | 选择、图层、编组、变换、编辑行为 |
| 社区完整画布示例 | https://github.com/liuyunlong2021-wq/huabu-LeaferJS | 仅用来参考组合交互，不作为底层事实源 |

执行规则：先找官方现成能力，直接按其 API 接入；官方没有时才写最小适配并记录原因。不得为了赶进度手写与 Leafer 编辑器平行的行为。

当前创作面板只有一个 `default` 画布。用户设计到一半后若要处理另一项任务，只能清空或覆盖当前内容，无法像切换项目文件一样暂停和恢复工作。

同时，当前方案曾计划通过 OpenCode、`JC-瞬间创作` 和 Canvas Bridge 完成改图。该链路重复了创作面板已经具备的模型选择、任务提交、轮询、下载和任务恢复能力，增加了不必要的进程边界与异步任务归属问题。

本方案将两件事分开：

- 创作面板负责媒体生成、画布选材引用和画布编辑。
- 文件树负责画布文件的创建、打开、重命名、复制和删除。
- OpenCode 未来只用于 Wiki 知识关系整理，不进入本期改图链路。

## 2. 目标

### 2.1 多画布

一个项目可以拥有多个独立画布。用户可以随时切换，之前画布上的图片、文字、箭头、画笔、编号、编组、位置和层级都能完整恢复。

### 2.2 画布选材直接创作

用户把图片或视频放入画布，在画布中选中一个或多个素材，在创作面板输入提示词并选择模型后，直接使用现有 `mediaTaskStore` 和媒体生成 API 提交任务。选中的素材就是模型参考；生成结果永远保留原素材，并追加在选中素材或素材组的右侧，方便并排比较。

### 2.3 本地文件真源

画布列表和画布内容全部来自项目目录中的文件。数据库和 localStorage 不维护第二份画布列表；删除应用缓存不会丢失画布。

### 2.4 职责边界

| 现有模块 | 本期职责 |
|---|---|
| `ProjectFileTree` | 展示和管理 `.jccanvas` 文件，点击后打开画布 |
| `canvasPersistence` | 扫描、读取、标准化和原子写入画布文件 |
| `canvasStore` | 保存当前画布路径和纯 JSON 场景，串行化保存与任务写回 |
| `CreationPanel` | 呈现 Leafer 场景、画布工具和媒体参数，不持有后台任务生命周期 |
| `mediaTaskStore` | 沿用现有提交、轮询、恢复和本地化流程，成功后调用画布写回 |

不新增第二套媒体执行器、画布索引数据库或通用命令总线。

## 3. 非目标

- 不让 OpenCode、Skill、MCP 或 Python 脚本参与本期媒体生成和改图。
- 不建设通用 Canvas Bridge。
- 不实现多人协作、云端同步和画布冲突合并。
- 不实现 Web 端多画布；Web 端没有桌面项目文件树。
- 不实现画布文件夹、画布专属媒体目录或媒体自动清理。
- 不实现画布缩略图缓存；文件树第一期使用画布图标和文件名。
- 不把 Wiki 知识节点混入媒体画布 schema；知识画布另行设计。

## 4. 用户体验

### 4.1 新建画布

用户可以从两个位置执行同一个命令：

1. 文件树 `jc-canvas` 目录右键“新建画布”。
2. 创作面板标题栏点击“＋”。

两个入口都在磁盘创建新的 `.jccanvas` 文件，并立即在创作面板打开。默认文件名依次为：

```text
未命名画布.jccanvas
未命名画布 2.jccanvas
未命名画布 3.jccanvas
```

### 4.2 切换画布

用户点击文件树中的 `.jccanvas` 文件：

1. 完成当前画布待保存内容的落盘。
2. 保存成功后清理当前 Leafer 场景和撤销栈。
3. 读取目标文件并恢复完整场景。
4. 创作面板标题显示目标文件名。

保存失败时不切换，并向用户显示错误，避免当前编辑内容丢失。

### 4.3 管理画布

`.jccanvas` 文件在文件树支持：

- 打开
- 重命名
- 复制
- 删除
- 在 Finder/资源管理器中显示

重命名直接重命名磁盘文件，标题随文件名变化。复制画布时生成新的 `canvasId`，场景内容和媒体引用保持不变。

删除画布只删除 `.jccanvas` 文件，不删除任何 `jc-media` 素材。删除当前画布后打开同目录中最近修改的另一张画布；没有剩余画布时创建新的空白画布。

### 4.4 画布选材直接创作

画布中的选中素材就是创作参考：

```text
选中一个或多个图片/视频
  → 创作面板按视觉顺序把支持的素材作为参考
  → 输入提示词并选择模型
  → 提交现有 mediaTaskStore
  → 结果下载到 jc-media
  → 写回提交时绑定的画布并追加在素材组右侧
```

原始素材绝不自动替换或删除。未选素材时，生成结果加入当前画布的可用空白位置。

## 5. 文件结构

```text
{projectDir}/
├── jc-media/
│   ├── images/
│   ├── videos/
│   └── audio/
└── jc-canvas/
    ├── 海报方案一.jccanvas
    ├── 海报方案二.jccanvas
    └── 紧急海报.jccanvas
```

- `jc-media` 保存媒体字节。
- `jc-canvas` 保存画布场景文件。
- 画布文件只引用项目内媒体路径，不保存 Base64、Blob URL 或媒体字节。
- 画布文件名是用户看到的画布名称，不在 JSON 内重复保存 `name`。

## 6. 画布文件契约

`.jccanvas` 是 UTF-8 JSON 文件，使用专用扩展名，避免与 Obsidian JSON Canvas 的 `.canvas` 格式混淆。

```ts
interface CanvasDocumentV2 {
  version: 2
  canvasId: string
  updatedAt: number
  viewport: {
    x: number
    y: number
    zoom: number
  }
  scene: LeaferNodeJson[]
  assets: Record<string, CanvasAsset>
}

interface CanvasAsset {
  id: string
  kind: 'image' | 'video'
  path: string
  source: 'creation' | 'drop' | 'paste' | 'import'
  model?: string
  prompt?: string
  parentAssetId?: string
  createdAt: number
}
```

### 6.1 ID 规则

- 新建画布使用 `crypto.randomUUID()` 生成 `canvasId`。
- 每个可编辑根节点拥有稳定 node ID。
- node ID 只要求在所属画布内唯一。
- 复制画布时只更换 `canvasId`，节点 ID 保持不变。

### 6.2 媒体序列化

Leafer 运行时可以使用 data URL 或解析后的本地显示 URL，但保存前必须经过标准化：

1. 图片和视频节点都通过 `assetId` 关联 `assets`。
2. `assets.path` 只保存相对项目路径，例如 `jc-media/images/abc.png` 或 `jc-media/videos/abc.mp4`。
3. `scene` 中不得出现 `data:`、`blob:`、图片 Base64 或视频字节。
4. 恢复画布时，再将 `assets.path` 解析为 Leafer 可显示的运行时 URL。

从剪贴板或项目外部拖入的媒体，必须先复制到对应的 `jc-media/images/` 或 `jc-media/videos/`，再加入画布。

### 6.3 不持久化的状态

以下状态属于临时 UI 状态，不写入 `.jccanvas`：

- 当前选中的节点
- 当前工具
- 右键菜单状态
- 撤销/重做内存快照
- 正在打开的参数弹窗

应用可以使用现有本地偏好记录“项目最后打开的画布路径”。该值只是导航缓存，丢失后通过扫描 `jc-canvas` 恢复，不是画布真源。

## 7. 保存与切换

### 7.1 自动保存

- 场景发生有效变化后 500ms debounce 保存。
- 平移或缩放结束后同样 debounce 保存，保证重新打开时恢复工作视角。
- 切换画布、关闭创作面板和应用退出前立即 flush。
- 同一画布同时只能有一个保存 Promise；后续保存复用或排队，不能并发覆盖。
- 内容和视角都没有变化时不写文件。

### 7.2 原子写入

保存采用同目录临时文件后 rename：

```text
海报方案一.jccanvas.tmp
  → 写入并关闭
  → rename 为 海报方案一.jccanvas
```

第一期只有当前应用写画布，不需要 lock、revision、轮询或冲突合并。原子写入用于防止崩溃和断电造成半个 JSON 文件。

### 7.3 打开规则

进入创作面板时按以下顺序选择画布：

1. 打开该项目最后使用且仍存在的画布。
2. 否则打开 `jc-canvas` 中最近修改的画布。
3. 目录为空时创建 `未命名画布.jccanvas`。

切换项目后必须重新扫描新项目的 `jc-canvas`，不得沿用上一个项目的当前画布。

## 8. 媒体任务与画布绑定

现有媒体任务是异步执行。用户提交任务后可能立即切换画布，因此任务必须记录提交时的写回目标。

```ts
interface CanvasTaskTarget {
  canvasId: string
  canvasPath: string
  operation: 'append'
  referenceNodeIds: string[]
  referenceBounds?: { x: number; y: number; width: number; height: number }
}
```

`MediaTask` 增加可选 `canvasTarget`。创建任务时写入并随现有任务数据一起持久化。任务完成事件只提供 `taskId`，消费者从任务记录读取 `canvasTarget`，不得使用“当前打开的画布”推断写回位置。

任务成功并完成本地媒体保存后，`mediaTaskStore` 调用画布持久化模块的单一结果写入函数。该函数负责判断目标是否为当前画布并完成写回；不能依赖 `CreationPanel` 的事件监听器正好处于挂载状态。组件只负责显示当前画布，不承担后台任务生命周期。

### 8.1 写回当前画布

如果任务目标仍是当前打开画布：

- 通过 `canvasStore` 修改当前纯 JSON 场景，由 `CreationPanel` 同步到 Leafer。
- 在引用素材或引用素材组右侧追加结果。
- 进入当前撤销栈。
- 触发正常自动保存。

### 8.2 写回后台画布

如果用户已经切换到其他画布：

- 读取任务绑定的 `.jccanvas`。
- 依据任务保存的引用素材组边界追加结果。
- 原子写回该文件。
- 不切换用户当前画布。
- 在任务历史显示“结果已写入：画布名称”。

### 8.3 目标失效

任务完成前可能发生以下情况：

- 目标画布被删除。
- 全部引用素材被删除。

处理规则：

- 结果媒体仍保存在 `jc-media`，不能丢弃。
- 不猜测新的追加位置。
- 将任务标记为“生成成功，未写入画布”。
- 用户可以从任务历史执行“加入当前画布”。

首期不实现多个任务对同一素材组的自动排版合并。后完成的任务仍追加到提交时记录的位置；若全部引用节点已删除，按目标失效处理。

## 9. 创作面板交互

### 9.1 标题栏

创作面板标题栏显示：

```text
创作面板 · 海报方案一                         ＋
```

- 名称来自当前 `.jccanvas` 文件名。
- 点击名称在文件树中定位当前画布。
- `＋` 创建新画布文件。
- 不在标题栏建设第二套画布列表。

### 9.2 画布选材

画布是创作面板唯一的媒体引用入口；移除创作面板提示词输入框上的附件上传控件。聊天输入框的附件能力不受影响。

- 单击空白区域取消选中。
- `Shift + 左键` 切换单个图片或视频素材的选中状态。
- 从空白区域拖拽可框选多个素材。
- 多选引用顺序固定为视觉顺序：从左到右、再从上到下。
- 当前模型支持的素材类型才会作为参考提交；不支持的素材保持选中，但在提交前给出明确提示。
- 图片和视频均可选作参考。视频按模型能力表作为 `referenceVideos` 提交。

### 9.3 素材进入画布

创作面板不再提供附件上传。素材通过以下三个画布入口进入，且都先复制到项目 `jc-media`，再写入画布引用：

1. 从 Finder/资源管理器拖拽图片或视频到画布。
2. 画布工具栏的“导入素材”图标。
3. 项目文件树中图片或视频右键“加入画布”；单击媒体文件也加入当前画布。

剪贴板图片或视频在画布获得焦点后粘贴，沿用同一导入路径。

### 9.4 新生成媒体

- 单选一个素材：结果依次追加在该素材右侧，间距 `24px`。
- 多选一组素材：结果追加在该组整体边界的右侧，间距 `24px`。
- 未选素材：结果加入当前画布的可用空白位置。
- 图片结果作为图片镜框加入画布；视频结果作为视频参考节点加入画布。
- 原始素材永远不自动替换、不删除。
- 任务失败不修改画布。

### 9.5 极简视觉规范

**提示词输入框**

- 创作面板提示词框与对话输入框使用同一套表面、边框、圆角、内边距、初始高度和聚焦态规则，初始高度必须视觉齐平。
- 提示词框是完整的圆角容器，不再是没有边框和圆角的裸 `textarea`。
- 只复用现有主题 CSS 变量，不写固定浅色背景或浏览器原生控件样式，亮色和暗色主题都必须一致。

**图片与视频参考节点**

- 每个图片和视频节点默认有 `1px` 主题边框、`6px` 圆角和与素材同尺寸的裁切区域，像一张简洁的照片镜框。
- 选中时只将镜框改为主题强调色，并增加克制的外沿，不使用白色控制点和粗大的紫色包围框作为主要视觉。
- 缩放、旋转等编辑控制只在用户实际编辑时短暂显示；未编辑时不遮挡素材。媒体只显示四个角控制点，默认始终等比缩放，不显示四边拉伸控制点。
- 视频节点立即显示文件名和播放标识，后台读取首帧后显示静态封面；首帧失败仍保留可选节点。

### 9.6 工具栏与对象操作

右侧工具栏只保留高频操作：导入、选择、箭头、文字、画笔、编号、撤销、重做、删除、适应画布、放大和缩小。

“更多”菜单收纳低频对象操作：

- 图层：上移一层、下移一层、置顶、置底。
- 对象：编组、解组、锁定、解锁。
- 变换：左转 90 度、右转 90 度、水平翻转、垂直翻转、向左倾斜、向右倾斜。

右键菜单提供同一批对象操作。多选时图层操作作为整体移动，组内顺序不变。所有对象操作进入现有撤销栈并自动保存。

不实现导出和独立图层面板：项目文件夹与 `.jccanvas` 已是本产品的本地真源，导出没有明确工作流价值。

### 9.7 视频参考节点与预览

- Leafer 官方当前版本没有可直接采用的浏览器视频节点实现；本产品仅使用官方 `Group`、`Rect`、`Image`、`Text` 组合静态参考节点，不接入画布播放器或逐帧渲染。
- 视频节点的位置、缩放、旋转和层级与原视频路径持久化到 `.jccanvas`；运行时首帧不保存 Base64，重开时重新读取。
- 单击选中并作为视频模型参考；双击打开应用内原生 `<video>` 预览；右键可用系统播放器打开。

## 10. 文件树行为

`ProjectFileTree` 将 `.jccanvas` 识别为画布文件：

- 使用画布图标。
- 单击在创作面板打开，不进入文本编辑器。
- 重命名限制为当前项目 `jc-canvas` 内，保留 `.jccanvas` 扩展名。
- 新建、复制、删除完成后刷新现有文件树，不维护额外列表缓存。

用户手动在 Finder 中复制或重命名 `.jccanvas` 后，刷新文件树即可看到结果。若复制文件保留了相同 `canvasId`，应用首次打开副本时为副本生成新 `canvasId` 并保存，避免任务写回歧义。

项目媒体文件不再默认用系统程序打开：

- 单击图片或视频，将该媒体加入当前画布。
- 右键图片或视频提供“加入画布”。
- 双击视频参考节点在应用内预览；右键“用系统播放器打开”作为备选。

## 11. 旧数据迁移

当前旧画布位于：

```text
{projectDir}/.jiucaihezi/canvas/default.json
```

首次打开新版本时：

1. 若 `jc-canvas` 已有画布，不执行迁移。
2. 若旧 `default.json` 存在，转换为 `jc-canvas/未命名画布.jccanvas`。
3. 将旧 `layers[]` 和 `annotations[]` 转换为完整 `scene[]`。
4. 成功写入并验证可读取后，将旧文件保留为 `default.json.v1.bak`。
5. 若没有旧文件，创建空白画布。

迁移失败时保留原文件并显示错误，不创建看似成功的空画布覆盖用户预期。

## 12. 错误处理

| 场景 | 处理 |
|---|---|
| 画布 JSON 损坏 | 不覆盖原文件；提示无法打开，并允许在文件树中定位 |
| 自动保存失败 | 保留内存场景；显示持续错误；切换前再次保存 |
| 项目目录只读 | 禁止新建和保存，画布保持只读显示 |
| 引用媒体不存在 | 保留节点和路径，显示缺失占位，不删除节点 |
| 后台任务目标失效 | 媒体保留，任务历史提供“加入当前画布” |
| 删除最后一张画布 | 删除确认后创建新的空白画布 |

## 13. 实施阶段

### Phase 0：可靠场景文档

- 实现 `CanvasDocumentV2` 和 scene serializer。
- Base64/data/blob URL 转项目媒体引用。
- 完整保存和恢复所有现有 Leafer 元素。
- 原子保存、自动保存去重和旧数据迁移。

验收：重启后图片、文字、箭头、画笔、编号、编组、位置、层级和视角一致；画布 JSON 不含媒体字节。

### Phase 1：文件树多画布

- 创建并扫描 `jc-canvas`。
- `.jccanvas` 文件打开、新建、重命名、复制和删除。
- 创作面板显示当前画布名称和新建快捷按钮。
- 切换项目和切换画布前可靠 flush。

验收：三个海报画布可以独立编辑并任意切换，关闭应用后继续上次内容。

### Phase 2：画布选材直接创作

- 画布单选、`Shift + 左键` 多选和空白拖拽框选。
- 移除创作面板附件上传和“替换原图 / 新增一版”控制。
- 选中图片/视频按视觉顺序作为现有媒体任务参考输入。
- 画布工具栏导入、外部拖入、画布粘贴和文件树“加入画布”统一走项目媒体导入。
- `.jccanvas` 资产支持图片与视频；视频作为静态参考节点恢复。
- `MediaTask` 持久化 `canvasTarget` 和引用素材组边界，当前画布和后台画布按任务目标安全追加结果。
- 提示词框对齐对话输入框；图片和视频使用主题镜框选中态。
- 低频图层、对象和变换操作收纳到“更多”；视频首帧和双击预览可用且持久化正确。

验收：在画布一选中图片或视频提交后切换到画布二，任务完成仍只在画布一的引用素材组右侧追加；原素材不变；亮色和暗色主题均保持统一镜框与输入框外观。

### Phase 3：独立规划 Wiki 画布

当 Wiki 关系图的节点类型、交互和输出格式明确后，单独编写 SDD。届时只评估知识关系导入所需的最小接口，不复用媒体改图链路，也不预设 MCP。

## 15. 决策记录

- 2026-07-12：文件树和项目本地文件是画布唯一真源。
- 2026-07-12：一个项目允许多个 `.jccanvas` 画布文件。
- 2026-07-12：创作面板顶部新建按钮只是文件操作快捷入口，不形成第二套画布列表。
- 2026-07-12：媒体生成和改图直接复用现有 `mediaTaskStore`，不经过 OpenCode 或 Skill。
- 2026-07-12：异步任务绑定提交时画布，不按完成时当前画布写回。
- 2026-07-12：OpenCode/Wiki 画布联动独立规划，本期不建设 Canvas Bridge 或 MCP。
- 2026-07-12：画布是创作面板唯一媒体引用入口；附件上传从创作面板移除，聊天附件不受影响。
- 2026-07-12：生成结果一律追加，原始图片和视频不自动替换；多个引用素材按视觉顺序提交。
- 2026-07-12：创作提示词框对齐对话输入框；图片和视频采用统一、主题适配的极简镜框与选中态。
- 2026-07-12：视频使用静态参考节点和应用内原生预览，不接入 `leafer-canvas-player` 或画布逐帧播放。
