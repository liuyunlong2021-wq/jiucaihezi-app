# 创模式核心原则一致性巡检

> 日期：2026-07-21  
> 范围：现行 Wiki（`架构/`、`开发/`、`学习/`、`hot.md`）；`归档/` 仅作历史，不参与修正  
> 用户确认的唯一原则：**创模式以模型原生能力为第一优先，产品能力补位，Skill 和工具按需增强；工具不能成为模型原生能力的门槛。**

## 现行风险

1. **❌ 必修｜架构入口缺少总原则**  
   `架构/产品架构.md` 只把创模式描述为“直连工具循环”，没有规定模型原生能力、产品补位和工具增强的优先级。后续开发容易把工具循环误当成创模式本体。

2. **❌ 必修｜统一 SDD 前后自相矛盾**  
   `开发/创作模式双端统一SDD.md` 第 1.2 节写明“工具不是发送门槛”，但第 1.1、2、3、7、8 节又把受控终端和媒体 Skill 写成视频理解的主要道路，并允许“只有摘要”长期保持。这会让实现用工具替代模型原生媒体输入。

3. **❌ 必修｜当前实现缺口被误读为目标设计**  
   同一 SDD 的真实测试只证明视频元数据进入模型，并未证明原视频进入模型。该事实必须保留，但应明确标为实现缺口，不能作为最终合同或能力声明。

4. **🔻 重要｜MCP 工具被描述为无条件全量进入**  
   `开发/创模式MCP工具接入SDD.md` 把“已连接工具自动进入三种模式工具池”写成目标，没有模型 function calling 能力和当前任务适用性前置条件。工具数量和协议不兼容可能反过来破坏直连请求。

5. **🔻 重要｜产品与用户说明缺少一致优先级**  
   `开发/创作工作台架构SDD.md`、`开发/韭菜盒子原生媒体编排能力SDD.md`、`学习/韭菜盒子Studio-创作Agent详细介绍-2026-07-20.md` 分别描述工作台、生成和 Agent，但没有统一说明“原生模型 → 产品补位 → Skill/工具增强”。
