# Web 创作面板控制台红字排障

> 日期：2026-07-20
> 状态：三项前端根因已修复，RunningHub Web CORS 已在服务器生效；前端修复需重新发布 Web 后验收。

## 结论

APP 能调用、Web 不能调用，不代表创作服务坏了。APP 不受浏览器跨域和页面 CSP 约束；Web 必须同时满足浏览器安全策略、跨域预检和公网 API 路由。

本次控制台三类红字来自三条独立链路，不应通过隐藏报错处理：

| 现象 | 根因 | 最小修复 |
|---|---|---|
| CSP 拒绝执行首页 inline script | 启动保护代码直接写在 `index.html`，不符合线上 CSP | 移到静态文件 `public/boot-guard.js`，首页只用 `src` 引用；Web/Desktop 产物审计同时允许并要求该文件 |
| `/api/creation/models` 返回 Pages 404 | Web 使用网站同源相对地址，请求被发到 Cloudflare Pages，而真实接口在 API 域名 | Web 与 APP 都调用 `DEFAULT_API_BASE_URL + /api/creation/models` |
| 打开画布时项目图片路径出现网站根目录 404 | 持久化场景的图片节点保留了项目相对 `url`，Leafer 恢复时先把它当网页 URL 请求，早于 OPFS Blob 水合 | 持久化图片节点不再保存 `scene.url`；真实路径只保留在 `assets[id].resource.path`，恢复后再生成可显示的 Blob URL |

此外，RunningHub `app-info` 的 Web 请求曾被浏览器 CORS 预检拦截。服务器已在 `/api/runninghub/` 代理层处理允许来源和 `OPTIONS`；验证请求返回 `204`，并由本机 Nginx access log 确认请求实际到达服务器。详见 [[运维/服务器运维#RunningHub Web CORS（2026-07-20）]]。

## 为什么 APP 正常、Web 报错

```text
APP 请求
  -> 桌面运行时直接访问公网 API
  -> 不经过浏览器 CORS 预检

Web 请求
  -> 浏览器检查页面 CSP
  -> 跨域时先发送 OPTIONS
  -> 服务器必须明确允许当前网站 Origin
  -> 预检通过后才发送真实 GET/POST
```

因此，不能用“APP 能用”推断“Web 一定能用”。Web 发布验收必须单独查看浏览器 Network 和 Console。

## 修改边界

- 启动保护的行为未变：仍收集早期脚本错误，并在超时或加载失败后移除启动屏；只改变脚本承载方式。
- 创作模型接口的数据合同未变，只纠正 Web 的请求目标。
- 画布资源身份未变：项目相对路径仍由 `assets` 保存，`scene` 只保存视觉状态，不把项目文件路径当网站资源地址。
- CORS 只允许正式站点和 `*.jiucaihezi.pages.dev` 预览站，不开放任意来源。
