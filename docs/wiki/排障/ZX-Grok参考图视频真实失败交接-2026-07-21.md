> 日期：2026-07-21

## 复现

1. 在创模式中选择一张已生成图片作为参考图。
2. 选择 `Grok 1.5 Video 6s`，点击生成。
3. 真实返回仍为：

```text
HTTP 400: {"code":"invalid_json","message":"json: cannot unmarshal object into Go struct field .Alias.image of type string","data":null}
```

## 已确认事实

- 第一次真实失败是旧路由 `POST /api/creations/uploads` 返回 404。
- 后续真实失败为上述 400，后端明确表示它收到的 `image` 是对象，而不是字符串。
- 产品客户端提交入口是 `POST /v1/videos`，不是 ZX 上游 `/v1/videos/generations`。
- 以下修复已在代码和 mock 测试中通过，但没有通过真实付费验收：
  - `79ddbddc`：ZX 强制跳过已删除的上传路由。
  - `86690b7e`：ZX 运行时请求体写入字符串 `image`。
  - `4b17c0cf`：`apiCall('/v1/videos')` 序列化前将历史 `image: { image_url }` 归一为字符串。
- 开发环境中已强制刷新 Desktop WebView，但用户再次测试仍报同一 400。因此不得再根据 mock 测试声称修复已完成。

## 接手排查顺序

1. 先抓真实请求体：在 Vite `/__jc_api` proxy 或 Desktop WebView 网络层记录 `POST /v1/videos` 的 JSON 字段类型，不记录图片 data URL 全文。
2. 如果 proxy 收到的 `image` 已是字符串，排查 `api.jiucaihezi.studio` 及 NewAPI 渠道是否在转发前重写成对象。
3. 如果 proxy 收到的 `image` 仍是对象，反向追踪实际调用点：`mediaTaskStore._executeTask()` -> `creationMediaRuntime.executeDirectVideoRequest()` -> `apiCall()`。不要再只改某一个请求构造器。
4. 抓到证据后，为实际请求形态先写红色回归测试，再修复。

## 关键代码入口

- `src/components/chat/ChatPanel.vue` 负责将已确认的媒体计划交给创作面板。
- `src/runtime/workbench/mediaPlanBridge.ts` 将计划转为 Creation 提交参数。
- `src/stores/mediaTaskStore.ts` 是任务单一执行入口。
- `src/runtime/creation/creationMediaRuntime.ts` 构建 ZX 直连视频请求。
- `src/api/media-generation.ts` 中的 `apiCall()` 是最后客户端 JSON 序列化边界。
- `vite.config.ts` 把开发态 `/__jc_api` 转发到 `https://api.jiucaihezi.studio`。
