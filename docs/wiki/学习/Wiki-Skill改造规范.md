# Wiki Skill 改造规范

本规范用于把 Wiki Skill 改造成 Studio 的模型能力外挂。用户只说目标，模型根据内容、项目状态和工具自主决定做什么；Skill 负责提供能力、边界和可验证的输出标准，而不是把专业流程和选择题交给用户。

## 1. 名称与目录、发现

- `SKILL.md` 的 `name`、目录名、注册表 `id` 与 `name` 必须完全一致，且使用 1-64 个小写字母、数字和连字符。
- `description` 以 `Use when ...` 开头，覆盖用户自然会说的目标词，不要求用户记忆 Skill 名称或流程术语。
- 删除欢迎闸门和能力菜单。用户目标明确时直接工作；只有事实冲突、不可逆操作、权限不足或目标确实不明时才提问。

## 2. 模型自主决策

每个 Wiki Skill 必须明确：**用户只表达目标，模型自主判断方法。**

1. 读取用户内容、现有 Wiki、Raw 与项目状态。
2. 判断内容语境和信息状态，而不是要求用户选择“小说模式”“关系图模式”。
3. 从可用能力中组合必要动作；不需要的能力不做。
4. 按能力标准输出、验证并报告实际结果。

Skill 可以给出“何时有价值”的判断信号，但不能把它写成“用户说某口令才允许做”。模型的判断不等于随意：事实来源、安全边界、输出格式和验证要求必须固定。

## 3. Reference 两层结构

复杂 Wiki Skill 的 Reference 必须分为两层，主 `SKILL.md` 只保留路由和共用边界。

| 层级 | 作用 | 示例 |
|---|---|---|
| `项目语境/` | 说明一类项目有哪些知识、已有结构如何理解、什么成果有价值；同类项目应合并 | 开发项目、故事项目、通用资料 |
| `能力标准/` | 说明模型决定做某项能力后，输入、输出、质量和安全边界 | 内容填充、来源索引、双链、热缓存、Canvas、Bases、资料摄入、开发收尾 |

- 项目语境不替模型决定能力；能力标准不假定单一行业目录。
- 一次任务可命中一个项目语境和多个能力标准。
- 新能力先写输出标准，再接入主路由；不能只在主 Skill 提一句“可以生成”。

### 唯一事实源

- 同一个项目语境只能维护一份 Reference。建库、填充、查询、巡检与修正 Skill 都读取这同一份，不能各自复制一套“开发项目”或“小说项目”。
- 项目 Reference 必须区分“建库标准”和“Raw 填充标准”：前者规定目录骨架和接管边界，后者规定确认事实怎样落位；不要把两者拆进不同 Skill 的重复 Reference。
- 每个主 Skill 都要有明确的“类型 -> Reference”路由。识别类型后先读对应 Reference；无法判断才询问用户，不能退化为模糊的通用模式。

## 4. 主 Skill 的写法

主 `SKILL.md` 必须清楚写出：

1. **功能说明与边界**：解决什么、不解决什么。
2. **用户入口**：用户可用自然语言表达的目标。
3. **自主路由**：模型如何从内容判断项目语境、证据来源和能力组合。
4. **不可破坏规则**：事实、权限、原文、已有结构与冲突处理。
5. **Reference 索引**：项目语境和能力标准各自的入口。
6. **验证与报告**：实际写入、验证命令和未验证项必须如实报告。

每项能力标准也必须明确 **输入**（模型可读取的内容、项目状态、前置产物）与 **输出**（实际写入的文件、格式化产物或可检查报告）。

不要把小说、开发、Canvas 等细节堆进主 Skill；也不要用固定线性步骤压制模型根据任务组合能力。

## 5. Raw 与派生 Wiki

- Raw 是原始过程与证据，保留在真实位置；Wiki 是结论、结构和导航。
- Raw 不复制、不移动、不删除。Studio 创模式使用 `.raw/sessions/jcses_*`；Studio 文/武使用 OpenCode `ses_*`；VS Code 使用当前项目工作区的 `chatSessions/`。
- 重要 Wiki 结论必须可通过 `来源索引.md` 回到 Raw、原始文件、SDD、代码、Git 或测试证据。
- 模型可根据证据决定要不要补双链、Canvas、Bases 或热缓存；一旦生成，必须符合对应能力标准。
- 不得创建 `对话记录/`、会话摘要页或聊天转录页来复制 Raw。用户明确交付或确认的正式作品正文可以进入其正式正文目录，但不能把含讨论的整段会话搬进去。

## 6. 写入与安全

- 先接管已有 Wiki；不存在时由模型调用建库 Skill 建立骨架，再继续填充。禁止创建平行目录或假装已写入。
- 只写有来源的结论。已有正文不整篇覆盖；内容冲突、正式稿判定、合并、移动、删除必须向用户确认。
- 开发项目新建 Wiki 默认在 `docs/wiki/`；旧 `wiki/` 项目保持原位，不自动迁移。
- 语义判断由模型完成；确定性脚本用于盘点、生成格式化产物或验证。

## 7. 验证与留痕

改造前先补会失败的契约测试，至少覆盖：

- 名称、目录和注册表一致；
- 主 Skill 明确“用户说目标、模型自主判断”；
- 项目语境与能力标准两层 Reference 存在且能被主 Skill 路由；
- 每个可选能力有模型决策条件和输出标准；
- Raw 来源索引、已有 Wiki 安全和通用回退路径；
- 脚本行为、语法、注册表、旧路径检索和 `git diff --check`。

从旧 Skill 改造时，契约测试还必须验证：旧目录、旧名称、旧 Reference 路径和旧 Skill 引用均已清零；注册表只列出实际存在的文件；旧的启动闸门、欢迎语和 A/B/C 菜单已删除。

测试通过只代表合同与脚本通过；没有进行的模型质量或人工验收必须明确标为未验证。
