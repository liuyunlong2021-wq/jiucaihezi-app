# Pi Agent Harness 与创模式学习分析

> 调研日期：2026-07-21  
> 上游仓库：[earendil-works/pi](https://github.com/earendil-works/pi)  
> 代码基线：`c8c3cd499f4d35c0f9cfebfec5f4e3822411a49f`（2026-07-20）  
> 结论性质：学习笔记，不是改造任务或技术方案

## 先说结论

Pi 和韭菜盒子的创模式确实像：两者都让模型根据任务决定是否调用工具，Skill 都是“先给目录、需要时再读全文”。但 Pi 是给开发者拼装 Agent 的底座，创模式是面向创作工作的完整产品。

所以最值得学的不是 Pi 的终端界面，也不是把它接进来，而是它把四件容易缠在一起的事分得很开：**运行循环、会话事实、上下文、可扩展能力**。创模式已经在媒体、MCP、项目文件、双端和按次授权上走得更深，不能为了“像 Pi”退回一个通用编程 CLI。

## Pi 到底是什么

Pi 是 MIT 许可证的 TypeScript monorepo。它不是一个单一聊天程序，而是四层产品：

| 层 | 包 | 用大白话说 |
| --- | --- | --- |
| 模型层 | `pi-ai` | 抹平 OpenAI、Anthropic、Google 等模型接口差异，也管理模型目录和认证。 |
| Agent 层 | `pi-agent-core` | 负责“发给模型 -> 模型要工具 -> 执行工具 -> 把结果发回模型”的循环。 |
| 编程产品层 | `pi-coding-agent` | 终端交互、默认读写/命令工具、会话、压缩、Skill、Extension、项目包管理。 |
| 界面/进程层 | `pi-tui`、`orchestrator` | 前者画终端 UI，后者提供进程集成与编排能力。 |

它的核心路线是：核心只保留少量默认能力，其他东西由 Skill 或 Extension 自己长出来。Pi 明确不内置 MCP、子 Agent、计划模式、待办清单、权限弹窗或后台命令；需要的人自己用 TypeScript Extension 做，或者装一个包。

这是一种刻意的取舍，不是“它还没来得及做”。

## 它的一次工作是怎么跑的

```text
用户消息
  -> 取当前会话分支，按需要加入摘要和项目说明
  -> 模型流式回复
  -> 有工具调用：校验参数、发出开始事件、执行工具、持续回报进度、发出结束事件
  -> 工具结果写入会话，再回到模型
  -> 没有工具调用：本轮结束
```

Pi 的 Agent Core 把这件事做成明确事件流：消息开始/增量/结束，工具开始/增量/结束，回合开始/结束，整个 Agent 开始/结束。工具默认可并行执行；有顺序依赖时，工具可要求整批改为串行。工具调用前后还有统一钩子，前钩子可以拦截，后钩子可以整理结果。

这让界面、审计、授权、统计不必各自猜“现在是不是在运行”“工具到底成功没有”。它们只消费同一套运行事实。

## 它的几个关键设计

### 1. Skill 是说明书，不是运行时插件

Pi 启动时只扫描 `SKILL.md` 的名字和描述，把一个很小的 Skill 目录放进系统提示词。模型任务匹配后，用普通 `read` 工具读完整 `SKILL.md`，再按其中的相对路径调用脚本、参考资料或资产。

这和创模式的渐进加载原则高度一致。差异是：

- Pi 把 Skill 当作可读文件，默认不限制 Skill 内脚本对本机的能力。
- 创模式把 `skill(name)`、声明资源白名单、项目根目录和用户授权接在一起。

后者更适合有媒体、文件和桌面执行能力的产品。不要为了模仿 Pi 而去掉创模式的 Skill 资源边界。

### 2. 会话不是一条聊天记录，而是一棵可回去的树

Pi 的会话是 JSONL 追加日志。每条记录有 `id` 和 `parentId`，当前会话只是树上的一个叶子。用户可以回到任意历史点继续，或者复制一条分支到新会话；旧路不会被覆盖。

上下文压缩也不删除原记录：它追加一条“旧部分摘要 + 从哪条记录开始保留”的记录。模型下次只看到摘要和最近消息，用户仍能从树里查看原始全过程。切换分支时，Pi 还可以把离开那条分支的关键进展另做摘要，带到新分支。

对于“先出三个创意方向，再把其中一个做成视频/画布/文案”的创作工作，这比线性重试更贴合真实工作方式。

### 3. 扩展点足够强，但也足够危险

Pi 的 Extension 是本地 TypeScript 代码，可以注册工具、命令、快捷键、UI 部件和事件处理器；还能在工具调用前拦截、修改上下文、接管压缩、保存自己的会话状态。它相当于把产品二开权限给了用户。

Pi 也坦白其边界：Extension 和默认 `bash` 都拥有启动 Pi 的用户权限；项目“信任”只决定是否加载项目内的扩展、Skill、配置，不是沙箱，也不能抵御项目文件里的提示注入。

这是开发者 CLI 可以接受的哲学，但不是创模式应照搬的安全模型。

### 4. 上下文压缩有明确的事实合同

Pi 在接近模型上下文上限时，会保留最新一段完整对话，概括更早内容。摘要固定涵盖：目标、限制、已完成/进行中/受阻、关键决定、下一步，以及读过和改过的文件。它还能在溢出后压缩并重试。

重点不在“让模型写摘要”，而在于摘要和原始会话都有明确归属、保留边界和 token 预算。这样压缩是可观察、可追查的会话行为，不是 UI 背后悄悄丢上下文。

## 与创模式逐项对照

| 维度 | Pi | 当前创模式 | 判断 |
| --- | --- | --- | --- |
| 核心定位 | 通用编程 Agent harness，主要是 CLI | 创作工作台，涵盖 Skill、Wiki、项目文件、媒体和画布 | 目标不同，不能把 Pi 当作替换品。 |
| 工具循环 | 统一事件流、可并行/串行、前后钩子 | 共享 Direct Engine，工具循环、取消、12 轮上限；工具过程在 UI 折叠呈现 | 方向一致；Pi 的运行事件合同更成熟。 |
| Skill | 描述入上下文，模型用 `read` 按需读正文 | 描述目录 + `skill(name)` 按需读；有效注册表和资源边界统一 | 创模式的安全与双端合同更强，应保留。 |
| 会话 | JSONL 事件树，原地分支、fork、压缩记录 | 创作会话与 OpenCode 会话隔离，按项目存储 | 创模式已解决隔离；缺少的产品能力是可控分支，而不是换 JSONL。 |
| 上下文 | 自动/手动压缩，保留原始历史，摘要带结构化事实 | 按模型容量保留最新完整对话，并注入项目 `CLAUDE.md` 和 `wiki/hot.md` | Pi 的“摘要可追溯”值得研究；不能让 App 越权自动改 Wiki。 |
| 权限 | 没有内建权限弹窗，建议容器化 | 项目根边界、绝对路径需用户批准、终端按任务授权 | 创模式明显更适合普通用户。不要降级成 Pi 的全权限模式。 |
| 扩展 | 任意 TypeScript Extension 和 npm/git 包 | 内置/本机 Skill、MCP、受控终端与现有面板 | 可以学“稳定事件接口”，不应直接开放任意前端/本机代码。 |
| 多媒体 | 非重点 | 图片/视频/文件、媒体生成、画布均是一等能力 | Pi 几乎不能提供这方面的产品经验。 |

## 最值得吸收的经验，按优先级排列

### A. 把创模式的运行过程收敛成稳定事件合同

最值得借鉴。创模式已有工具状态和折叠摘要，但后续每加一种工具（MCP、媒体、终端、文件）都应只产生同一类事实，例如：调用已请求、等待授权、已开始、进度更新、成功、失败、已取消。UI、会话持久化、统计、重试和双端合同测试从这些事实派生。

收益：能减少“某个页面自己判断工具完成”的竞态，也更容易做真实的 Web/Desktop 结果级测试。

边界：这不是新增全局事件总线。仍应遵守现有 SDD 的原则，在公共运行时回调边界收敛，而非复制 Pi 的整套框架。

### B. 把“创作分支”做成明确的用户工作流候选项

创作任务天然有探索：换脚本方向、换视觉风格、把同一素材做成不同版本。Pi 的会话树给出的不是技术炫技，而是一个清晰的产品动作：从历史节点另开一条路，原来的路保留。

值得先验证的最小体验是“从某条用户指令复制为新创作会话”，而不是先实现复杂树控件、跨项目分支或自动分支摘要。只有真实创作流程证明用户需要频繁回溯，再考虑树内导航和分支摘要。

### C. 让上下文压缩成为可见、可验证的会话记录

当前“按模型容量保留最新完整对话”的方向正确，但长期创作时仍会遇到容量问题。可学习 Pi 的三个原则：

1. 原始消息永远留在会话事实中；压缩只改变本次给模型看的窗口。
2. 摘要必须记录目标、关键决定、已产出资产、待办和失败原因，不能只是泛泛总结。
3. 摘要来源、覆盖范围、token 用量和失败状态都可查。

创模式特有的摘要还应包括：项目文件/画布资产引用、媒体任务 ID、用户授权状态。它不应自动写进 `wiki/hot.md`，因为那会把一次对话的临时状态污染成项目长期事实。

### D. 为工具执行建立“授权前”和“执行后”的唯一边界

Pi 的 `beforeToolCall` / `afterToolCall` 钩子说明了一个简单事实：安全校验、用户确认、结果脱敏和审计不能散在每个工具实现里。

创模式已经有更好的按任务授权规则。未来若需要补强，应优先把已有规则统一放到公共工具执行边界：所有能改文件、跑命令、触达外部 MCP 的调用都经过同一个“授权判定 -> 执行 -> 规范化结果”的路径。不要引入通用可执行 Extension 来解决它。

### E. 保持核心小，但把产品边界定得更硬

Pi 的“核心小”并不是代码少，而是核心只承诺少数稳定协议。创模式可以吸收这个方法：媒体编排、Skill 注册、MCP、文件工具、会话、画布之间都用已定义的合同互通，避免某个面板直接侵入另一个面板的内部状态。

不过“少”不意味着砍掉创作特性。创模式的差异化正是项目、媒体、画布和知识库能在同一任务里闭环。

## 明确不建议照搬的部分

1. **不要把 Pi 当作创模式的运行时依赖。** 它的默认工具和 TUI 是编程代理取向；接入会制造第二套模型、会话、工具和权限链路，直接违背创模式的单一共享发送链路。
2. **不要取消 MCP。** Pi 的“No MCP”是作者为 CLI 选择的生态立场。创模式已有用户连接 MCP Server 的产品能力，且它服务的并不只是本地命令行工具。
3. **不要开放任意 Extension 执行。** Pi 把扩展视为同一台开发机上的可信代码；创模式面对的是更广泛用户和更丰富外部资源，必须继续坚持项目边界、按次授权和可见记录。
4. **不要复制 JSONL 作为目的。** Pi 选择 JSONL 是为了 CLI 可移植、可追加、可手工查看。创模式先要解决的是会话语义和产品交互；存储介质应服从现有 Web/Desktop 数据边界。
5. **不要引入 Pi 没有的“通用 Agent 平台”幻想。** 子 Agent、计划模式、待办等在 Pi 都是有意留空的。创模式同样应该先针对具体创作任务补事实能力，而不是先搭一套万能框架。

## 建议的学习顺序

先读这四块源码即可，不需要先通读整个二十多万行 TypeScript 仓库：

1. `packages/agent/src/agent-loop.ts`：模型与工具循环、并行/串行、取消和事件。
2. `packages/agent/src/harness/skills.ts`：Skill 扫描与渐进披露，和创模式最直接对应。
3. `packages/agent/src/harness/session/`：树形会话的最小数据模型；重点看 `parentId`、叶子节点与 fork。
4. `packages/coding-agent/src/core/compaction/` 与 `session-manager.ts`：压缩如何不破坏历史，并让摘要变成可追溯记录。

第二轮再看 `packages/coding-agent/src/core/extensions/`。它适合学习“扩展接口为什么要围绕生命周期事件设计”，但不适合作为创模式开放权限的模板。

## 一句话落点

Pi 最有价值的启发是：把 Agent 的每次行动都当成可持久化、可回放、可拦截的运行事实；把业务能力放在这个稳定骨架之上。创模式应吸收这根骨架，同时保留并继续强化自己已经具备的创作媒体能力、双端统一和用户权限边界。
