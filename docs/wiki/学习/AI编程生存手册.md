# AI 编程生存手册

> 写给不写代码但用 AI 做产品的人。每个概念都用你项目里的真实 bug 解释。
> 每修一个 bug，AI 会在 commit 里标注「常识 #X」，你翻到这里就能看懂。

---

## 一、日常高频（每次改代码都可能遇到）

### #1 竞态 (Race Condition)

> 两个操作同时跑，谁先完成不一定。后完成的可能覆盖先完成的结果。

**你的真实 bug**：登录时 APP 还没完全启动就去读 Key，读回来是空的——因为「启动」和「读 Key」在赛跑。

**遇到什么症状**：同一个操作，第一次成功第二次失败；或者快点成功，慢点失败。

**告诉 AI**：「这可能是竞态，检查时序」

---

### #2 超时 (Timeout)

> 一个请求发出去，等了很久没回应，不能永远等下去。

**你的真实 bug**：图片生成 HTTP 请求没有超时，网络卡了用户永远转圈。后来设了 300s 超时。

**遇到什么症状**：转圈停不下来，不报错也不完成。

**告诉 AI**：「这个网络请求需要超时，设 X 秒」

---

### #3 持久化 (Persistence)

> 数据在内存里——关 APP 就没了。写到硬盘——关 APP 还在。

**你的真实 bug**：登录状态存在 Keychain 里，但 Keychain 读取超时 8s，Intel Mac 慢盘经常超时，用户关了 APP 就要重登。后来改用 CLI 文件为唯一持久化源。

**遇到什么症状**：关了 APP 再开，登录没了、会话没了、设置丢了。

**告诉 AI**：「这个状态需要持久化」

---

### #4 回退 (Fallback)

> A 方案在某些环境不生效，准备一个 B 方案兜底。

**你的真实 bug**：CSS `-webkit-box` 在旧版 Safari 不生效，文字溢出。加了 `display: block` 回退。

**遇到什么症状**：新功能在 M 芯片 Mac 上好好的，到旧电脑/Windows 上就炸。

**告诉 AI**：「加个 fallback，兼容旧环境」

---

### #5 同步 vs 异步 (Sync vs Async)

> 同步 = 一件事做完才做下一件。异步 = 发出去不等结果，先做别的，结果回来了再处理。

**你的真实 bug**：`initApiKey()` 是异步的，但 `boot()` 没有 `await` 它——结果后面的代码在 Key 还没读到就开始跑了。

**遇到什么症状**：页面渲染完了但数据还是空的，过一会又突然出现。

**告诉 AI**：「这可能需要 await」

---

## 二、理解后能省很多事（中频）

### #6 前端 vs 后端

> 前端在你电脑上跑（界面），后端在服务器上跑（数据）。

| | 前端 | 后端 |
|---|---|---|
| 韭菜盒子在哪 | `src/` (TypeScript/Vue) | `src-tauri/` (Rust) + 服务器 (NewAPI/Go) |
| 出问题表现 | 界面卡、白屏、样式乱 | API 报错、数据不对 |
| 你怎么判断 | 看在 APP 里能不能看到 | 看 Network 面板 / 服务器日志 |

**告诉 AI**：「先判断是前端还是后端的问题」

---

### #7 状态 (State)

> 程序记住的「当前是什么情况」。一个变量就是一个状态。

**你的真实 bug**：`activeOpenCodeSessionId` 是模块级变量，切换项目时忘记清空，导致会话泄漏到别的项目。

**遇到什么症状**：切了一个页面/项目后，数据和上一个混在一起。

**告诉 AI**：「切换时状态没有清理」

---

### #8 缓存 (Cache)

> 把算过的结果存起来，下次直接用。快，但可能过期。

**你的真实 bug**：工具仓库扫描结果缓存了 5 分钟，用户装了新工具看不到——因为缓存还没过期。

**遇到什么症状**：明明改了/装了东西，界面上还是旧的。

**告诉 AI**：「是不是缓存没刷新？」

---

### #9 序列化 (Serialization)

> 把内存里的对象变成字符串（JSON），存到硬盘或发给服务器。反之叫反序列化。

**你的真实 bug**：媒体任务状态存 SQLite 时，`JSON.stringify` 丢了一些字段，读回来就少了。

**遇到什么症状**：数据存进去再读出来，少了东西或变了样。

**告诉 AI**：「检查序列化/反序列化是否丢失字段」

---

### #10 类型安全 (Type Safety)

> TypeScript/Rust 在你写代码时就检查「这个东西真的有这个属性吗」。

**你的动作**：每次改完跑 `pnpm exec vue-tsc -b`。零错误 = 至少没有拼错属性名。

**告诉 AI**：「跑一下类型检查」

---

## 三、偶尔会遇到但很重要（低频高伤害）

### #11 原子操作 (Atomic)

> 要么全做完，要么全不做。没有「做了一半」的中间状态。

**例子**：用户付费流程——扣了钱但没开通会员 = 灾难。必须保证「扣钱」和「开通」是一个原子操作。

**告诉 AI**：「这两个操作需要原子性」

---

### #12 死锁 (Deadlock)

> A 等 B，B 等 A，谁都动不了。

**例子**：两个 Rust 锁互相等——`session.lock()` 等 `operation.lock()`，`operation.lock()` 等 `session.lock()`。

**遇到什么症状**：程序突然卡住，不报错也不继续，只能强退。

**告诉 AI**：「可能死锁了，检查锁的获取顺序」

---

### #13 内存泄漏 (Memory Leak)

> 用完了不释放，内存越占越多，越来越慢直到崩溃。

**例子**：每次请求新建一个定时器但忘了 clearInterval，定时器堆积。

**遇到什么症状**：APP 刚开很流畅，用久了越来越卡。

**告诉 AI**：「检查是否有内存泄漏（定时器/事件监听没清理）」

---

### #14 幂等性 (Idempotency)

> 同一个操作执行一次和执行十次，结果一样。不会重复扣款、重复创建。

**例子**：用户狂点「发送」按钮，如果没做防重复，会发出去 10 条一样的消息。

**遇到什么症状**：用户操作一次但出现了两次效果。

**告诉 AI**：「加个防重复/幂等保护」

---

### #15 CORS (跨域)

> 浏览器安全策略：一个域名的网页不能随便调另一个域名的 API，除非服务器明确允许。

**你的真实 bug**：本地开发时 `localhost` 调 `api.jiucaihezi.studio` 被浏览器拦截，需要 Vite proxy 绕过去。

**遇到什么症状**：浏览器 Console 里看到 `CORS error` 或 `blocked by CORS policy`。

**告诉 AI**：「CORS 错误，需要配 proxy 或服务器允许」

---

### #16 环境变量 (Environment Variable)

> 同一份代码，在不同环境跑的时候需要不同的配置。开发环境连本地数据库，生产环境连线上数据库。

**例子**：`OPENCODE_CONFIG_CONTENT` 环境变量决定 OpenCode 连哪个 AI 提供商。

**遇到什么症状**：「我本地好好的，部署上去就不行了」。

**告诉 AI**：「检查环境变量/环境差异」

---

### #17 构建 vs 运行 (Build vs Run)

> 你写的代码（源码）→ 编译/打包（构建）→ 用户能跑的程序（运行）。

- `pnpm tauri dev` = 开发模式，改了代码自动刷新
- `pnpm tauri build` = 构建，打包成 `.dmg` 安装包
- 类型检查（`vue-tsc -b`）是构建的一部分

**遇到什么症状**：开发模式正常，打包后不正常。

**告诉 AI**：「开发正常但构建后异常」

---

### #18 依赖 (Dependency)

> 你用了别人的库（npm install），那个库又用了别的库……一层层下去。

**关键常识**：
- `package.json` = 前端依赖清单
- `Cargo.toml` = Rust 依赖清单
- `pnpm-lock.yaml` / `Cargo.lock` = 精确锁定每个依赖的版本

**告诉 AI**：「升级 XX 库会不会有 breaking change？」

---

### #19 版本号 (Semver)

> `v1.2.3` = 主版本.次版本.补丁版本
> - 补丁 (1.2.3 → 1.2.4)：修 bug，放心升
> - 次版本 (1.2.3 → 1.3.0)：新功能，一般兼容
> - 主版本 (1.2.3 → 2.0.0)：不兼容的大改动，小心

**告诉你**：韭菜盒子 `v1.2.6-dev` = 主版本 1，次版本 2，补丁 6，`-dev` 表示开发中。

---

### #20 日志 (Logging)

> 程序在后台跑的时候留下的文字记录。出问题时看日志比盲猜快 100 倍。

**韭菜盒子的日志在哪**：
- 前端：浏览器 Console（Tauri 里按 Cmd+Option+I）
- Rust 后端：`eprintln!` 输出到终端
- 服务器：`docker compose logs new-api`

**告诉 AI**：「在这里加个日志，出问题时能追踪」

## 四、AI 协作专属

### #21 上下文窗口 (Context Window)

> AI 一次能「看到」的对话长度是有限的。太长的对话，前面的内容会被「遗忘」。

**你的对策**：
- 对话长了就开新对话，把 AGENTS.md + SDD 丢给新对话
- 不要在一个对话里跨多个不相关的任务
- Commit + 更新 SDD = 把「对话记忆」转为「文件记忆」

---

### #22 幻觉 (Hallucination)

> AI 自信满满地说了一个不存在的方法名、不存在的库、不存在的功能。

**你的对策**：
- 「有没有现成的库？」→ 让 AI 搜 GitHub/npm，不要信它脑子里的
- 「这个 API 存在吗？」→ 让 AI 读源码/文档确认，不要信它编的
- 所有 AI 说的「你可以用 XX」都要经过类型检查验证

---

### #23 渐进式披露 (Progressive Disclosure)

> 不要一次性把所有信息丢给 AI。先给最重要的，AI 需要时再给细节。

**你的 AGENTS.md 就是这个原理**：
- 开头 = 快速参考（最重要）
- 中间 = 架构详情（按需读）
- 末尾 = 附录（深度排障用）

**告诉 AI**：「详细方案在 docs/sdd/XX.md」

## 五、OpenCode/Session 专项（0712 排查实战新增）

### #24 Schema 校验 400

> API 端点有严格的字段白名单。你多传一个字段 → 400，而且可能**顺带损坏你的 session**。

**真实案例**：`PUT /session/:id` 只接受 `title/metadata/permission/time`。我们塞了 `model` 字段 → 400 → session 损坏 → 后续全 404。

**识别**：控制台 400 后面紧跟着同一 session ID 的 404。

**告诉 AI**：「去 OpenCode 源码查这个端点的 Schema，只传它接受的字段」

### #25 伪终端事件

> 事件名带 "error" ≠ 操作结束了。`session.error` 是「运行中出了个问题已记录」，操作还在继续。

**真实案例**：`session.error`→`finalizeOpenCodeRun('error')` 导致文模式消息生成完了被提前掐断消失。真正表示结束的只有 `session.idle`。

**识别**：消息流式输出中途/刚完成闪现消失。控制台 `session.error` 比 `session.idle` 先到。

**告诉 AI**：「session.error 不是终端事件，不要 finalize。真正的终点只有 session.idle」

### #26 源码对照法

> 这个项目是 OpenCode 的翻译版。任何行为不确定时，**先读 OpenCode 源码**，不要猜。

**今天三个坑全用这方法查出来的**：
1. `part_` 前缀 400 → 读 schema 发现校验 `prt_`
2. `updateOpenCodeSessionModel` 400 → 读 update handler 发现只接受 4 字段
3. `session.error` 该不该 finalize → 读 `prompt_async` handler 发现只是 catch→publish

**告诉 AI**：「去 `/Users/by3/Documents/jiucaihezi-opencode/packages/` 搜 XX，对照我们的实现」

### #27 SQLite WAL 不是临时垃圾

> SQLite 的 `-wal` 可能包含已经提交、但还没合并回主数据库的数据。运行中或异常退出后直接删 `-wal/-shm`，会让刚写入的数据消失。

**真实案例**：OpenCode sidecar 切换时没有等待旧进程退出，又手工删除共享数据库的 WAL/SHM。第一条消息刚创建的 session 丢失，于是后续请求全部 404。

**告诉 AI**：「不要手删 SQLite WAL/SHM；先确保旧进程退出，让 SQLite 自己恢复和 checkpoint」

### #28 唯一真源不是“定期覆盖”

> 同一份消息同时存在 Server、临时事件 Map、Vue messages 和 IndexedDB，再靠完成时全量拉取覆盖，看起来有备份，实际会互相打架。

**真实案例**：OpenCode 已保存正确 user text part，但旧 UI 先手拼消息、结束后再覆盖；切模式或重同步时“你好”消失或变成别的 part 内容。

**正确做法**：Server 事件只进入一个 Sync Store，Vue 只做响应式投影；乐观消息使用与请求相同的 ID，官方确认后原位合并。

**告诉 AI**：「先找唯一真源；不要用第二份快照修补第一份状态」

### #29 模型能力不等于通道协议结果

> 模型会编程、模型支持工具调用、当前 API 通道返回标准 `tool_calls`，是三件不同的事。一次响应没有工具 part，只能证明这次链路没形成标准工具调用，不能证明模型没有能力。

**真实案例**：`deepseek-v4-flash` 一次把 DSML 当普通文本返回，我们就把模型硬编码成 `tool_call:false`。对照 OpenCode 官方目录后发现它明确支持工具调用；撤销黑名单后，真机成功执行 `search + read`。

**告诉 AI**：「先查官方模型能力，再抓通道原始响应；不要用关闭能力掩盖协议适配问题」
