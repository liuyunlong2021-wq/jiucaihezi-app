# GitHub 推送与发布边界

> 盘点日期：2026-07-20
> 结论：GitHub 推送、Web 发布和桌面 App 打包是三件不同的事；都不应以“整个工作区”为单位操作。

## 三个动作

| 动作 | 实际输入 | 当前项目的出口 |
| --- | --- | --- |
| GitHub push | 已 `git add` 且已提交的文件 | `git push origin main`；未暂存文件不会被上传 |
| Web 发布 | 构建生成的 `dist/` | `pnpm run build` 通过后，`npx wrangler pages deploy dist --branch=main` |
| 桌面 App 发布 | CI 从 tag 检出的源码构建 | `git tag -a vX.Y.Z -m "vX.Y.Z"` 后 `git push origin vX.Y.Z` 触发三平台 CI |

`node_modules/`、本机构建缓存、`src-tauri/target/` 和未被 Git 追踪的文件不会因 push 自动上传；`dist/` 也不会进入桌面安装包。反过来，Git 已追踪的文件即使后来加入 `.gitignore`，仍会在下一次提交中继续进入 GitHub，直到显式从索引移除。

## 本次真实盘点

### 应保留在仓库

- 产品源码和资源：`src/`、`public/`、`src-tauri/`（不含忽略的 `target/`、`gen/`、`binaries/`）、根目录构建配置与 `scripts/`。
- 可复现依赖描述：`package.json`、`pnpm-lock.yaml`、`Cargo.toml`、`Cargo.lock`。
- 产品所需服务源码：`gateway/`、`rh-adapter/`、`attachment-processor/`、`document-converter/`。
- 项目文档和 Wiki：`docs/`；是否公开其产品介绍、PPT、PDF 由仓库公开策略决定，不影响安装包内容。

### 不应再进入 GitHub

- `.raw/sessions/`：当前 3 个会话文件共约 4.2 MB，其中 2 个已经被 Git 追踪。它们是个人会话原始材料，不是产品源码；当前产品决策也明确禁止 App 自动生成这类账本。
- `.git.backup/`：当前约 252 MB、7,356 个已追踪文件，包含另一份 Git 对象库和本机编辑器索引；它不是产品内容，也显著增大仓库。
- `.superpowers/`：当前已有 20 个已追踪的本机工作流状态文件；不属于产品运行所需内容，应停止继续提交。

`.env.local`、`node_modules/`、`dist/`、`src-tauri/target/`、`jc-media/`、`.venv/` 和 `.worktrees/` 当前均被忽略，正常不会被新的 Git 提交带入。`.raw/`、`.git.backup/`、`.superpowers/` 尚未被项目 `.gitignore` 覆盖，不能依赖“全量 push”来保护它们。

## 发布前门禁

当前 `dist/` 含 `.DS_Store`，`pnpm run audit:web-dist` 失败。不得发布该旧目录；先运行完整 `pnpm run build`，确认末尾的 `audit:web-dist` 通过，再部署。

桌面发布也不是直接 push `main`：先确认完整门禁通过，再推新版本 tag。GitHub Actions 会从该 tag 的干净源码检出后构建安装包。

## 推荐操作顺序

1. 先把个人会话、`.git.backup/` 和 `.superpowers/` 从 Git 跟踪中移除，并补充忽略规则；已有提交历史中的敏感会话若需彻底消失，另行执行历史清理。
2. 逐个检查 `git status` 中的源码、测试、Wiki 和脚本；只 `git add` 本次确实要交付的文件，再提交并推送。
3. Web：运行 `pnpm run build` 成功后发布 `dist/`。桌面：完成发布门禁后推版本 tag。
