# 画布开发与排障

> 适用范围：创作面板内的 LeaferJS 无限画布。最后更新：2026-07-14。

## 一句话架构

画布不是独立产品：它是创作面板的素材工作台。项目文件树中的图片/视频进入画布，用户选中素材后直接作为创作模型的参考；生成结果也自动回到画布，继续比较、标注和二次创作。

```
项目文件树 / 拖放 / 粘贴 / 生成完成
  -> CreationPanel.addMediaToCanvas()
  -> LeaferJS App.tree
  -> .jccanvas 场景文件
  -> 重新打开后 restoreCanvasScene()
```

## 保存模型

每张画布是项目目录下的一份文件：`jc-canvas/*.jccanvas`。这是唯一存档，不另建隐式画布数据库。

文件中有两部分：

- `assets`：图片/视频的项目相对路径、类型、来源、提示词、模型等可引用信息。
- `scene`：Leafer 节点快照，保存位置、尺寸、旋转、缩放、箭头、文字、画笔和编号等视觉状态。

保存由拖拽结束后的 500ms 防抖触发；`Cmd/Ctrl+S` 调用 `flushCanvasSave()` 立即写入。切换画布前也会先保存当前画布。

## 素材与创作

- 点击项目文件树中的图片或视频，会加入画布并切换到创作面板。
- 新媒体进入画布后自动回到选择模式并选中，确保可以立刻拖动。
- 选中一个或多个素材后，创作面板依据模型能力把图片/视频作为参考提交。
- 生成成功的图片或视频自动加入当前画布；它们和导入素材走同一个 `addMediaToCanvas()` 入口。
- 视频在画布中显示首帧、标题和播放按钮；点击播放使用原生视频预览，不在 Leafer Canvas 内自建播放器。

## 工具行为

常用工具：选择 `V`、箭头 `A`、文字 `T`、画笔 `B`、编号 `N`、撤销/重做、删除、缩放、整理媒体并适应窗口。

- 文字和编号必须使用 Leafer 的 `getPagePoint()`，否则在缩放/平移后会出现“鼠标与落点错位”。
- 画箭头和画笔保持连续绘制模式；按 `V` 或 `Esc` 回到选择模式。
- 新媒体进入时也会自动回到选择模式，避免绘制模式抢走拖动事件。
- 变换默认等比缩放：`lockRatio: true`。
- 图层上移/下移、置顶/置底、旋转、翻转、倾斜放在“更多”和右键菜单。

## 视口与布局规则

放大和缩小永远围绕当前视口中心。不要直接改 `zoomLayer.scale`，必须调用 `setCanvasViewportScale()`，同时修正 `zoomLayer.x/y`。

“整理媒体并适应窗口”有明确职责：

1. 只重排图片和视频，居中网格布局。
2. 箭头、文字、画笔、编号不移动，因为它们没有可靠的“归属素材”关系，猜测会破坏标注。
3. 适应范围只计算媒体，不让远处的笔迹把所有素材压成小点。
4. 后续从文件树加入的素材读取现有媒体边界，放在最右侧、顶部对齐，不再回到旧的固定坐标。

## 已解决问题与根因

### 1. 打开画布失败，报 `destroyed`

旧场景可能含 Leafer 运行时节点 `SimulateElement`。`UI.one()` 无法恢复该节点，继续 `tree.add()` 会触发 `undefined.destroyed`。恢复时必须跳过该节点和恢复失败的节点；保存时也过滤它。

### 2. 删除/保存后旧标注又回来

上一个恢复错误会令 `canvasReady` 一直是 `false`，于是删除和自动保存都不会执行。先保证恢复失败可控，后续保存链路才成立。

### 3. 生成图变成“幽灵”，不可选、不可移动、不可删除

画布文件里旧生成图的 `editable`、`draggable` 缺失；左侧原图是 `true`，右侧生成图为空。恢复媒体节点时，对未锁定的 `assets` 节点强制设置：

```ts
restored.set({ editable: true, draggable: true })
```

这是一项兼容旧 `.jccanvas` 文件的迁移。不要对箭头/文字/编号统一强设，且必须尊重用户锁定的对象。

### 4. 新生成/导入素材无法拖动

箭头、画笔、编号等绘制工具会全局接管拖动事件。媒体进入画布时必须先 `canvasTool('select')`，让新素材可直接操作。

### 5. 适应窗口后，新图片出现在很远的位置

旧算法按固定的左上角网格投放素材；整理后媒体已移动到画布中心，因此新素材会脱离布局。新算法读取当前媒体的右边界，放在其右侧。

### 6. 文件树缩略图导致首次打开创作面板卡顿

一批可见图片会解码，视频还要读取首帧；若它们与画布恢复同时并发，会抢同一 WebView 主线程。文件树缩略图必须：

- `IntersectionObserver` 只发现可见行；
- 空闲时才开始；
- 单通道队列，`MAX_CONCURRENT_THUMBNAILS = 1`；
- 加载失败退回文件图标。

不要把缩略图预加载成“打开项目就全量生成”。

### 7. 切换项目后，定向生成结果写进了新项目

画布定向任务提交时必须冻结完整目标：`owner`、`canvasId`、`canvasPath`。其中 Desktop 的 `owner` 是绝对 `projectDir`，Web 的 `owner` 是 `webProjectId`；不能在异步完成、轮询恢复或事件派发时重新读取全局当前项目。

- `CreationPanel` 在任何参考素材 URL 的 `await` 之前快照这三个字段，并把 `owner` 持久化到 `CanvasTaskTarget`。
- Desktop 的 `downloadAndPersistMediaAsset()` 用 target owner 调 `writeProjectMedia()`；`writeCanvasResult()` 用同一个 owner 校验相对路径、调 `writeCanvasTaskResult()`，并在 `canvas:task-result` 中携带它。
- 正常完成与刷新后的 `_resumePolling()` 共用上述两个 helper，禁止另写一条恢复专用的当前项目路径。
- 旧持久化任务可能没有 `owner`。这类定向任务必须标记 `canvasWriteStatus = 'unwritten'`，不能回退到当前项目目录，也不能修改当前画布。

回归覆盖必须模拟“任务在 A 提交，完成前切到 B”，并同时覆盖正常完成、恢复轮询和无 owner 的旧任务。

### 8. 生成结果与旧画布快照互相覆盖

问题不是“最后一次保存”本身，而是两个异步写入都从不同时间点的内存快照出发：任务完成后需要读改写 `.jccanvas`，同时用户的防抖保存、项目切换、文件重命名或组件卸载可能把旧场景排到它之后。后写的旧快照会删掉刚生成的媒体。

现在只保留一个按 `owner:path` 分区的持久化队列。`writeCanvasTaskResult()` 在同一队列内完成 `read -> applyCanvasTaskResult -> write`，并返回最终文档；不再使用 epoch、generation、append replay 或跨画布全局锁。

- 当前画布接收定向结果前，`canvas:before-task-write` 必须异步等待 `CreationPanel` 刷新当前快照，并持有同一 `CanvasGate` 到磁盘结果恢复完成；输入在门控期间不可操作。
- 第二个同 `owner:path` 任务等待前一个 gate。需要恢复的同路径重复打开也等待 gate，醒来后重查 load token、owner、path 和是否已有新 gate，绝不用旧请求打断后来的任务。
- 目标任务从提交起为 `canvasWriteStatus: 'pending'`；完成列表清理和单项删除不能丢掉它。重命名/删除在生命周期屏障前后都检查 pending 状态，避免物理文件操作与结果写入交错。
- 项目变为空 owner 或组件卸载时不再追加最终 `flushCanvasSave()`；`flushCanvasSave()` 也只允许保存仍是当前选中项目 owner 的画布。任务或生命周期屏障已先持久化过的快照不能再覆盖任务 RMW。
- 结果恢复失败时保持 `canvasReady = false` 与 `canvasRestoring = true`，因此旧场景仍不能写盘。失败处理只释放任务 gate，交给用户显式 `canvas:open` 从已落盘的任务结果重新恢复；不能把自动路径恢复为可交互或重新排入旧快照。

不要把 rename/delete 改成多文件事务队列：本轮约束是单 `owner:path` 队列。文件树先执行异步屏障、拒绝 pending target，当前画布在 busy 状态不能产生新快照，文件操作成功后才切换面板路径。

## 排障顺序

1. 先看对应 `.jccanvas`：检查 `assets` 是否有媒体 ID，`scene` 是否有同 ID 节点，以及节点的 `editable` / `draggable` / `locked`。
2. 对比能操作的节点和异常节点，不能只看视觉截图。
3. 检查恢复路径 `restoreCanvasScene()`，确认运行时节点被跳过、媒体 URL 被重新解析。
4. 检查新增媒体是否都走 `addMediaToCanvas()`，不要为生成结果另写一条节点创建链路。
5. 遇到卡顿先检查并发：媒体解码、视频首帧、画布恢复是否同时发生；优先排队或延后低优先级工作。
