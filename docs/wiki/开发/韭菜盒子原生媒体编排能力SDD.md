# 韭菜盒子原生媒体编排能力 SDD

> 日期：2026-07-20
> 状态：实施完成，自动验证通过；真实付费与跨平台人工验收待完成
> 产品名称：韭菜盒子原生媒体编排能力
> 依据：`AGENTS.md`、[[开发/对话框调用创作面板SDD]]、[[开发/创作工作台架构SDD]]、[[开发/文件系统/索引]]、[[开发/ZX-Grok视频双模式修复SDD]]

## 1. 一句话目标

用户在对话中直接引用项目素材、画布素材、本轮附件或当前对话最近生成的媒体，用自然语言继续生图或生视频；应用准确找到素材、展示可审阅的确认卡，用户确认后仍由现有创作面板和统一媒体任务引擎执行。

第一期的标志性闭环：

```text
用户生成一张图片
  -> 用户说“用刚才这张图做一个 10 秒视频”
  -> 应用展示真实参考图缩略图、视频模型、时长、提示词和价格
  -> 用户确认
  -> 现有 CreationPanel + mediaTaskStore 生成、落盘、回流画布
```

## 2. 产品归属

这是产品原生能力，不是 Skill。

| 层级     | 责任                                                                                   |
| -------- | -------------------------------------------------------------------------------------- |
| 产品内核 | 素材身份、项目权限、引用解析、模型能力校验、确认、提交、任务状态和结果落盘             |
| 模型     | 理解“这张图”“第二张”“刚才生成的图”等自然语言，在应用提供的候选素材和模型范围内形成计划 |
| Skill    | 可选地提供专业提示词、镜头语言和行业流程；不能拥有素材路径、任务提交或付费执行         |

核心媒体编排在没有安装、加载或选择任何媒体 Skill 时也必须可用。`jc-instant-create` 等 Skill 可以增强提示词质量，但不得再是 `MEDIA_PLAN_POLICY` 的硬前置条件。

## 3. 当前根因

现有链路已经具备计划、确认和执行，但缺少统一素材上下文：

1. `mediaPlan.ts` 允许模型直接输出 `referenceImages`、`referenceVideos` 字符串，模型理论上可以编造路径或 URL。
2. `ChatPanel.attachMediaPlan()` 只会把本轮聊天附件自动塞进计划，无法使用项目文件、画布选中素材和最近媒体任务结果。
3. `ProjectResource`、Canvas Asset、`MediaTask` 都有各自真实身份，但没有汇合成同一种对话素材引用。
4. `MediaPlanCard` 只显示“已附几个素材”，不显示实际用了哪一张，用户无法在付费前发现拿错图。
5. 当前 `MEDIA_PLAN_POLICY` 要求先加载 `jc-instant-create`，核心产品能力仍被 Skill 文案间接控制。

根因不是“缺一个路径输入框”，而是产品没有一份应用拥有、模型只能选择、提交前可重新验证的素材引用合同。

## 4. 已确认方案与替代方案

### 4.1 采用：应用拥有的素材引用

应用收集候选素材，生成本轮短期引用 ID；模型只能返回这些 ID。应用在确认和提交阶段根据 ID 重新解析真实项目资源或任务结果。

优点：跨平台、安全、可验证、可展示缩略图，也能支持文件改名和删除事件。

### 4.2 不采用：把真实路径交给模型

Desktop 绝对路径无法用于 Web，项目改名后会失效，也会扩大本机路径泄露和越权读取风险。项目相对路径只允许作为用户输入入口，必须立即转换成应用引用，不能直接进入模型计划和任务持久化。

### 4.3 不采用：仅用“最近一张图”硬猜

实现最短，但多图、跨项目和历史会话中很容易拿错图。最近生成素材只能成为模型可选择的候选，不能绕过确认卡静默提交。

## 5. 核心数据合同

### 5.1 模型可写计划

模型只负责创作意图，不持有真实素材地址：

```ts
interface MediaPlanDraft {
  kind: 'image' | 'video'
  title: string
  prompt: string
  modelId: string
  ratio?: string
  resolution?: string
  duration?: string | number
  referenceIds?: string[]
}
```

`referenceIds` 必须来自应用为本轮提供的候选目录。模型输出真实 URL、data URL、绝对路径或未知 ID，一律不能进入可提交计划。

### 5.2 应用素材引用

```ts
type MediaReferenceKind = 'image' | 'video'
type MediaReferenceSource = 'attachment' | 'project' | 'canvas' | 'task'

interface MediaReference {
  id: string
  kind: MediaReferenceKind
  source: MediaReferenceSource
  label: string
  value: string
  explicit: boolean
  invalidReason?: string
  locator:
    | { type: 'project'; resource: ProjectResource }
    | { type: 'task'; taskId: string }
    | { type: 'attachment'; messageId: string; index: number }
}
```

约束：

- `id` 只在一次对话运行的素材快照中有效，不能由模型自造。
- Canvas Asset 必须先归一为其项目资源；画布节点 ID 不是长期媒体身份。
- 已生成媒体优先使用 `taskId`，提交时读取任务当前的 `projectPath`、`assetUri` 或已验证 `resultUrl`。
- locator 是长期身份；`value` 只是确认卡/提交快照。项目与任务素材在提交前按 locator 重读，项目视频不持久化大体积 data URL，临时 blob URL 不进入计划。
- 不发明新的全局素材数据库；`ProjectResource`、`MediaTask` 继续是事实来源。

### 5.3 本轮素材快照

发送消息时冻结一份 `MediaContextSnapshot`，包含当前项目、当前会话和候选引用。模型回复、确认卡和最终提交都使用同一快照，之后用户改变画布选择或切换项目不能偷偷改变这张确认卡实际引用的素材。

## 6. 素材入口与确定性规则

### 6.1 明确引用入口

完整目标支持四个来源，第一期全部接通：

1. 对话输入框本轮上传或粘贴的附件。
2. 项目文件树对图片/视频执行“引用到对话”，或拖到对话输入区。
3. 画布选中媒体后执行“引用到对话”。
4. 当前项目、当前会话最近成功生成的图片/视频任务结果。

文件树和画布入口最终都复用 `FileUploader.addProjectResources()` 或同一素材引用入口，不各写一套读取、预览和上传逻辑。

### 6.2 路径粘贴

路径是高级入口，不是主交互：

- 当前项目内的相对路径粘贴后，立即通过 `ProjectFileService` 解析成带缩略图的引用标签。
- Desktop 当前项目内的绝对路径可归一为项目资源；项目外路径不得直接引用，必须先走现有安全导入流程。
- Web 不解释本机绝对路径。
- 找不到、是目录、不是受支持媒体或越出当前项目时，明确报错，不把原路径交给模型猜。

### 6.3 选择优先级

应用只自动绑定用户本轮明确加入输入框的引用，优先级为：

1. 明确引用到输入框的项目/画布素材；
2. 本轮上传或粘贴附件；
3. 模型从当前会话最近生成候选中选择的引用 ID。

明确引用存在时，不用最近任务覆盖。没有明确引用时，模型可以根据“刚才这张图”“第二张”等语义，从应用给出的候选目录选择；应用不使用关键词正则替模型猜指代。

多个素材超过模型注册能力时，不截断、不偷偷换模型：确认卡标出冲突并要求用户移除、更换素材或重新规划。

## 7. 模型目录与 Skill 解耦

每轮由应用从 `creationModelRegistry` 和实时可用性生成精简模型目录，只包含：

- 合法模型 ID 和用户可读名称；
- image/video 类型；
- 是否支持参考图/参考视频及数量；
- 比例、分辨率、时长能力；
- 当前价格和可用状态。

模型只能从目录中选择，`validateMediaPlan()` 和 `buildCreationRunPlan()` 继续做最终校验。目录不复制成第二份硬编码模型表。

新的原生 `MEDIA_PLAN_POLICY` 只说明计划合同、候选素材和确认规则，不再要求加载特定 Skill。用户明确选择 Skill 时，Skill 提供创作方法；模型目录、素材引用和执行边界仍由产品覆盖，Skill 不能绕开。

## 8. 完整数据流

```text
项目文件 / 画布选择 / 本轮附件 / 最近媒体任务
  -> buildMediaContextSnapshot()
  -> 输入框显示引用标签，模型只看到短期 ID + 类型 + 标签
  -> 模型输出 MediaPlanDraft(referenceIds)
  -> materializeMediaPlan(draft, snapshot)
  -> 校验 ID、项目归属、素材存在性、模型能力和参数
  -> MediaPlanCard 显示真实缩略图、模型、模式、时长、价格、提示词
  -> 用户确认
  -> 提交前再次解析 locator，防止删除/改名/跨项目
  -> buildMediaPlanSubmission()
  -> 现有 CreationPanel + mediaTaskStore
  -> 现有上传、提交、轮询、jc-media 落盘和画布回流
```

禁止新增第二套媒体 API、上传器、轮询器、任务队列、下载器或媒体落盘路径。

## 9. 确认卡

付费任务确认卡必须让普通用户一眼看懂：

- 实际参考素材缩略图、名称和来源；
- 生成类型、模型用户名称和有效模式（文生视频/图生视频等）；
- 提示词、比例、分辨率、时长；
- 注册表已有价格信息；
- “开始生成”“移除素材”动作；第一版不在旧卡内原地更换，用户移除后重新引用素材并形成新计划。
- 素材失效、模型不兼容或模型不可用时的明确原因。

确认卡不能只显示数量，不能把原始模型 ID 当主要名称。用户确认前不提交；提交中锁定按钮防重复；失败后保留原计划和引用，可修正或重试。

## 10. 生命周期与失败策略

| 场景                | 必须行为                                                                   |
| ------------------- | -------------------------------------------------------------------------- |
| 项目资源改名        | 活跃引用消费 `ProjectResourceChange.renamed` 更新 locator 和标签           |
| 项目资源删除        | 引用标记失效，确认按钮禁用，不复活旧路径                                   |
| 项目切换            | 旧项目快照不能在新项目提交；卡片提示回到原项目或重新选择                   |
| 最近任务仍在生成    | 不进入候选；成功后下一轮自动进入最近任务候选                               |
| 最近任务失败/取消   | 不作为可用候选                                                             |
| Web blob URL 过期   | 从 `ProjectResource`/`MediaTask` 重新生成预览和提交数据，不持久化 blob URL |
| 模型无参考图能力    | 不静默降级为文生视频，要求换兼容模型或移除素材                             |
| 素材数量超限        | 明确指出上限，用户决定保留哪些                                             |
| 模型返回未知引用 ID | 计划无效，不出现可付费确认按钮                                             |
| 重复点击            | 同一计划只允许一个在途提交                                                 |

## 11. 跨平台边界

- `ProjectResource.runtime + owner + path` 是项目素材的现有身份合同；UI 不自行拼 Desktop 路径或 Web OPFS 地址。
- 二进制读取继续经过 `ProjectFileService.readBinary()`；外部文件继续经过现有导入动作。
- Desktop、Web 对同一个 `MediaReference` 使用同一业务逻辑，仅在既有项目资源适配器内部处理文件系统差异。
- Windows 路径分隔符、Intel Mac 和 Web 无绝对路径场景都必须进入测试与人工矩阵。
- 不把本机绝对路径、data URL 或鉴权 URL 注入模型提示词。

## 13. 分期实施

### Phase 0：合同与安全边界

1. 先写失败测试，定义 `MediaReference`、快照和模型可写 `referenceIds`。
2. 禁止新计划携带真实 `referenceImages/referenceVideos`；为历史已保存计划保留只读兼容转换。
3. 从注册表生成模型目录，移除原生链路对 `jc-instant-create` 的硬依赖。

### Phase 1：明确引用闭环

1. 本轮附件、项目文件树和画布选中素材进入同一输入框引用集合。
2. 项目相对路径可转换成引用标签；项目外路径走既有导入。
3. 当前项目、当前会话最近一张成功图片进入候选，完成“用刚才这张图”的单图指代。
4. 计划卡显示缩略图、模型、模式、参数和价格。
5. 完成“单张图片 -> 图生视频”真实付费人工验收。

### Phase 2：最近生成与自然指代

1. 将候选历史从最近一张扩展为当前会话最近多张图片和视频。
2. 模型用引用 ID 理解“第一张”“第二张”等多素材指代，应用验证选择结果。
3. 结果生成后可以连续进入下一轮，形成图片 -> 视频 -> 继续编辑链路。

### Phase 3：生命周期和双端收尾

1. 接入改名、删除、项目切换和任务状态变化。
2. 完成 Web、Windows、Intel Mac、Apple Silicon 人工矩阵。
3. 删除被新合同替代的字符串引用旁路，更新 Wiki 和审计记录。

每个 Phase 必须独立可用、可回滚、测试先行；不得一次性重写 ChatPanel、CreationPanel 或媒体任务引擎。

## 15. 人工验收

Desktop 和 Web 分别执行：

1. 上传一张图片，要求生成视频，确认卡显示正确缩略图，确认后生成图生视频。
2. 在项目文件树选择图片“引用到对话”，无需重新上传即可生成。
3. 在画布选择图片“引用到对话”，输入框出现同一项目素材标签。
4. 先生成图片，再说“用刚才这张图做 10 秒视频”，确认卡命中刚生成的图片。
5. 连续生成两张图，分别说“第一张”“第二张”，确认缩略图与指代一致。
6. 在确认前改名、删除素材或切换项目，确认不会使用旧路径或错误项目。
7. 选择不支持参考图的模型、超出图片数量或使用失效任务，确认不会静默降级或提交。
8. 快速双击确认，确认只产生一个任务；失败后可以保留计划重试。
9. 重启应用/刷新 Web 后打开历史会话，项目/任务引用可重新解析，临时预览地址没有被持久化。
10. 不安装、不选择 `jc-instant-create`，仍能完成上述核心闭环；选择 Skill 后只改变创作质量，不改变安全和执行链路。

## 16. 完成标准

满足以下条件才能称为“韭菜盒子原生媒体编排能力”完成：

1. 四类素材来源进入同一合同，核心能力不依赖 Skill。
2. 模型永远不拥有真实路径、上传、扣费和任务执行权限。
3. 用户确认卡能明确看到实际素材和费用信息，不会盲确认。
4. 图片生成后一句话继续生成视频的闭环在 Desktop/Web 都通过。
5. 项目改名、删除、切换、重启和任务失败不会引用错素材。
6. 媒体执行仍只有现有 CreationPanel 和 mediaTaskStore 一套。
7. 自动测试、正式构建、产物审计和跨平台人工矩阵都有真实证据。

其中 1、2、3、5、6 的代码合同与自动验证已完成；4、7 中的真实付费、刷新恢复和三平台人工矩阵仍是发布验收项，不得写成已通过。

## 17. 明确不做

- 不在第一期扩展音频编排、口播、字幕、FFmpeg 或外部 GitHub 工具。
- 不让模型扫描整个项目猜素材，只提供受控候选。
- 不建立第二套媒体库、素材数据库或媒体任务 Store。
- 不自动提交付费任务，不因模型建议静默换模型或增加费用。
- 不为了“通用”先抽象任意工作流 DAG；先把图片 -> 视频闭环做到可靠。
- 不把 UI 选择状态、blob URL、绝对路径当长期素材身份。

## 18. 回滚边界

每一期都保留现有本轮附件计划作为回退路径。回滚新素材引用时，不影响模型注册表、CreationPanel、mediaTaskStore、已有媒体任务和 `jc-media` 文件；历史新引用计划降级为不可重新提交的只读卡片，不能猜测旧路径继续扣费。

## 19. 实际实施结果（2026-07-20）

### 19.1 最终架构

```text
本轮附件 / 项目文件树 / 画布选择 / 当前会话最近成功任务
  -> mediaReference.ts 生成受控 referenceId + locator
  -> mediaPlan.ts 从创作模型注册表和实时可用状态生成每轮政策
  -> 模型只返回 referenceIds
  -> ChatPanel 物化并展示确认卡
  -> 提交前按 ProjectFileService / mediaTaskStore 重读
  -> CreationPanel -> mediaTaskStore.submitTask() 原执行链
```

- Desktop 只在创模式直连链路注入动态媒体政策；文/武 OpenCode 链路不变。
- Web 直连聊天使用同一动态政策和引用合同。
- `creationModelRegistry` 是模型能力/价格事实源，运行时可用性状态负责剔除后端临时下线模型。
- `ProjectResource` 与 `MediaTask` 继续是素材事实源，没有新增媒体 Store、上传器、轮询器或任务引擎。

### 19.2 已完成能力

- 本轮图片/视频附件、项目文件树菜单和拖拽、画布选中素材、最近 5 个同项目同会话成功媒体任务进入统一候选。
- 支持项目相对路径、`@{相对路径}`、macOS/Windows 项目内绝对路径；Desktop 项目外绝对路径先导入 `jc-imports`。
- 明确引用永远优先；未知 ID、跨项目、删除素材、不兼容模型、超限参数和实时下线模型不能提交。
- 改名更新 locator/标签，删除标记失效；项目切换即时禁用确认并提示原因；移除全部引用后清理旧项目归属。
- 确认卡显示真实图片缩略图、素材名、用户模型名、有效模式、时长和注册价格，可移除素材。
- 确认状态在第一次异步读取前锁定，快速双击不能产生两个付费提交。
- 任务引用优先读取已落盘 `projectPath/assetUri`，失败才回退到任务的已验证结果 URL。
- 历史只有 `referenceImages/referenceVideos` 的旧计划保留原值，不会被新 locator 刷新逻辑清空。
- `jc-instant-create` 不再是原生媒体计划的硬前置；Skill 只增强创作方法。
