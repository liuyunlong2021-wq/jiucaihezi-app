# 创作工作台架构 SDD

> 日期：2026-07-15
> 状态：创模式底座已合入；Desktop 电商商品图工作台已作为首个落地试点，详见 [[开发/电商工作台SDD]]。其余工作台仍为方案。
> 前置条件：`docs/wiki/开发/创作模式双端统一SDD.md` 已完成、验证并合入 `main`
> 替代关系：本文件以创作模式为执行内核，替代 [[开发/创作面板模式化架构SDD]] 中依赖 OpenCode 的执行设计；旧文件保留为 UI 与产品构想档案。

## 1. 一句话目标

韭菜盒子的创作产品只有一颗心脏：**创作模式**。用户在项目中选择一个创作工作台，由 Skill、项目文件、媒体任务和画布共同完成从想法到作品的流程。

创作模式的能力顺序只有一条：**模型原生能力优先，产品能力补位，Skill 和工具按需增强。** 工作台组织用户任务，但不能把固定步骤、Skill 或工具变成直连模型处理原始输入的前置条件。

```text
项目 + 创作工作台 + Skill
  -> 创作模式直连模型与项目工具
  -> Wiki / 剧本 / 媒体资产写入当前项目
  -> 创作面板与画布呈现、确认、继续创作
```

## 2. 产品边界

### 2.1 名称与层级

| 层级 | 名称 | 作用 |
|---|---|---|
| 运行模式 | `文`、`武`、`创` | 决定谁执行对话：OpenCode plan、OpenCode build、创作模式直连运行时 |
| 创作工作台 | 漫剧制作、漫剧剧本、电商、自媒体、PPT、短故事、小说 | 决定用户此刻要完成什么创作任务 |
| Skill | 项目内按需加载的业务说明与流程 | 决定模型要读什么、写什么、何时调用项目工具 |

创作工作台不是新的 Agent，不是新的后端，也不是 OpenCode 的皮肤。它是创作模式上的可见流程入口。

### 2.2 继承与替换

| 保留 | 替换 |
|---|---|
| 项目文件树是内容真源 | OpenCode Agent / `ses_*` 会话不参与创作工作台 |
| Chat 做文字讨论、选项和确认 | 创作会话使用 `scopeKey: "creative"` |
| 创作面板与画布展示图片、视频、音频 | 操作台不再把 prompt 交给 OpenCode |
| Skill 驱动业务流程 | 创作模式的 `skill/read/glob/grep/write/edit` 工具循环执行项目工作 |
| 媒体结果进入当前项目 `jc-media` | 媒体任务继续走既有 `mediaTaskStore` |

文、武模式保持现有行为。创作工作台只在 `创` 模式中出现，不启动 OpenCode。

## 3. 工作区

桌面端和 Web 端在创作模式使用同一信息结构：

```text
Rail | 项目文件树 | 创作工作台与文字对话 | 创作面板 / 画布 / 编辑器
```

1. **Rail**：显示创作工作台入口。入口以用户任务命名，不展示 OpenCode、Agent 或内部工具名。
2. **项目文件树**：显示当前项目的 Wiki、剧本、素材和 `jc-media`；它是用户可检查的内容真源。
3. **创作工作台与 Chat**：工作台显示当前任务的步骤、结果摘要和明确操作；Chat 负责自由讨论、A/B/C 选择、修改和确认。
4. **创作面板、画布、编辑器**：图片、视频、音频统一在此预览和编辑；Chat 不嵌媒体播放器或大图预览。

小屏幕可折叠文件树或创作面板，但新用户桌面端默认同时看到项目、工作台和创作面板。

## 4. 统一执行规则

### 4.1 一次工作台操作

```text
用户点击工作台动作或自由输入
  -> 当前项目的创作会话发送明确意图，并选中对应 Skill
  -> 模型按需调用 skill / read / glob / grep / write / edit
  -> 文本结果写入当前项目或返回 Chat
  -> 媒体请求交给既有媒体任务
  -> 结果进入当前项目 jc-media，并由画布引用
```

工作台按钮不直接调用模型 API，也不复制 Skill 内容。按钮只选择用户意图、当前 Skill 和当前项目，然后复用创作模式既有发送链路。

### 4.2 会话与状态

1. 每个工作台使用当前项目的创作会话；切换工作台不丢失同一项目的创作历史。
2. 创作会话与 OpenCode `ses_*` 会话完全隔离。
3. 工作台步骤状态只由以下事实推进：项目文件已写入、媒体任务已完成、或用户明确确认。
4. 模型在文字中声称“完成 80%”不是步骤状态依据。
5. 用户可重新进入已完成步骤并发起重做；重做产生新项目资产，不静默覆盖用户确认过的结果。

## 5. 首批创作工作台

### 5.1 漫剧制作：串行工作流

| 步骤 | 用户看到的结果 | 项目真源 |
|---|---|---|
| 1. 装载项目与剧本 | 当前剧本、已识别人物/场景摘要 | 当前项目 Wiki / 剧本文件 |
| 2. 风格定调 | 风格选项、画幅、色彩与用户确认 | Skill 写入的风格文档 |
| 3. 角色、场景、道具 | 资产说明与生成媒体 | Wiki 资产说明 + `jc-media` |
| 4. 分镜 | 可执行分镜与视频提示词 | 项目分镜文件 |
| 5. 生成与组装 | 已生成镜头、可预览媒体、可导出成片 | `jc-media` 与导出文件 |

每一步由对应 Skill 决定具体 Wiki 结构和渐进式读取方式。工作台只显示步骤、当前成果、重做与进入下一步的动作。

### 5.2 电商：商品图试点已落地，其余工具待验证

Desktop 已完成“商品图 + 参考图反推”闭环：工作台/AI 协作记录共享同一创模式会话；模型只给受限媒体计划，CreationPanel 执行、轮询、落盘并回流结果。反推试点使用显式 `workbench.json`，不会从 Skill 正文猜界面。详见 [[开发/电商工作台SDD]]。

修图、产品视频和视频反推仍需各自真实闭环，不能因为它们在产品构想里存在就提前露出入口。

### 5.3 自媒体：串行出片工作流

写文案、配图/视频、配音、BGM、合成五步串行。用户可选择 AI 写稿、自备稿或自备素材；已提供的内容跳过相应生成步骤，后续步骤仍从项目文件读取。

### 5.4 其他入口

漫剧剧本、PPT、短故事、小说保持 Skill 入口可用。只有在其真实用户流程被验证后，才各自增加专用工作台；不预先复制漫剧的五步 UI。

## 6. Desktop 原生制作能力

创作模式核心保持双端共有。模型/渠道能原生处理时先直连；以下能力只在原生能力不支持、超出限制或用户要求进一步加工时，作为 Desktop 的受控补位与增强，不取代模型原生输入，也不向模型开放任意 Shell：

| 能力 | 用户动作 | 结果 |
|---|---|---|
| 转写 | 视频/音频转字幕 | SRT 与项目文字文件 |
| 拆镜/截帧 | 分析视频镜头 | 镜头片段、关键帧与分析资料 |
| 渲染合成 | 字幕、高光、素材合成 | 可预览和导出的成片 |

这些能力可由 Whisper、FFmpeg、Python 或 HyperFrames 实现，但工作台只调用固定的产品能力。Web 端可以完成资料分析、脚本和媒体生成；遇到本地渲染时明确提示在 Desktop 完成，不能伪造“已出片”。

## 7. 实施顺序

### Task 0：前置验证

- 验证 `docs/wiki/开发/创作模式双端统一SDD.md` 的完成标准已在 Web 与 Desktop 达成。
- 确认 `创` 模式可在当前项目中运行 Skill、读写文件、保存创作会话、写入项目媒体和画布。

### Task 1：创作工作区

- 在 `创` 模式建立 Rail、项目文件树、工作台/Chat、创作面板/画布/编辑器的工作区关系。
- 工作台入口与 `文/武/创` 运行模式分层命名，不混称“模式”。
- 默认入口为漫剧制作；其他首批入口可切换且保留现有 Skill 使用入口。

### Task 2：漫剧制作工作台

- 实现五步状态、结果摘要、用户确认、重做与回看。
- 工作台动作复用创作模式发送链路和 Skill，不新建第二套对话或工具执行器。
- 状态依据项目事实和媒体任务，不能解析模型自然语言猜测进度。

### Task 3：项目媒体与画布联动

- 每个媒体任务在提交时绑定当前项目；成功结果进入 `jc-media` 并可从画布打开。
- Chat 只显示文字摘要和跳转动作，媒体预览只在创作面板/画布完成。

### Task 4：电商与自媒体工作台

- Desktop 电商已先实现商品图与参考图反推试点；后续只在修图、产品视频或视频反推各自闭环验证后增加入口。
- 自媒体实现五步流程和三种输入方式。
- 复用 Task 2 的步骤状态和项目成果呈现，不复制对话、文件或媒体执行逻辑。

### Task 5：Desktop 原生制作增强

- 先实现固定的转写、拆镜/截帧、渲染合成能力，再接入相应工作台动作。
- 每项能力独立展示安装/可用状态、运行状态和输出路径。

### Task 6：回归与文档

- Web/Desktop 同一 Skill、同一项目相对路径与媒体归属行为一致。
- 验证切换创作工作台、切项目、切文/武/创不会串会话或串项目资产。
- 验证媒体任务取消、失败和重做不会错误推进工作台步骤。
- 更新 Wiki、历史和产品架构文档。

## 8. 完成标准

1. 用户不理解 OpenCode 也能从项目进入任一创作工作台，完成 Skill 驱动的创作。
2. 漫剧制作可从剧本/Wiki 推进到项目内的分镜和媒体资产，并在画布中查看。
3. 电商和自媒体的媒体结果都归属当前项目，不散落在聊天记录或浏览器下载目录。
4. Web 与 Desktop 共用创作模式、Skill、项目工具语义和项目相对路径；Desktop 只额外提供受控原生制作能力。
5. 文、武模式的 OpenCode 行为保持不变；创作工作台不依赖 OpenCode runtime、`ses_*` 或 Shell。
