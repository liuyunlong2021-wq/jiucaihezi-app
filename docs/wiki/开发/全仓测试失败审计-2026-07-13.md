## 关键证据

1. `sdkContract.test.ts` 使用 `Authorization: Bearer <password>`；Rust 启动代码和 OpenCode 官方服务端均使用 Basic Auth。这 5 项失败来自测试夹具，不是 APP 鉴权实现。
2. `session.test.ts` 仍断言 `client.session.prompt()`；当前官方信息流已使用 `promptAsync + global.event`，对应专项测试全部通过。
3. `chatMessagePresentation.test.ts` 同时保留新旧架构断言。例如它仍要求保存 `openCodeSessionId` 双 ID，而当前 SDD 明确要求 Desktop 从创建起只使用真实 `ses_*`。
4. 媒体测试仍把 `grok-video-3`、`newapi/t8/seedance-2-0-pro` 当作可执行模型；当前注册表已明确标记渠道下线/`broken`。
5. `skillsManageStore.ts` 中 Discover/Obsidian 函数仍是死亡代码，但不再从 Store 返回；归档审计已经决定删除这批旧 Vault 能力，相应测试未同步。
6. 浏览器日志中的 `navigator.platform=MacIntel` 不是硬件证据。本机事实为 Apple M1 Max（`uname -m=arm64`），因此 Intel baseline 尚未真机验收。

## 安全处理顺序

1. **测试夹具修复**：SDK 合同改用与 APP 相同的 Basic Auth，并输出响应 error，避免五项级联失败。
2. **旧架构测试迁移**：删除 legacy prompt、双 ID、每轮 SSE 等作废断言；保留并扩充 `desktopOpenCodeSyncCutover.test.ts` 的行为合同。
3. **媒体合同对账**：按当前模型注册表和后端路由更新下线模型测试；RH 字段只按真实接口文档改。
4. **Skill/工具死代码清理**：确认 UI 无调用后，成对删除死亡实现和测试；不恢复已经决定移除的功能只为过测试。
5. **独立小问题**：把 `ChatPanel.vue` 原生 `confirm()` 换回项目统一 `confirmAction`。
6. 重新运行 `test:focused`；只有剩余失败能够稳定复现为产品行为时，才修改产品代码。
