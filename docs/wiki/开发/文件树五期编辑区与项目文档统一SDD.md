## 1. 第五期铁律

**项目文件和项目素材是韭菜盒子唯一的数据事实源；项目资源服务是唯一文件总管，文件树是它的可视化入口。**

编辑区、创作面板和画布都不拥有第二份文件规则或存储实现。只要一个功能涉及项目里的文件或素材，必须先由项目资源服务的资源规则判定；跨面板编排复用共享动作，单一文件树操作可以直接调用同一 `ProjectFileService`。各面板只负责发起意图、展示结果和编辑当前资源。

```text
项目文件 / 项目素材
  -> ProjectResource + ProjectFileService（必要时经共享动作编排）
  -> 资源事件
  -> 编辑区、创作面板、画布同步展示同一资源
```

这不是“编辑区模仿文件树”的视觉要求，而是数据、保存、导出、打开、新建、改名、删除都只有一个事实来源。以后禁止在编辑区或画布中新增绕开文件树规则的文件 API、导出器、路径拼接或独立存储逻辑。

## 2. 本期目标

第五期是这轮文件系统改造的最后一期**编辑区 UI 接入**阶段。四点五期已经完成画布文件、媒体和 Desktop 外部拖放的存储收口；本期只让编辑区接入既有合同，并为画布补齐“新建项目文档/定位”这类用户入口。不得重写画布持久化、媒体导入、保存队列或 lifecycle gate。

本期最终交付五项直接可见的结果：

1. 编辑区只有“保存”，保存当前正在编辑的项目文件；不再有“全部保存”。
2. 编辑区“导出”只导出当前项目文件的原始文件，不再选择 Word、PDF、HTML、Markdown 等转换格式；它与文件树“导出所选资源”是同一个动作。
3. 编辑区右键是产品自己的中文编辑器快捷菜单，显示工具栏和更多菜单里适用于当前位置的常用编辑动作；不显示系统英文菜单。
4. 文件树能打开的项目文本，编辑区都能按文件树资源规则正确打开、编辑和保存；画布、媒体、二进制继续按文件树路由处理。
5. 编辑区、文件树和画布的新建项目文档共用目标目录、命名、冲突和打开规则；新建文档一定进入当前项目并自动打开，不存在独立的“编辑区文件”。改名、删除、批量和画布媒体继续复用一期、二期、四点五期已经完成的资源事件合同。

### 2.1 已完成起点与本期禁区

| 已完成能力 | 本期只能复用，不能重造 |
|---|---|
| 一期资源身份、安全文本读取、打开路由 | `ProjectResource`、`openProjectResource()`、`ProjectFileService`；二进制、NUL、截断内容不进可写编辑器。 |
| 二期批量操作 | 复制、移动、覆盖、删除后的 batch 资源事件和画布 lifecycle gate。 |

| 四期编辑会话 | Tab 的 resource、revision、dirty、冲突与删除状态；本期只收口用户入口，不重做状态机。 |
| 四点五期文件总管 | 画布 JSON 提交、媒体导入/读取、资源事件、Desktop 外部文件安全导入。 |

本期不处理 Finder 图片落到画布却偶发命中对话区，也不处理文件树定向刷新极端并发的旧快照回写；这两项已在四点五期记录为后续问题，不能借第五期编辑区 UI 改造扩大范围。

## 3. 根因

问题不是五个单独的 UI bug，而是早期编辑区保留了面向临时草稿和富文本文档的独立路径：

| 旧旁路 | 造成的问题 | 第五期处理 |
|---|---|---|
| 编辑区自有“全部保存” | 用户不知道保存范围，且与当前文件操作不一致 | 删除“全部”概念，只保留保存当前项目文件 |
| 编辑区自有 Word/PDF/HTML/Markdown 导出器 | 同一个项目文件有两套导出语义，格式转换会改变文件含义 | 删除编辑区格式转换导出，复用文件树导出当前资源 |
| 编辑器拦截右键后无产品菜单 | 用户只能看到失效右键或系统英文菜单 | 建立产品内中文编辑器右键菜单 |
| 所有文本先转 Markdown/Tiptap | 非 Markdown 文件被误解析、空白或改写 | 文件树资源分类决定编辑区显示模式 |
| 编辑区“新建文档”单独创建草稿 | 新文件不一定进入项目文件树 | 所有新建都由文件树资源动作创建 |

根因归结为一条：**编辑区仍保留自己的项目文件语义，而不是消费既有项目资源合同。** 第五期必须清理这些 UI 旁路，不对每个症状再各加一层补丁，也不把文件树的普通单资源操作额外包一层无意义的动作对象。

## 4. 第五期的统一合同

### 4.1 使用四点五期文件总管

四点五期已定义并验证项目资源服务是以下能力的唯一拥有者；文件树只是这些能力的可视化入口：

| 文件能力 | 唯一规则与执行位置 | 编辑区 / 画布职责 |
|---|---|---|
| 文件身份、类型、路径安全 | `ProjectResource` 与共享分类函数 | 只接收资源，不自行猜扩展名或拼路径 |
| 打开路由 | `openProjectResource()` | 根据路由显示编辑器、画布、媒体预览或拒绝说明 |
| 新建文件、文件夹、画布 | `ProjectFileService`；画布专用编排复用共享动作 | 发“新建项目文档/素材”的意图 |
| 保存 | `ProjectFileService.writeText()` 的 revision 条件写入 | 交出当前编辑内容和 revision，不直接写磁盘、OPFS 或 IndexedDB |
| 导出 | 共享 `exportResources()` 动作 | 将当前 `ProjectResource` 交给该动作 |
| 改名、删除、批量操作 | `ProjectFileService` + 资源事件 | 消费同一资源事件，更新 Tab、画布引用和当前视图 |
| Desktop/Web 差异 | 项目资源服务适配器内部处理 | 不在编辑区、画布写平台分支 |

第五期调用四点五期已经建立的资源服务和必要的共享动作；不能复制 Tauri、浏览器、OPFS 或冲突处理代码，也不再审计或重造项目资源服务。共享动作不是强制中间层：只被一个文件树入口使用的单资源操作可直接调用 `ProjectFileService`。

### 4.2 保存：只保存当前文件

编辑区工具栏的磁盘图标和菜单文案统一为“保存”。它只处理当前活动 Tab 对应的项目资源。

1. 当前 Tab 是干净项目文件：按钮禁用。
2. 当前 Tab 是脏且可写项目文件：调用文件树统一的条件写入，成功后更新 revision 和脏状态。
3. 当前 Tab 已冲突、删除、只读或保存错误：不覆盖、不复活旧路径；显示文件树规则中已有的明确下一步。
4. 当前内容不是项目文件：不伪装成“保存到项目”。只有文件树明确提供的“新建项目文档”或“另存为项目文件”动作可以创建项目资源。
5. 删除“保存全部”“全部保存”命令、快捷键、结果提示和相关 UI，不保留隐蔽入口。

保存语义必须与文件树双向一致：编辑区保存后文件树立即反映同一资源的新 revision/脏状态；文件树的改名、删除、外部变化后编辑区不得继续保存旧路径。

### 4.3 导出：按当前资源原格式导出

编辑区的“导出”不是文档格式转换器。它等价于：**在文件树选中当前 Tab 的资源，然后执行“导出所选资源”。** 第五期应抽出一个由编辑区和文件树共同调用的导出命令宿主；`projectFileActions.exportResources()` 现有的资源校验可复用，但不得继续由两个组件各自维护 Desktop/Web 目录选择和冲突处理。

| 当前资源 | 编辑区点导出后的结果 |
|---|---|
| `.md` | 导出该 `.md` 文件 |
| JSON、YAML、代码、TXT、锁文件等 | 导出同扩展名、同内容的原文件 |
| `.jccanvas` | 导出该画布文件 |
| 图片、音频、视频 | 导出该媒体原文件 |
| 当前没有项目资源、资源已删除或当前内容尚未保存 | 不显示可用导出动作，并提示先保存为项目文件或重新选择资源 |

要求：

1. 删除编辑区内 Word、PDF、HTML、Markdown 转换导出、导出预览、导出模板、导出选项、分片导出以及对应状态和依赖。
2. 编辑区只调用文件树现有的资源导出动作，继续继承 Desktop 的目录选择与重名处理、Web 的目录权限与冲突处理。
3. 不得把 Tiptap JSON、HTML、Markdown 序列化结果当成项目文件导出内容。导出内容以文件树中的当前资源为准。
4. 当前 Tab 有未保存修改时，先执行同一“保存”动作；保存成功后才允许导出，保存取消或失败则不导出旧版本。

### 4.4 右键：产品内中文编辑器快捷菜单

编辑区右键必须显示韭菜盒子自己的中文菜单，不使用 Desktop 或浏览器系统菜单。菜单动作不是再写一套功能，而是复用编辑区顶部工具栏和三点菜单已经存在的命令。

基础菜单按当前上下文显示，不适用的动作隐藏或禁用：

| 上下文 | 右键菜单动作 |
|---|---|
| 有文本选区 | 剪切、复制、粘贴、加粗、斜体、下划线、删除线、AI 润色/扩写/缩写/提炼 |
| 无选区的富文本文档 | 撤销、重做、粘贴、标题、列表、插入图片、插入表格、查找替换 |
| 原样文本文件 | 撤销、重做、剪切、复制、粘贴、全选、查找替换；不显示会改变文件语义的富文本格式动作 |
| 当前项目文件 | 保存、导出当前文件、在文件树中定位 |

要求：

1. 右键事件必须 `preventDefault()`，只显示产品中文菜单。
2. 工具栏、三点菜单和右键菜单使用同一命令标识和处理函数；任意一个入口修复后，其他入口同样生效。
3. 不显示系统英文菜单，不在编辑器右键中塞入 Desktop 特有动作，也不展示与当前资源无关的文件树操作。
4. 右键菜单支持鼠标关闭、Esc 关闭、点击外部关闭；焦点和选区不能因打开菜单丢失。

### 4.5 打开：文件树决定资源类型，编辑区只决定编辑表现

项目资源服务给出的 `ProjectResource.kind` 和安全读取结果是唯一打开依据。编辑区不得再以自己的扩展名黑名单、Tiptap 假设或磁盘路径分支重新判断。

| 文件树资源结果 | 最终去向 |
|---|---|
| Markdown 文档 | 编辑区富文本模式 |
| 其他安全文本 | 编辑区原样文本模式 |
| 画布 | 创作面板 |
| 图片、音频、视频 | 文件树已有的预览/加入画布路由 |
| 二进制、NUL、截断内容 | 文件树已有的不可编辑提示、预览、导出或站内打开路径；不创建空白编辑 Tab |

原样文本的定义是：读取、显示、修改、保存、导出都以同一字符串/字节内容为准，不经过 Markdown 解析、Tiptap 序列化、HTML 渲染或格式化。它不是第二套代码编辑器，也不需要自行发明保存和导出规则。

### 4.6 新建：只有新建项目文件，没有编辑区私有文件

“新建文档”统一更名为“新建项目文档”。编辑区工具栏、编辑区右键、文件树工具栏、文件树空白处右键、目录右键和画布中的新建文本入口都调用同一项目文件命令宿主；该宿主复用 `ProjectFileService.createText()`，不要求将既有文件树单资源操作迁进新的通用动作层。

目标目录遵循文件树的既有规则：

1. 选中目录时，在该目录创建。
2. 选中文件时，在其父目录创建。
3. 无选择时，在项目根目录创建。

完整流程：

```text
新建项目文档意图
  -> 项目资源服务依据资源上下文确定目标目录和文件名
  -> ProjectFileService.createText(owner, relativePath, '')
  -> 读取资源 revision
  -> 发布资源创建事件并刷新父目录
  -> openProjectResource(resource)
  -> 切换编辑区、打开新 Tab
```

默认名称、名称校验和重名处理均以项目资源服务的共享规则为准。失败、取消或重名拒绝时，不创建未绑定草稿、不打开空白 Tab。新建项目文档的命令宿主必须常驻，不能因为文件树 UI 当前未挂载而丢失请求。

### 4.7 画布与素材最终对齐

画布不是第二个素材库。画布节点只引用文件树的项目资源；素材的创建、导入、媒体预览、加入画布、导出、改名、删除和批量操作均继续通过文件树合同。

第五期完成后，编辑区和画布必须有同一结果：

1. 文件树改名、删除、移动或覆盖一个资源后，编辑区 Tab 和画布引用都消费同一资源事件。
2. 编辑区或画布不保留绕开 `ProjectFileService` 的真实文件写入。
3. 当前资源可从编辑区或画布定位回文件树；文件树选中资源也可按统一路由回到对应面板。
4. 画布媒体和编辑器文本都不拥有“导出副本”规则，均复用文件树导出。

## 5. 实施顺序

### Task 0：保存和导出归位

1. 删除“全部保存”并改成“保存当前项目文件”。
2. 删除编辑区所有格式转换导出功能和旧导出器依赖。
3. 将编辑区导出接到文件树“导出当前资源”共享动作；脏内容先保存。
4. Desktop/Web 分别验证导出结果、取消和同名冲突都与文件树一致。

### Task 1：编辑器中文右键与统一命令

1. 提取工具栏、更多菜单、右键共用的编辑命令表。
2. 实现按选区和文本模式过滤的中文右键菜单。
3. 删除原生 `contextmenu` 放行、系统菜单依赖及任何英文菜单路径。
4. 验证快捷菜单不会绕开保存、导出或文件树定位规则。

### Task 2：资源打开、新建入口和画布对齐

1. 由文件树资源路由统一决定富文本、原样文本、画布、媒体和不可编辑资源。
2. 删除编辑区私有新建草稿/磁盘文件路径；所有项目新建走文件树命令宿主。
3. 画布只接入同一新建项目文档和定位入口；画布保存、媒体导入、导出、改名和删除继续复用四点五期已有实现。
4. 对侧栏卸载、项目切换、资源改名、删除、批量移动等边界做 Desktop/Web 回归。

### Task 3：删除遗留、双端验收与文档收尾

1. 删除旧导出格式、第二套文件保存、第二套资源分类和不再使用的事件。
2. Desktop/Web 走同一份人工验收，不允许只在一个端“看起来可用”。
3. 更新本 SDD、[[开发/文件树五期]]、`hot.md`、`log.md` 和实际代码合同；只有证据完整后才标记第五期完成。

## 7. Desktop/Web 人工验收

在 Desktop 和 Web 各完整执行一次：

1. 修改当前 Markdown 项目文件，确认工具栏只有“保存”；保存后只更新当前文件，文件树状态同步。
2. 在编辑区点导出，确认得到当前文件的同名同扩展名原文件；再从文件树导出同一文件，确认两端行为、冲突提示和取消结果一致。
3. 确认编辑区不再出现 Word、PDF、HTML、Markdown、模板、预览、分片等格式转换导出入口。
4. 在富文本选中内容、富文本无选区、JSON/YAML/代码原样文本中分别右键，确认都是中文产品菜单，且仅显示当前可用动作。
5. 打开 Markdown、JSON、YAML、`pnpm-lock.yaml`、代码、TXT，编辑并保存后重新打开；确认内容不丢失、不改写。
6. 打开图片、音频、视频、画布、未知二进制、NUL 文件和截断文本；确认结果与文件树路由一致，不能打开空白可写 Tab。
7. 未选中资源、选中目录、选中文件后，分别从编辑区、文件树和画布新建项目文档；确认最终路径、文件树刷新、自动打开完全一致。
8. 对已打开文本和已上画布媒体执行文件树改名、删除、移动、覆盖和导出；确认三个区域没有旧路径复活、孤立副本或不同步状态。

完成条件：以上全部通过，且 `pnpm run test:focused`、`pnpm exec vite build`、`git diff --check` 通过。`vue-tsc` 如存在历史无关错误，必须精确列出文件和错误，不能把其失败包装为第五期完成。

## 9. 实施结果（2026-07-19）

1. 编辑区工具栏只保留当前活动项目文件的“保存”；脏状态、revision 条件写入、冲突和删除继续使用既有项目编辑会话。
2. 删除 `editorExport.ts`、格式转换导出、预览、模板导出和分片导出。编辑区先保存当前资源，再通过 `project:export-resources` 委托文件树导出命令宿主；文件树和编辑区复用同一份 Desktop/Web 目录选择、冲突和取消流程。
3. 富文本和原样文本均拦截系统右键并显示中文菜单。菜单按选区与模式提供已有的编辑命令；项目资源额外提供保存、导出和定位。
4. 项目文件只通过 `resource` 打开和 `ProjectFileService` 保存。遗留磁盘/SQLite 内容保留为不自动保存的临时内容，不能再写回项目或被关闭动作自动落库。
5. 编辑区和创作面板的新建项目文档均发出既有 `project:new-document` 意图，由文件树命令宿主按选中目录、文件父目录或项目根目录创建并打开。

自动验证：`pnpm run test:focused:build`、相关 focused tests、`pnpm exec vite build` 与 `git diff --check` 通过。Vite 仍输出仓库既有的动态导入和 chunk size 警告。
