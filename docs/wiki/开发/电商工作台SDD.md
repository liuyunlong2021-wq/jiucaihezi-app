# 电商工作台 SDD

> 日期：2026-07-17  
> 状态：Phase 1-3 已完成（Desktop）；Web 创模式运行时尚未具备同等工具循环，入口暂不暴露
> 范围：创模式的电商工作台；不修改文模式、武模式或 OpenCode 运行时

## 1. 一句话决断

电商不是另一套 Agent 或另一套媒体系统。它是**同一个创模式直连内核的一张工作台外壳**：把商品图、参考图和交付目标组织成清楚的任务，再交给模型、Skill 和现有媒体任务能力完成。

```text
电商工作台 / AI 协作记录（第三列的两种视图）
                       ↓
同一个创模式会话、.raw 事实账本、热记忆、Skill、工具循环、三按钮审批
                       ↓
媒体计划（提示词 + 已注册模型建议 + 合法参数 + 素材引用）
                       ↓
现有 CreationPanel / mediaTaskStore / 画布（第四列执行和展示结果）
```

切换“AI 协作记录”只更换第三列视图，不创建会话、不丢草稿、不重新连接模型。

## 2. 产品目标

1. 小白不需要理解提示词、模型、尺寸、语言或十几个电商平台选项，也能完成商品图任务。
2. 用户始终可以在底部用自然语言补充、纠正或提出新要求，模型能力不被按钮限制。
3. 商品实物、交付用途和投放位置等模型无法可靠猜出的事实由用户提供；专业提示词、模型/尺寸建议和执行步骤由 Skill 与模型完成。
4. 用户可在工作台内创建自己的“电商工作台 Skill”，创建后立刻作为自建工作台使用。
5. 电商 Skill 负责策划和提示词；CreationPanel 负责付费任务的提交、轮询、项目落盘和画布呈现。
6. 不重复实现媒体生成、任务轮询、画布预览、文件上传、会话、历史、工具循环或审批。

## 3. 体验结构

### 3.1 第三列：两个视图，同一会话

默认显示“电商工作台”。右上角 `AI 协作记录` 将第三列切为现有完整 Chat；Chat 顶部 `返回电商工作台` 可回到原工作台。两种视图读取同一个创模式会话。

| 视图 | 默认显示 | 不显示在主位的内容 |
|---|---|---|
| 电商工作台 | 当前任务、必要素材/选项、当前建议、最近任务摘要、底部自由输入 | 完整逐条对话和工具细节，收进 `AI 协作记录` |
| AI 协作记录 | 现有 Chat 全历史、工具过程、权限条、错误与继续对话 | 工作台表单主体，返回后保留 |

模型需要用户做 A/B/C 选择时，工作台应把“当前待决定事项”显示为紧凑选择区；完整上下文仍可在协作记录中查看。

### 3.2 电商首页

首屏不做密集配置器，也不展示空功能页。V1 只显示一个真实可用的 `商品图` 任务。

```text
电商创作台                                      [AI 协作记录]

商品图
上传：商品图（建议） + 参考图（可选）
交付目标：主图 / 场景图 / 详情页长图 / 一组商品图
发布位置：让 AI 推荐 / 淘宝京东 / 抖音小红书 / 海外站
补充：自由描述、修改要求、品牌语气或不可改变的事实

AI 当前建议：先出低成本草稿，确认构图后再出高质量版本
结果去向：右侧 CreationPanel 与画布
```

删除电商工作台中的“改图”入口。局部重绘、扩图、抠图和调色属于现有 CreationPanel/画布能力，不能重造。

`参考图分析`、`宣传文案`、`视频` 在 V1 验证后逐项增加；在对应能力未可用前不显示空页签。

### 3.3 为什么不复刻密集配置器

截图中的平台、语言、风格、模型、质量和分辨率都不是无用能力，但大多数只是模型/Skill 能根据任务推导出的执行参数。将它们全放在首屏会让用户替模型做专业工作。

只有以下信息是首屏必要输入：

- 商品图：真实产品外观、包装、品牌文字等必须保留的事实来源。
- 可选参考图：只用于分析构图、光线、色彩与版式；不得复制对方品牌、商品、商标或受保护内容。
- 交付目标：用户到底要主图、场景图、详情图还是一组图。
- 发布位置：影响画幅、文案语言和安全留白；用户不确定时可让 AI 推荐。
- 自由补充：任何按钮未覆盖的要求。

模型、尺寸、语言和成本/质量策略以默认推荐运行。只有用户点开“高级设置”或模型需要追问时才显示。

## 4. 内置商品图任务

### 4.1 职责拆分：计划和执行

电商工作台采取“计划/执行分离”：模型最擅长理解商品、参考图、营销目标和风格；CreationPanel 最擅长稳定调用已注册的媒体模型、显示状态、轮询、保存项目并进入画布。两者不能互相替代。

```text
用户事实
  -> 创模式 + 内置电商商品图 Skill：分析并产出媒体计划
  -> 工作台显示计划：提示词、推荐模型/比例、草稿数量、素材约束
  -> 用户点“开始生成”
  -> CreationPanel / mediaTaskStore：提交、轮询、项目落盘、画布预览
  -> media-task-settled：结果摘要和项目相对路径回写同一创模式会话与 .raw
```

“开始生成”是明确的付费动作。模型可以建议，也可以在用户要求时修改计划；未经用户点击，工作台不提交媒体任务。

### 4.2 媒体计划合同

计划不是一段只能复制粘贴的文字，而是可由工作台审阅、CreationPanel 执行的受限数据。V1 只允许图片计划，最小内容为：

| 字段 | 来源 | 规则 |
|---|---|---|
| `prompt` | 电商 Skill | 完整可见、可修改的图片提示词 |
| `modelId` | 模型建议 | 必须存在于现有 Creation 模型注册表；不存在则让用户选择或改为默认推荐 |
| `ratio` / `resolution` | 模型建议 | 必须是该模型已声明的合法值 |
| `draftCount` | 工作台策略 | V1 限制为有限草稿数量，先确认构图再生成高质量版本 |
| `productAsset` / `referenceAssets` | 用户上传 | 只使用当前项目素材；商品图标为必须保留，参考图标为只借鉴画面语言 |
| `taskTitle` | 工作台 | 用于任务列表、画布和后续对话定位 |

工作台在提交前校验这些值，再把它们写进现有 CreationPanel 状态和提交计划。不能让模型任意拼接接口参数、模型 ID 或外部 URL。

### 4.2.1 媒体计划桥

媒体计划桥是 App 内部的标准组件，不是 MCP、服务器或新网络协议。它做三件确定的事：

```text
工作台/Skill 产出 MediaPlan
  -> 校验模型注册表、合法参数与当前项目素材
  -> 调用已有 CreationPanel / mediaTaskStore
  -> 监听已有 media-task-settled，回写会话活动和项目相对路径
```

未来电商、漫剧和自媒体可以使用同一座桥，只要它们交付同一种图片/视频/音频媒体计划；不同工作台只改变计划来源、表单外观和业务 Skill。V1 仅实现图片计划，先由电商商品图验证，不能提前抽象成“万能工作台引擎”。

### 4.3 专业驱动

V1 新建一个内置电商商品图 Skill，使用 `skill-creator` 创建并评估。它不是重复媒体能力，而是负责电商任务的专业判断：

1. 读取商品图、可选参考图、交付目标、发布位置与用户补充。
2. 复用 `JC-GPT-Image2-Skill` 中的产品图/编辑提示词规则，明确哪些内容必须保留、哪些视觉语言可以借鉴；不执行该 Skill 的 CLI 生成路径。
3. 生成可检查、可编辑的媒体计划；信息不足时只追问缺失的关键事实。
4. 参考 `JC-瞬间创作` 的能力表和模型路由知识提出建议；不在电商工作台中调用其 `jc_media.py` 提交/轮询路径。
5. 用户确认后，将计划交给现有 CreationPanel 和 `mediaTaskStore` 执行。结果由现有任务引擎写入项目、CreationPanel 和画布。

`JC-瞬间创作` 不删除。它继续服务普通 Chat、命令行和其他非电商工作台场景；电商工作台只禁止重复的媒体执行职责。

### 4.4 结果回流

`mediaTaskStore` 已是 ChatPanel 和 CreationPanel 的统一任务引擎。电商工作台的任务必须以 CreationPanel 来源提交，因此由它自动轮询、保存项目和发出完成事件；模型不负责反复查询“图片出没出来”。

任务提交和结束时，App 把任务标题、状态、项目相对路径和真实错误作为同一创模式会话的活动记录写入 `.raw`。用户再次提出“用刚才那张图继续做”时，工作台自动将该项目内结果作为下一轮可引用素材，而不是要求用户去文件树手动寻找和重新上传。

### 4.5 真实性边界

仅凭一句产品描述生成的是视觉概念，不保证真实商品的包装、成分、文字和规格。要产出可销售的商品图，工作台必须优先引导上传真实商品图或可核验的素材。宣传文案中的规格、功效、合规承诺同样必须由用户给出或确认，模型不能擅自编造。

## 5. 自建电商工作台 Skill

### 5.1 用户流程

“自建”是电商工作台的固定页签，而不是顶栏持续增加任意 Skill。没有自建 Skill 时，只显示空状态与 `+ 新建电商工作台 Skill`。

```text
自建
  -> 新建：用户用自然语言描述“我想让它做什么”，可附样例素材
  -> AI 预览：Skill Creator 生成 Skill 草稿与操作台预览
  -> 用户确认/补充：可要求删减或增加业务字段
  -> 创建并开始：Skill 进入自建列表，并立即激活
```

新建页在第三列原地展开，不跳转 Chat。用户不需要看到 JSON、字段类型或模型参数。

预览是必须步骤。不能根据一句话直接发布，因为用户需要先确认：这个工作台会问什么、会交付什么、是否适合自己的业务。

### 5.2 技术定义：声明式 UI，不是 Skill 写页面

RunningHub AI 应用能动态展示参数，是因为其 `app-info` 返回了机器可读的节点元数据；前端把它映射成现有动态字段。电商工作台采用同一思想，但元数据来自本地 Skill 包。

一个兼容的电商工作台 Skill 包包含：

```text
<skill>/
  SKILL.md       # 给模型的专业方法与工具规则
  workbench.json # 可选、受限的电商输入声明
```

`workbench.json` 只描述用户输入，不包含 Vue、JavaScript、网络地址、Shell 命令或模型密钥。V1 只允许：

| 类型 | 用途 |
|---|---|
| `text` / `textarea` | 产品信息、卖点、补充要求 |
| `select` / `choice` | 交付目标、市场、有限业务选项 |
| `file` | 商品图、参考图、资料；限定允许的 MIME 类型 |

每个字段必须有稳定 ID、用户可见标签、类型、是否必填和有限选项。Skill Creator 根据用户需求生成它；运行时使用受校验的通用字段渲染器显示它。

没有 `workbench.json` 的 Skill 仍是普通 Skill：可在 Chat 中选择和使用，但不自动长出表单。只有 `surface: "ecommerce"`、字段通过校验且声明了合法结果类型的 Skill 才能加入“自建”列表。

### 5.3 使用自建 Skill 时不变的东西

自建 Skill 只改变本轮加载的专业说明和工作台字段，以下能力绝不分叉：

- 当前创模式会话和完整可回看的 AI 协作记录。
- 项目 `.raw/sessions/jcses_*.jsonl` 事实账本与 `wiki/hot.md` 热记忆。
- 现有 `skill`、读写文件、媒体工具、工具循环、取消和失败恢复。
- Desktop 的三按钮审批；Web/Desk 各自既有项目文件存储。
- CreationPanel、媒体任务、画布预览和 `jc-media` 结果归属。

## 6. 开发顺序

### Phase 0：设计与冲突控制

1. 本 SDD 评审并冻结边界。
2. 等创作面板布局分支确定第三列/第四列的最终组件归属。
3. 电商工作台使用独立工作树实施；`WorkspaceLayout.vue`、`ChatPanel.vue`、`CreationPanel.vue` 由单一集成负责人合并，避免双分支同时改同一布局文件。

### Phase 1：工作台外壳和同会话切换

- 电商工作台与 AI 协作记录切换，不新建 session、不掉消息、不丢未发送草稿。
- 保留原 Chat 的权限、停止、失败恢复和续发能力。
- 先以开发入口验证，不改变 Rail，也不移动现有设置入口。

### Phase 2：内置“商品图”闭环

- 创建并验证内置电商商品图 Skill。
- 完成商品图/参考图/交付目标/发布位置/自由输入到媒体计划的事实传递与合法值校验。
- 跑通“方案 -> 用户点击开始生成 -> CreationPanel 执行 -> 完成事件回流同会话 -> 画布查看”。
- 验收真实商品图保留约束、参考图不复制约束、取消、失败和继续对话。

### Phase 3：Rail 接入

- 商品图闭环稳定后，Rail 增加 `电商` 入口，进入电商工作台。
- 不在此阶段删除旧 Rail 项或隐藏旧能力；先验证新入口可用。

### Phase 4：自建电商工作台 Skill

- 实现 `workbench.json` 校验和受限字段渲染。
- 在“自建”页实现描述、AI 预览、确认、创建并开始。
- 复用现有 Skill Creator 的 Skill 生成与格式校验，不新造 Skill 格式。
- 验收普通 Skill 不被误识别为电商表单，非法声明不会渲染或执行。

### Phase 5：扩展和信息架构收尾

- 依次接入参考图分析、宣传文案和视频，每项先有真实闭环再露出页签。
- 新入口稳定后，再把低频的 Skill 仓库、工具仓库和审查记录移到设置。
- 编辑器、CreationPanel 和现有改图能力保持可达，不能因搬迁丧失功能。

## 7. 验收清单

1. 在工作台和 AI 协作记录之间来回切换，始终是同一创模式会话；历史、草稿、附件、运行状态和权限不丢失。
2. 商品图任务能上传真实产品图，可选参考图，选择交付目标并发送自由要求；模型产出可编辑且经校验的媒体计划，用户点击开始生成后结果回到当前项目和右侧媒体面板。
3. 用户拒绝、允许一次、始终允许、停止、失败、继续均遵循创模式既有语义。
4. Desktop 电商工作台使用创模式既有的项目会话、`.raw` 和 CreationPanel；Web 要等 Web 创模式具备同等直连工具循环后再接入，不能先暴露半成品入口或伪造跨设备同步。
5. 自建 Skill 创建前有工作台预览；只有有效 `workbench.json` 才进入自建列表；普通 Skill 的既有使用方式不变。
6. Rail/设置迁移后，现有编辑器、技能仓库、工具仓库、审查、文件树和创作面板仍可从明确入口访问。

## 8. 明确不做

- 不创建“造模式”运行时、第二套创作历史、第二套工具循环、第二套媒体 API 或第二个审批系统。
- 不将所有 Skill 强制变成表单；写作、研究、Wiki 等普通 Skill 继续以自然语言为主。
- 不允许 Skill 执行自定义前端代码、网络请求或脚本来生成界面。
- 不让电商 Skill 直接运行 `jc_media.py`、自行轮询或自行保存媒体；媒体执行只走 CreationPanel 的既有任务引擎。
- 不在商品图 V1 中一次性实现改图、视频、平台市场、模板商城、价格策略或全行业表单引擎。
- 不先搬空 Rail/设置再补新入口。

## 10. Phase 1-3 实施结果（2026-07-17）

已完成的 Desktop 商品图闭环：

1. Rail 新增“电商创作台”，进入创模式并打开既有 CreationPanel；没有删除或迁移其他 Rail 项。
2. 第三列使用 `v-show` 在工作台和 AI 协作记录之间切换，ChatPanel 不卸载；草稿和已审阅计划按同一个 `creative_*` 会话关联，不复制消息历史。
3. `JC-电商商品图` 是规划专用内置 Skill。模型只能给出 `jc-media-plan`，不能自行调用媒体 API、CLI、轮询或下载。
4. 媒体计划用现有 Creation 模型注册表和 `buildCreationRunPlan` 校验。用户点击“开始生成”后才进入原有 `mediaTaskStore.submitTask({ source: 'creation' })`，由它负责真实提交、轮询、项目落盘和画布入图。
5. 成功或失败的任务结果会写回当前创模式会话，并追加到项目 `.raw/sessions/jcses_*.jsonl`。

刻意不做：Web 电商入口、改图页签、视频/文案页签、自建 Skill 表单和第二套媒体执行器。它们分别属于 Web 创模式补齐、Phase 4 或后续真实闭环，不能以空 UI 或复制代码冒充完成。
