## 1. 进程生命周期

- [ ] **冷启动耗时**: `opencode_ensure_server` 首次启动是否 < 8s？
- [ ] **进程复用**: 同 project+config 不重启进程？
- [ ] **进程切换**: 切 project 时正确杀旧启新？
- [ ] **应用退出**: `RunEvent::Exit`、SIGINT/SIGTERM、Windows Ctrl+C 都等待 sidecar 退出？
- [ ] **僵尸进程**: 退出 APP 后没有 PPID=1 的孤儿 `opencode serve`？
- [ ] **端口冲突**: `reserve_local_port` 正确分配空闲端口？
- [ ] **SQLite 安全**: 不手工删除运行中数据库的 `-wal/-shm`？

## 2. 事件桥接 (eventBridge.ts)

- [ ] **唯一全局流**: 应用生命周期只建立一个 `client.global.event()`？
- [ ] **平台 fetch**: Tauri loopback SSE 走 `safeFetch`/Rust bridge，不触发 WebView CORS？
- [ ] **队列合并**: 高频 part/status 事件按官方规则 coalesce，非合并事件不丢？
- [ ] **重连上限**: 连续失败指数退避且最多 5 次，同一失败周期只报一次？
- [ ] **事件类型**: 所有 OpenCode event type 都能收到？
- [ ] **目录分发**: event 的 directory 进入对应目录状态，不串项目？

## 3. 唯一 Sync Store

- [ ] **唯一真源**: Desktop session/message/part/status 只以 `openCodeSyncStore` 为准？
- [ ] **Reducer**: message/part/status/permission/question 事件全部经 `eventReducer.ts`？
- [ ] **Bootstrap**: 打开目录/会话时补齐 sessions/messages/status，不覆盖更新鲜事件？
- [ ] **项目隔离**: 切项目后当前 session、缓存和工具结果不串目录？
- [ ] **Vue 投影**: 组件只读 Store，不再维护第二份 Desktop 消息状态？

## 4. 发送与中止

- [ ] **即时反馈**: 第一个异步等待前出现用户消息、动态图标和停止按钮？
- [ ] **乐观 ID**: optimistic message/part 与 `promptAsync` 使用同一官方 ID，确认后原位合并？
- [ ] **失败回滚**: session.create/promptAsync 失败后占位移除、输入恢复？
- [ ] **中止**: 停止后 session 回到 idle，下一条消息可继续发送？
- [ ] **无旧内核**: 不存在每消息 SSE、状态轮询、完成后全量覆盖或 120 秒 watchdog？

## 5. Provider 配置投影 (providerProjection.ts)

- [ ] **模型路由**: `toOpenCodeModelProjection` 正确映射所有 provider？
- [ ] **API Key**: Keychain 读取 → 注入 OpenCode config？
- [ ] **config 签名**: 同 config 复用进程，不同 config 重启？
- [ ] **Ollama 路径**: 本地模型免 key 但 attachment=true？
- [ ] **工具能力**: 官方标记支持工具的模型投影 `tool_call:true`，不按异常样本硬编码黑名单？
- [ ] **Ollama thinking**: Qwen3 默认 `reasoning_effort:none`，首 token/CPU 与同模型基线接近？

## 6. Session 管理

- [ ] **创建**: `createOpenCodeSession` 参数与 SDK 一致？
- [ ] **切换**: 切 session 时清理旧 SSE 订阅？
- [ ] **列表同步**: `listOpenCodeSessions` 按 project 目录过滤？
- [ ] **真实 ID**: Desktop 从创建起使用同一个 `ses_*`，不再维护本地↔OpenCode 双 ID？
- [ ] **权限/问题**: pending 请求能显示、回复/拒绝后正确归并并继续运行？

## 7. IPC 调用层

- [ ] **Tauri invoke**: 所有 `invoke('opencode_*')` 参数类型与 Rust 端一致？
- [ ] **超时处理**: IPC 调用有合理的超时和错误处理？
- [ ] **Request 保真**: SDK `Request` 的 method/header/body/AbortSignal 经 Rust bridge 不丢失？

## 8. 消息显示 (ChatPanel + MessageBubble)

- [ ] **Part 渲染**: `normalizeOpenCodeParts` 处理所有 part type？
