# 三项生产故障闭环 SDD

> 日期：2026-07-21
> 状态：Word 转换已完成生产闭环；画布与 RH 的 Web 发布和真实业务验收待执行
> 范围：Web Office 文档转换、Web 项目图片画布恢复、RunningHub 异步视频任务回收

## 1. 结论

三个现象不是一个前端 Bug，也不能用“重新试一次”或“按芯片分支”处理。共同问题是把“仓库代码/自动测试通过”误当作“生产服务、浏览器存储和异步上游任务已真实闭环”。

| # | 用户现象 | 已确认根因 | 不属于根因 |
| --- | --- | --- | --- |
| 1 | Web 上传 Word 显示“云端文档转换失败” | 前端已请求 `/documents/markdown`，但生产 Nginx 没有该精确路由；请求落入 NewAPI 首页并返回 HTML 200，前端 JSON 解析后报通用转换失败 | Word 文件格式、MarkItDown 能力、用户登录状态 |
| 2 | 项目图片点击后画布只显示框 | 已发布 Web bundle 仍把项目相对路径写入 Leafer `scene.url`；Leafer 将 OPFS 路径当网站 URL 请求，早于 Blob URL 水合而 404 | 图片损坏、画布尺寸、项目文件树本身 |
| 3 | Intel Mac 的 RH Seedance 图生视频报失败，但 RH 有产出 | NewAPI 包装任务 `task_*` 的状态回收/结果映射把已提交任务误判失败；客户端继而调用不存在的退款路由 | Intel CPU、RunningHub 生成能力、客户端到 API 的基本连通性 |

## 2. 已验证证据

### 2.1 Web 文档转换

- [Web文档转换服务SDD](Web文档转换服务SDD.md) 从一开始就标记为“代码已实现，待生产部署”；生产部署必须创建 `document-converter` 并安装 Nginx `/documents/markdown` 路由。
- 前端 [documentMarkdown.ts](../../../src/utils/documentMarkdown.ts) 在 Web 端向 `https://api.jiucaihezi.studio/documents/markdown` 发送 multipart 请求，成功合同是 JSON `{ status: 'success', content }`。
- 2026-07-21 的匿名生产探测实际收到 NewAPI HTML 首页而不是 JSON。HTTP 200 不能证明转换成功；这里的 200 恰是路由缺失被首页 fallback 吞掉的证据。
- 2026-07-21 已在 VPS 部署 `document-converter`：容器仅暴露 `127.0.0.1:8810->8000`，本机 `/health` 返回 `200 application/json` 和 `{"status":"ok","service":"document-converter"}`；`nginx -t` 通过，公网 `/documents/markdown` 的 `OPTIONS` 返回 `204` 及预期 CORS 头。部署时自动备份了原 Nginx 配置至 `/etc/nginx/backups/api.jiucaihezi.studio.conf.bak-20260721-123231`。
- 真实 `.docx` 首次上传返回 `422`，完整复现确认基础 `markitdown` 缺少 `docx` 可选依赖；独立提交 `b29da91b` 将依赖改为 `markitdown[docx]`，并阻止将 Python Traceback 返回给浏览器。相同镜像对仓库真实 DOCX 转换得到 `5206` 字节 Markdown 后已部署 VPS。
- 用户在正式 Web 创模式重新上传真实 `.docx` 已成功：转换文本显示在会话，且无 `/documents/markdown` 422。这是 Word 上传闭环的生产验收，不将其外推为 PDF、Excel、PPT 的验收结果。

### 2.2 Web 画布项目图片

- 当前仓库的 [canvasDocument.ts](../../../src/components/canvas/canvasDocument.ts) 已在规范化时删除项目图片节点的 `url`，只把身份路径保留在 `assets[id].resource.path`，恢复时才生成 Web Blob URL。

### 2.3 RunningHub 视频任务

用户的时间线包含两类独立事件，不能混为一次失败：

1. `09:49` 的两次 `403 insufficient_user_quota`：余额 `$4.187188` 小于预扣 `$5.600000`。请求在 NewAPI 计费门前被拒绝，绝不能退款或归因给 RH。
2. 后续 RH Seedance 任务：NewAPI 日志显示 `POST /v1/videos` 成功，模型 `rh-seedance2-mini-image` 按 `8.00s * $0.70/s` 记录预扣；rh-adapter 返回原始 RH 任务 `2079385455681757186`。

服务器日志和实际查询证明该任务已完成：

```text
客户端 -> GET /v1/videos/task_bccy1RfxvueILX0NnBmulR6meZMdXvLO
Nginx  -> 09:58:39 至 09:59:58 共 8 次 HTTP 200
NewAPI -> 将包装 ID 映射到 RH 原始任务
rh-adapter -> GET /v1/videos/2079385455681757186 均为 HTTP 200
直接查询 rh-adapter -> status=completed, progress=100, url=<mp4>
客户端 -> POST /v1/tasks/task_bccy1RfxvueILX0NnBmulR6meZMdXvLO/refund
NewAPI -> HTTP 404；按次计费任务跳过差额结算
```

因此“HTTP 请求失败: error sending request ... /v1/videos/task_*”是包装任务轮询返回的失败业务状态或状态映射错误，而不是 Intel Mac 到 API 的网络失败。Nginx 已证明客户端轮询请求抵达且得到 HTTP 200；RH 已证明原始任务产出存在。

## 3. 目标与非目标

### 3.1 目标

1. Web Word/Office 上传只在转换服务真实返回成功 JSON 后进入 `files[]`；生产缺路由必须返回可诊断错误，不能伪装成 HTML 200。
2. Web 打开或新增项目图片时，Leafer 只接收运行时 Blob URL；项目相对路径不得持久化到 `scene.url`。
3. 任何已拿到上游任务 ID 的 RH 任务在轮询异常时保持“可恢复”，不自动标记失败、不给用户显示退款承诺；已确认上游完成时必须回填结果。
4. 发布门禁同时验证代码、Web/服务产物和真实生产链路。

### 3.2 非目标

- 不为 Intel、Apple Silicon 或浏览器分别复制 RH 请求实现；没有芯片差异证据。
- 不把 Web 项目 OPFS 二进制迁移到服务器或 R2。
- 不在前端伪造退款；退款只能由存在、鉴权、可审计的服务端能力执行。
- 不修改 NewAPI 官方容器源码。包装任务状态的正确合同应先通过 NewAPI 配置/版本能力和 rh-adapter 适配验证；若其无法提供可靠回收，客户端改用已确认的 RH 原始任务合同。

## 4. 设计决策

### 决策 A：文档转换独立部署并使用严格响应合同

保留无状态 `document-converter`：Nginx 只把 `/documents/markdown` 转给其 `127.0.0.1:8810`，其余 NewAPI 路由不变。前端必须同时检查 HTTP 成功、`Content-Type` 为 JSON 和 `payload.status === 'success'`；收到 HTML 时显示“文档转换服务未部署或路由错误”，而不是笼统的转换失败。

这是最小边界：不改 NewAPI、不保存用户文件，也不把二进制 Office 内容塞入模型接口。

### 决策 B：画布文档只保存资源身份，不保存浏览器显示地址

`CanvasDocumentV3` 的真源是 `assets[id].resource.path` 和资源版本；`scene` 只保存视觉属性。保存、迁移和复制均要删除项目图片 `scene.url`。打开画布后由项目文件服务读取 OPFS 二进制并生成 Blob URL，关闭画布释放 URL。

旧画布在新代码打开后必须经过一次规范化保存，才能移除历史 `scene.url`。这是一项数据迁移，不应要求用户重新导入图片。

### 决策 C：RH 任务以“提交确认”和“完成确认”分离状态

当 NewAPI 或 rh-adapter 返回任意可查询任务 ID 时，任务状态必须先变为 `submitted` 并持久化：记录公共 ID、原始 RH ID（若有）、模型、项目归属和轮询路线。之后的轮询错误只能变为 `recoverable`，不得直接触发退款。

任务只有在满足以下任一条件时才可标记 `failed`：

1. RH 原始任务明确返回终态失败；或
2. NewAPI 的包装任务和可查询的 RH 原始任务都明确返回终态失败。

包装 ID `task_*` 不是 RH 的原始 ID。若 NewAPI 返回包装 ID，同时响应包含原始 RH ID，优先持久化并轮询原始 ID 经 `/rh/tasks/{id}`；若只得到包装 ID，先按 NewAPI 合同轮询，但遇到非终态错误保留任务并提供“恢复查询”。

客户端不得调用 `/v1/tasks/{id}/refund`，除非服务端先提供并返回可审计的退款合同；此次已证实该路由为 404。对已知上游成功的任务，结果回填优先于退款。

## 7. 发布与回滚

发布顺序必须是：先部署独立文档转换服务并验收 -> 发布 Web 画布构建并强制刷新验证 -> 在测试账户完成一条最小 RH 视频任务的提交、轮询、恢复和结果落盘 -> 再扩大生产流量。

任何一步失败都停止后续发布：

- 文档服务失败：回滚 Nginx 新增 location 和 converter 容器，不影响 NewAPI。
- Web 画布失败：回滚 Pages/Web 构建；不要删除用户 OPFS 或 `.jccanvas`。
- RH 状态失败：停止自动退款和新模型流量；保留 `submitted/recoverable` 任务及原始 ID，使用 rh-adapter 查询恢复，不删除上游任务。
