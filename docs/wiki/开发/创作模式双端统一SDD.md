> 目标：把已经在 Web 运行的“直连模型 + Skill + 当前项目工具 + 创作面板”定义为独立的创作模式，并适配到 Desktop。创作模式升级一次，Web 与 Desktop 同时获得业务能力。

## 1. 产品定义

模式入口固定为：`文`、`武`、`创`。

| 模式 | 运行方式 | 用户任务 |
|---|---|---|
| 文 | 现有 OpenCode plan | 对话、分析、写作 |
| 武 | 现有 OpenCode build | 编程、本机工具、复杂 Agent |
| 创 | 直连模型运行时 | Skill、Wiki、项目文件、媒体生成、画布 |

创作模式不是 Web 页面复制，也不是从武模式拆 OpenCode。当前 Web 是已验证可运行的基线，不等于每一项行为都正确；以“模型优先、Skill 渐进加载、真实媒体请求、单一共享发送链路”校正后，Web 与 Desktop 必须调用同一份业务代码。Desktop 只替换本地会话存储、本地项目文件工具和 Tauri 传输，不能重新实现模型、Skill、工具循环或媒体输入能力。

### 1.1 单一业务源原则

- 以 Web 的真实请求与可见行为为基线；遇到它与本 SDD 的模型、Skill 或媒体原则冲突时，先修正 Web，再由双端调用同一份业务代码。
- 通过移动/抽取 `sendWebCloudMessage` 的既有业务代码形成一个公共发送函数；Web 与 Desktop 都调用它，禁止复制后分别维护。
- 创模式以受控终端作为一等工具：模型决定命令，韭菜盒子负责项目目录边界、确认、执行、取消、结果回传与可见记录。复用现有 Tauri 命令执行能力，不启动 OpenCode、不新增第二个 Agent 运行时。
- 不新增手写 FFmpeg/Whisper/转写/抽帧引擎、模型能力目录、通用 Adapter 类或第二套媒体协议；媒体 Skill 通过受控终端调用已安装的 FFmpeg、Whisper、Python 或 Skill 脚本。
- 每个用户可见能力必须有双端合同测试；“构建通过”或“工具单测通过”不能代替 Web/Desktop 行为一致验收。

### 1.2 模型、工具与 Skill 原则

- 模型是决策者：收到用户文本、可用媒体和可用工具后，决定直接答复还是调用工具。工具是模型的手脚，不是创作模式的发送门槛。
- 视觉能力与 function calling 是独立能力。模型同时支持两者时，同一个请求必须同时包含媒体内容和 `tools`；模型不支持工具时仅省略 `tools`，仍发送文本和媒体，绝不拒绝整次创作请求。
- 不新增模型能力目录。公共发送函数只复用现有 `supportsVision(...)` 与已有模型条目的 `toolCall` 声明，并在 Web/Desktop 的同一位置决定请求字段。
- Skill 遵循 Agent Skills 的渐进披露：先提供名称和描述目录，模型按任务匹配调用 `skill(name)` 后才读取完整 `SKILL.md`。不得把选中 Skill 的全文预注入 system prompt。
- 用户在 UI 明确选中一个 Skill，等同显式调用：公共发送函数只传递该名称并要求本轮先调用 `skill(name)`；用户未选中时，模型依据目录自行选择。两种路径均不预注入全文。
- Skill 选择器、模型可见目录与 `skill(name)` 的解析器必须使用同一份“有效注册表”。禁止出现 UI 可选、模型工具却报“不存在”的目录分叉。

### 1.3 2026-07-16 确认的 Skill 来源合同与交互修复

- App 的文、武、创三种模式只使用两类 Skill：用户本机 `~/.agents/skills` 自动扫描结果，以及应用内置 `public/skills` 自动扫描结果。不得把 OpenCode 的其他运行时目录、Claude 插件目录或手工登记名单混进选择器、模型目录或 `skill(name)`。
- Web 只使用 `public/skills` 自动扫描结果。本机 Skill 不同步到 Web；Web 也不维护另一份手工登记名单。
- `public/skills` 保持应用内置资源：不得再在启动或扫描时复制、软链或注册到用户 `~/.agents/skills`。旧版本由应用创建且带 `.seed_complete` 标记的副本可以清理；真实用户目录及其 Skill 不得修改。
- Desktop 的有效 Skill 表由“本机优先、内置补齐”合并一次生成，并同时供文/武选择器、创模式模型目录、手动选择和 `skill(name)` 读取。名称重复时使用本机版本，保证用户的本地 Skill 可完整运行；内置版本仍可在“内置”页单独浏览。
- Skill 仓库新增“内置”页：只显示名称、描述和“内置”标记；不能打开正文、编辑、删除、安装或复制。模型仍可按 `skill(name)` 正常读取内置正文和声明资源。
- 三个交互缺陷必须在共享边界修复：创模式取消要立即结束正在等待的流；OpenCode 的空同步快照不得抹掉已经显示的消息；编辑器发送必须保留用户输入的换行。后两项同样覆盖创模式，不另建分叉逻辑。

实现记录（2026-07-16）：已落实上述来源合同。Desktop 选择器、创模式目录和 `skill(name)` 统一使用“`~/.agents/skills` 本机优先 + `public/skills` 内置补齐”的单一名单；Web 仅从 `public/skills/index.json` 建立名单。启动和扫描不再复制内置 Skill 到用户目录，并只清理旧迁移写入 `.seed_complete` 的副本或确实指向内置源的旧软链接。Skill 仓库已增加只读“内置”页。创模式停止会中断 Tauri 流读取，OpenCode 空快照不会覆盖可见消息，输入换行会原样发送。

## 2. 统一运行时

```text
用户消息 + 图片/视频/文件输入 + Skill 目录 + 当前项目工具
  -> 公共创作发送函数
  -> runDirectChatCompletion
  -> 模型按需调用 skill / read / glob / grep / write / edit / terminal
  -> 当前项目工具执行并回传结果
  -> 模型继续，直到给出最终文本
```

工具执行过程不展示模型的原始思维链。每轮 function call 与结果会保存到会话，并复用现有折叠工具摘要；带工具调用轮次的临时文本不得留在最终答复气泡。最终不含工具调用的一轮只展示 `content`，可随该轮流式到达逐步呈现，不能显示 `reasoning` 或 `reasoning_content`。

工具过程继续使用现有工具调用和结果回调，不新建全局事件总线。运行中显示当前工具，完成后折叠为“已完成 N 步”；事件不是模型思维链，不持久化隐藏推理文本。

复用现有：

- `src/runtime/direct/directEngine.ts`：工具循环、取消、12 轮上限；
- `src/utils/directMessageBuilder.ts`：模型消息拼装；
- `src/utils/skillContentResolver.ts`：内置 Skill 目录、按需加载及资源读取；
- `src/stores/mediaTaskStore.ts`、`src/components/creation/`、`src/components/canvas/`：媒体任务、项目媒体和画布。

相对路径仍以当前项目为根并拒绝 `..`；Desktop 用户批准本次创作任务后，模型也可读、查、写、改获准的绝对路径（包括系统临时目录），终端也可在该绝对目录运行。上传附件和已加载 Skill 脚本仍分别通过受控引用/资源清单提供，不能绕过 Skill 已声明资源限制。首个真实电脑操作由用户选择“拒绝 / 允许 / 始终允许”；“始终允许”覆盖本任务后续文件与终端工具，重新发送后自动失效。

## 3. 双端适配

| 能力 | Web 创作模式 | Desktop 创作模式 |
|---|---|---|
| 项目真源 | IndexedDB + OPFS | 用户选择的本地项目文件夹 |
| 项目工具 | `webProjectFiles` | 现有 Tauri `dev_*` 文件 IPC，root 固定为当前项目目录 |
| 图片读取 | OPFS Blob data URL | Tauri `dev_read_file` 返回的本地文件 data URL |
| 模型请求 | 同一公共发送函数 | 同一公共发送函数；Tauri 仅替换请求传输 |
| 模型能力 | 公共函数复用既有能力判断 | 调用同一判断逻辑；Desktop 本地模型是允许的平台扩展 |
| 图片/文件/视频输入 | 以实测 Web 请求合同为准 | 同一消息构造；本地附件仅转换成该合同中的同一形式 |
| 媒体与画布 | 当前 Web 实现 | 当前 Desktop 实现 |

Desktop 不向模型暴露 `dev_*` 名称。模型看到的工具名必须与 Web 一致：`skill`、`read`、`glob`、`grep`、`write`、`edit`；Desktop 另提供明确标记为本机能力的 `terminal`，Web 不伪装成可执行。

Desktop 工具实现规则：

1. `read`：目录列举、UTF-8 文本分页；项目媒体和用户批准的绝对路径图片均按既有工具合同返回 data URL 或摘要。
2. `glob`：项目内走现有 `dev_list_files`；用户批准的绝对目录走对应外部递归清单，再使用同一 glob 合同过滤。
3. `grep`：项目内走 `dev_list_files`/`dev_read_file`；用户批准的绝对目录走对应外部文件接口，最多返回 1000 项。
4. `write`：相对路径调用 `dev_write_file`；获准绝对路径调用外部写入接口，均自动建立父目录。
5. `edit`：相对路径调用 `dev_replace_in_file`；获准绝对路径调用外部精确替换接口，默认替换一处。
6. `skill`：公共 `public/skills` 是内置基线；Desktop 还必须复用既有 `skillsManageStore` 的已注册本机 Skill 扫描、详情和受限文件读取入口。选择器、目录 prompt 和 `skill(name)` 必须由同一有效注册表生成，不复制或另扫一份 Skill 仓库。
7. `terminal`：复用 Tauri `dev_run_command` 的超时、取消与 stdout/stderr 回传能力；默认项目目录运行，获准时也可使用绝对工作目录。首个真实电脑操作显示审批，用户选择“始终允许”后本任务后续文件与终端工具直接执行，不调用 OpenCode。模型调用记录必须保存在创作会话的工具轨迹中。

不增加 Office 或新的 FFmpeg/Whisper 实现到创作模式。已注册媒体 Skill 可以通过受控终端使用用户设备已安装的 FFmpeg、Whisper、Python 或其自带脚本，不能退回武模式。用户选择“始终允许”后，模型可以读取它写入的临时文件；附件令牌的真实缓存路径仍不写入模型可见的命令输出。原始视频是否进入模型不能由“模型支持视频”或“Web 得到回答”推断，必须由实际请求体证明；在证明前，当前代码仅承诺向模型发送视频元数据摘要。

## 4. 会话与界面

1. Desktop 在现有 ChatPanel 模式按钮中增加 `创`；Web 直接使用创作模式，不显示文/武选择。
2. 创作会话使用独立 `creativeSessionStore`，以现有 IndexedDB `conversations/messages` 保存，记录 `scopeKey: "creative"` 和当前项目标识。
3. 创作会话绝不创建 `ses_*`、绝不写入 `openCodeSyncStore`、绝不继承文/武会话权限。
4. FileTree 的历史列表在创作模式显示当前项目的创作会话；切回文/武后继续显示原 OpenCode 会话。
5. 创作模式发送前必须已有项目：Desktop 要有 `projectStore.projectDir`，Web 要有 `projectStore.webProjectId`。缺少项目时不发送，提示先选择项目。
6. 现有创作面板、媒体历史、画布和项目文件树不复制 UI；创作模式只把对话、Skill 和项目工具接到它们已有的数据边界。

交互修正：工具状态改由执行器明确写入 `succeeded | failed | cancelled`，摘要不再扫描结果正文中的“失败/error”等字样，因此 Skill 教程内容不会被误报为执行失败；真正失败时在折叠详情显示原始错误。创模式不展示原始思维链，也不使用逐帧揭示器：工具执行期间以折叠摘要呈现，工具循环完成后一次性提交最终助手文本。

稳定性修正：创模式首次发送先将用户消息与响应式 assistant 占位消息持久化，再激活会话，避免会话水合读到空记录并覆盖可见消息；回调始终更新数组内的 Vue proxy。未激活会话在创建时绑定项目归属，切项目、切创模式时同步清空共享 OpenCode 消息并进入水合锁；同会话重发使用运行 token，旧请求不能释放新请求的水合锁。已存在创模式入口守卫，但异步 `await` 返回后仍可能继续启动 OpenCode；该竞态列入 Task D，修复前不得宣称完全隔离。

旧 focused 测试、类型检查、Rust 检查和构建均通过，但缺少双端请求体、多模态输入、模型能力和实时工具事件的合同测试，因此不能据此宣称 Desktop 与 Web 已完全一致。

## 9. 2026-07-16 长视频创作稳定性修复（本轮执行）

2. **正常长任务不在 12 轮停止**：移除创模式默认 12 轮低上限，改为 64 轮最后熔断。它不是正常工作限制，只防模型异常循环；同一失败工具调用紧接着原样重复时直接回传失败，模型必须换办法。单轮可并行的多个工具调用仍只计一轮。
3. **不把创作输出压成 4096 tokens**：创模式请求不再传固定 `max_tokens: 4096`，使用模型/渠道自己的输出能力。流解析必须保留并展示 `finish_reason`，以便区分模型正常长度结束与网络中断。
4. **SSE 不使用 120 秒总超时**：流式聊天保留连接超时和用户的停止操作，但不再用总时长切断已持续输出的长任务。HTTP 200 后的流错误保留 request-id 供网关查询。
5. **最终正文断流自动续写**：仅当模型尚未返回工具调用时，首次 SSE 断流自动从已输出正文末尾续写一次；已经完成的工具绝不重新执行。第二次仍断流时保留已显示正文和真实错误，交给用户继续或重试。
6. **写入工作区以工具结果为准**：模型不能只说“我会写入”。要求保存时必须发出 `write`/`edit` 工具调用并收到成功结果；流在工具调用前断开时，不得显示成权限或写入失败。

验收：文本路径不会触发“当前对话附件不存在”；24 镜头视频不因 12 轮停止；长输出不含 4096 或 120 秒产品硬截断；最终正文首次流断自动续写且不重复工具；写入工作区后工具轨迹出现成功 `write` 或 `edit`。
