# 抖音小程序可行性评估与实施计划

> 目标：将韭菜盒子 Studio 核心功能（AI 对话 + 媒体生成）以**轻量版**形式搬到抖音小程序，服务现有用户。
> 状态：调研评估阶段，需确认技术路线后执行。

---

## 一、技术路线评估

### 方案 A：uni-app 编译（推荐）
- **原理**：用 uni-app（Vue 3 语法）重写核心功能，编译到 `mp-toutiao` 平台
- **优点**：Vue 3 语法熟悉、一套代码多端复用、社区成熟
- **缺点**：需重写 UI 组件和渲染层，部分 DOM 依赖包无法使用
- **适合**：需要长期维护、未来可能扩展微信/支付宝

### 方案 B：纯原生 TTML/TTSS/JS 开发
- **原理**：从零用抖音小程序原生技术栈开发
- **优点**：性能最佳、完全可控、审核通过率高
- **缺点**：无代码复用、开发成本高、双端维护
- **适合**：功能极简、追求性能极致

### 方案 C：WebView H5 嵌入（不推荐）
- **原理**：在小程序 web-view 组件中加载 H5 页面
- **优点**：Web 代码几乎不改、最快上线
- **缺点**：web-view 限制多（无登录态、无支付、无分享、性能差）、体验割裂
- **适合**：临时过渡方案（不推荐）

**结论：推荐 uni-app 方案**。理由：Vue 3 语法与现有代码重合度高，编译产物质量好，未来可扩展其他端。

## 二、技术挑战与解决方案

### 🔴 核心挑战

| 挑战 | 原因 | 解决方案 |
|------|------|----------|
| **无 `window.fetch`** | 小程序使用 `tt.request()` | 统一封装 `request()` 适配层，替换所有 fetch 调用 |
| **无 `ReadableStream`** | SSE 流式无法直接用 | 方案 1：改用 polling 轮询；方案 2：改用 WebSocket 代理 |
| **无 DOM/BOM** | Tiptap、highlight.js、KaTeX、Mermaid、VueFlow 全不能用 | 统一替换为小程序原生组件或纯文本渲染 |
| **无 IndexedDB** | 存储层需替换 | 使用 `tt.setStorageSync` / `tt.getStorageSync` + 云存储 |
| **包大小 4MB 限制** | 正式包上限 | 代码拆分、按需加载、资源压缩 |
| **代码审核** | 双重审核（代码+内容） | 提前准备资质材料，选对类目 |
| **Tauri 本地能力** | 文件系统、浏览器、Shell 不可用 | 裁剪为纯云端 AI 对话 + 媒体生成 |

### 🟡 渲染层替代方案

| 现有组件 | 小程序替代方案 |
|----------|---------------|
| ChatGPT 对话气泡 | 原生 `view` + `text` + `image` + `scroll-view` 实现 |
| Markdown 渲染（marked+DOMPurify） | 服务端渲染为 HTML → 小程序 `rich-text` 组件 |
| 代码高亮（highlight.js） | 服务端预处理，返回纯文本或染色 HTML |
| 数学公式（KaTeX） | 服务端渲染为图片 URL |
| Mermaid 图表 | 服务端渲染为图片 URL |
| 富文本编辑器（Tiptap） | 原生 `textarea` + 简易 toolbar |
| 创作画廊（GalleryCard） | 原生 `view` + `image` + `swiper` |
| 媒体生成结果 | 原生图片/视频组件 |

## 五、API 调用适配

现有 Web 代码所有 `fetch()` 调用需要替换为 uni-app 的 `uni.request()`。

### chatCloud.ts 改造（核心）

```typescript
// 现有 Web:
const response = await fetch(url, { method: 'POST', body, headers, signal })

// 改造为 uni-app:
const [err, resp] = await uni.request({
  url,
  method: 'POST',
  data: body,
  header: headers,
  responseType: 'text',
})
```

### SSE 流式处理（关键难点）

抖音小程序不支持 `ReadableStream`。替代方案：

**方案 1：Polling（推荐 MVP）**
```
用户发送请求 → 服务端创建任务 → 返回 taskId → 前端每 1s polling 结果
```
- 优点：实现简单，抖音小程序原生支持
- 缺点：实时性略差（1-2s 延迟）、额外的服务器开销
- 适用：MVP 阶段快速上线

**方案 2：WebSocket 代理（推荐正式版）**
```
WebSocket 连接 proxy → proxy 转发 SSE → proxy 逐帧推送 WS 消息
```
- 优点：接近实时的流式体验
- 缺点：需要额外部署 WS 代理服务
- 适用：正式发布、追求体验

## 六、存储方案

| 数据 | 方案 |
|------|------|
| 用户配置（apiKey、model、theme） | `uni.setStorageSync`（小程序本地缓存） |
| 会话历史 | `uni.setStorageSync` 存列表 + 选中的对话详情 |
| API Token | `tt.login()` 换取 code → 服务端换取 session_key → 加密存储 |
| 媒体生成记录 | 云存储（图片/视频 URL）+ 本地索引 |

**Token 安全性**：
- 小程序使用 `tt.login()` 获取 code
- 服务端用 code 换取 session_key（抖音开放平台）
- 服务端托管 NewAPI Token 加密返回
- 前端缓存加密 Token 到 storage（非明文）

## 九、备选方案：不做原生小程序，改用 Web 移动端 PWA

如果用户量不大、不想投入小程序审核成本，替代方案：

- **响应式 Web 增强**：当前 Web 版本已有移动端适配（768px 断点），可进一步优化移动端体验
- **PWA（渐进式 Web 应用）**：添加到手机桌面，获得类似原生体验
- **优点**：无审核、即时更新、成本极低
- **缺点**：无抖音流量入口、无支付、无原生能力

## 十、决策点

在开始实施前，需要确认：

1. **技术路线**：uni-app vs 原生 TTML/TTSS/JS
2. **SSE 方案**：Polling（快速上线）vs WebSocket 代理（体验更好）
3. **认证方式**：小程序授权登录 vs 现有 NewAPI Token 直接使用
4. **是否需要支付**（增加开发周期和审核复杂度）
5. **是否有现成的后端服务**承接 Token 托管、流式代理等
