# 编辑区优化升级建议

## 问题一：导出功能不好用

### 1.1 PDF 导出最严重 — 请优先修复

当前实现：`EditorPanel.vue:1229` 的 `exportAsRealPDF()` 使用 `window.open('', '_blank')` 打开新窗口 + `window.print()` 依赖用户手动"另存为 PDF"。

| 问题 | 说明 |
|------|------|
| Tauri WebView 中 `window.open` 被拦截 | 符合已知问题（CLAUDE.md §9.2），`window.open()` 无效，需用 `openExternal()` |
| 弹窗拦截导致静默失败 | PDF 按钮点了没反应，无任何用户反馈 |
| 分片导出为 PDF 产生 N 个冗余打印窗口 | `performChunkedExport` 代码自身注释承认 "PDF chunking not meaningful"，但循环仍为 PDF 调 N 次 `exportAsRealPDF` |
| 无法直接生成 PDF 文件 | 对比 DOCX 有完整 OOXML 生成，PDF 只有打印兜底 |

**建议方案**（两种选一，推荐方案 A）：

- **方案 A（推荐）**：用 `html2canvas` + `jspdf` 或 `pdf-lib` 直接生成 PDF 二进制 → `saveGeneratedFile()` 走 Tauri 保存对话框。彻底解决弹窗和打印依赖。
- **方案 B（轻量）**：保持现有打印流程，但改为 `openExternal()` + 一个包含 CSS 打印样式的隐藏 iframe 触发系统打印。修复弹窗拦截和分片冗余问题。

### 1.2 导出路径冗余

| 问题 | 位置 |
|------|------|
| `exportDoc('docx')` 和 `confirmExportOptions()` 的 DOCX 分支有两条相似但不同的路径 | `EditorPanel.vue:948` vs `:1118` |
| `exportDoc()` 对 PDF 有死代码 | 第 986 行的 `import('editorExport')` 在 format='pdf' 时永远不会执行到（979 行提前 return） |

**建议**：统一导出入口，消除 `confirmExportOptions` 的特殊分支。所有格式走 `exportDocument(format)` 统一路由。

### 1.3 导出文件路径和状态不一致

| 问题 | 说明 |
|------|------|
| `saveGeneratedFile` 在浏览器模式返回 `'downloaded'`，Tauri 模式返回 `'saved'` + `path` | `editorExport.ts:165` 没有对 `'downloaded'` 做区分处理，直接当 `'success'`，导致诊断报告 missing path |
| `openLastExportedFile()` 使用 `file://` 协议 | `EditorPanel.vue:1021` — 跨平台不可靠，Tauri WebView 可能不识别 |

**建议**：明确 `ExportResult.status` 枚举统一为 `'success' | 'cancelled' | 'error'`（去掉 `'downloaded'`），浏览器模式也设 path 为 `''`。`openLastExportedFile` 改用 Tauri `open` 命令或 `revealItemInDir`。

### 1.4 手写 DOCX/ZIP 脆弱性

当前 `localDocxV2.ts` 手搓 OOXML XML + 手写 ZIP CRC32，无依赖库。

**建议（远期）**：引入 `docx` (npm) 或 `jszip` 取代手写。当前如果跑得通可暂缓，但需注意图片嵌入和复杂表格的边界情况。

## 问题二：对话区 ↔ 编辑区联动不足

### 2.1 "放入编辑区" 改进

当前 `MessageBubble.vue:524` 的 `putIntoEditor()` 已存在，但有两个明显问题：

| 问题 | 说明 |
|------|------|
| **始终替换内容** | `EditorPanel.vue:376` 监听 `import-to-editor` 后直接 `setContent()` 替换整个文档，用户正在编辑的内容被覆盖 |
| **显示阈值太保守** | `showImportBtn` 要求 `compact >= 900 字符` 或 `>= 420 + 有结构`。短代码片段（如 200 字的 SQL / Python 脚本）不显示按钮 |

**建议**：

- **新增"追加到编辑区"（append 模式）**：`import-to-editor` 事件增加 `mode: 'replace' | 'append'` 参数。MessageBubble 加两个按钮："替换编辑区" 和 "追加到编辑区"。EditorPanel 对 append 模式在文档末尾插入新内容。
- **降低显示阈值**：检测内容包含代码块、表格、列表等结构时，300 字符以上即可显示按钮；或改为始终显示（短内容可当"新建笔记"用）。
- **加入代码块级导入**：渲染代码块时增加"导入编辑区"悬浮按钮（copy 按钮旁边），只导入该代码块内容到编辑器。

### 2.2 ReviewPanel → 编辑区联动

ReviewPanel（`src/components/chat/ReviewPanel.vue`）当前是**纯展示**，284 行代码无任何编辑入口。

| 缺失功能 | 说明 |
|----------|------|
| 点击 diff 文件打开到编辑器 | ReviewPanel 的 diff 文件名没有链接，无法在 Tiptap 中打开该文件内容 |
| 应用 patch 到编辑器 | diff patch 是纯文本，无"应用更改"或"查看差异在编辑器"功能 |
| 编辑 diff 内容后保存 | diff 不可交互，用户看到代码变更但无法直接修改 |

**建议**：

- **diff 文件名可点击**：点击 ReviewPanel 中的文件路径 → 若文件已存在于 `useFileStore`，发射 `open-in-editor` + `switch-panel('editor')`；若为新文件，调用 `import-to-editor` 模式创建
- **新增"在编辑器中查看差异"按钮**：在每条 diff 记录上加一个按钮，将 patch 内容转换为 Tiptap 对比视图（或并排显示 old/new）
- **新增"应用此变更"按钮**：将 patch 应用到编辑器当前内容（先提示用户确认，防止误覆盖）

### 2.3 编辑区 → 对话区反向链路

当前是单向的（对话→编辑），缺少编辑区内容发回对话的路径。仅有的 `append-chat-input` 是用于追加文本到输入框。

**建议**：

- 编辑器工具栏加"发送到对话"按钮：将当前编辑器内容作为新对话消息发送（或粘贴到输入框待发送）
- 编辑器选中文本右键菜单加"发送到对话"选项

## 优先级排序

| 优先级 | 改进项 | 影响面 |
|:---:|--------|--------|
| P0 | **PDF 导出修复**（方案 A） | 核心功能不可用 |
| P0 | **ReviewPanel 文件可点击打开到编辑器** | 用户无法对 diff 做任何操作 |
| P1 | "放入编辑区"支持 append 模式 + 降低阈值 | 提升编辑效率 |
| P1 | 统一导出入口（消除冗余路径） | 代码维护性 |
| P2 | 导出状态统一（`downloaded`→`success`+`path`） | 诊断准确性 |
| P2 | 编辑区→对话反向链路 | 工作流闭环 |
| P3 | 代码块级导入编辑区 | 精细编辑 |
| P3 | DOCX/ZIP 引入依赖库 | 长期稳定性 |

## 验证方式

1. **PDF 导出**：桌面端点击导出→PDF，应弹出 Tauri 保存对话框（不是打印对话框），生成有效 PDF
2. **ReviewPanel 联动**：OpenCode 会话产生 diff 后→点击文件→编辑器打开该文件内容
3. **放入编辑区**：短消息（<500 字含代码块）显示按钮；点击"追加"→编辑器末尾插入新内容
4. **类型检查**：`pnpm exec vue-tsc -b`
5. **构建**：`pnpm exec vite build`
