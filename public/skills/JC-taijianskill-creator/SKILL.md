# JC-taijianskill-creator（轻量版）

用于创建纯文本 Skill 并迭代优化的 Skill。

> **能力边界**：这是纯文本环境的 Skill 创建器。你**没有**脚本执行、子 Agent 调度、文件系统写入、浏览器打开等能力。但你拥有顶级的 LLM 推理能力——用对话模拟替代脚本，用思维链替代子 Agent，用文本输出替代文件操作。

Skill 创建的高层流程：

- 明确 Skill 要做什么，大致怎么做
- 撰写 Skill 草稿
- 创建几个测试 prompt，在对话中模拟"有 Skill"和"无 Skill"的对比
- 帮用户评估结果：定性（好不好）和定量（断言通过率）
- 根据用户反馈重写 Skill
- 重复直到满意
- （可选）优化 Skill description 以提升触发准确率

你的职责是判断用户处于哪个阶段，然后推进到下一阶段。

## 与用户沟通

Skill 创建器可能被各种技术背景的人使用。请根据上下文判断沟通方式：

- "评估"和"基准测试"——可以说，但注意对方是否理解
- "JSON"和"断言"——需要用户有明显技术背景才直接使用
- 不确定时，简短解释术语
- 默认用中文沟通，技术术语可保留英文

## 创建 Skill

### 捕捉意图

从理解用户需求开始。当前对话可能已经包含用户想捕捉的工作流。

1. 这个 Skill 要让模型能做什么？
2. 什么时候触发？（用户会说什么、在什么场景下）
3. 期望的输出格式是什么？
4. 要不要设置测试用例？有客观可验证输出的 Skill（格式转换、数据提取、代码生成、固定步骤）适合测试用例。主观输出的 Skill（写作风格、艺术）通常不需要。根据 Skill 类型推荐合适的默认方案，但让用户决定。

### 访谈与研究

主动提问边界条件、输入/输出格式、示例、成功标准。准备好上下文再写测试 prompt。

用你的知识搜索类似 Skill、最佳实践、领域规范。准备好信息以减少用户负担。

### 撰写 SKILL.md

基于访谈填充以下组件：

- **name**: Skill 标识符（kebab-case，如 `xiaohongshu-copywriter`）
- **description**: 何时触发、做什么。这是主要触发机制——既要描述功能，也要描述触发场景。所有"何时使用"的信息放这里，不放正文。**注意**：模型倾向于不触发 Skill，所以 description 要"推"一点。例如，不是"构建仪表盘显示内部数据"，而是"构建仪表盘显示内部数据。当用户提到仪表盘、数据可视化、内部指标、或想展示任何公司数据时使用此 Skill——即使用户没有明确说'仪表盘'。"
- **compatibility**: 需要的工具和依赖（可选，很少用到）
- **正文**: 角色定位、核心能力、工作流程、输出格式、约束条件、参考示例

### Skill 写作指南

#### Skill 的解剖结构

```
skill-name/
├── SKILL.md (必需)
│   ├── YAML frontmatter (name, description 必需)
│   └── Markdown 指令
└── 附加资源 (可选)
    ├── scripts/    - 可执行代码（纯文本环境不可用）
    ├── references/ - 按需加载的文档
    └── assets/     - 输出中使用的文件（模板、图标、字体）
```

#### 渐进式加载

Skill 使用三级加载系统：
1. **元数据**（name + description）——始终在上下文中（~100 词）
2. **SKILL.md 正文**——Skill 触发时加载（理想 < 500 行）
3. **附加资源**——按需加载（无限制）

**关键模式：**
- SKILL.md 保持 500 行以内；接近此限制时增加层级，给出清晰指引
- 从 SKILL.md 明确引用 references 文件，说明何时阅读
- 大型 reference 文件（>300 行）加目录

#### 写作原则

使用祈使句。解释**为什么**重要，而不是大写 MUST 强制。运用心智理论，让 Skill 通用而不局限于特例。先写草稿，再以新人眼光审视改进。

**定义输出格式**——可以这样写：
```markdown
## 报告结构
始终使用以下模板：
# [标题]
## 执行摘要
## 关键发现
## 建议
```

**示例模式**——加入示例很有用：
```markdown
## 提交信息格式
**示例 1：**
输入：添加了 JWT 用户认证
输出：feat(auth): implement JWT-based authentication
```

### 测试用例

写完 Skill 草稿后，提出 2-3 个真实测试 prompt——真实用户会说的话。和用户确认后再运行。

测试用例格式（参考 `references/schemas.md`）：
```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "用户的测试 prompt",
      "expected_output": "期望输出描述",
      "expectations": []
    }
  ]
}
```

## 运行和评估测试用例（对话模拟版）

> 以下流程是官方 skill-creator 完整测试流程的**对话模拟版本**。原版依赖子 Agent 并行执行 + 文件系统 + eval-viewer 浏览器界面。本版本用 LLM 能力替代每一步。

这是连续流程——不要在中途停下来。

### 第1步：同时模拟有 Skill 和无 Skill 的对比

对每个测试用例，在对话中展示两个版本的回答：

**有 Skill 的回答**：你按照 SKILL.md 指令执行测试 prompt，展示输出。
**无 Skill 的回答**：你忽略 SKILL.md 指令，用通用能力执行相同 prompt，展示输出。

**两个版本在同一条消息中展示**，让用户直观对比差异。为每个测试用例记录测试元数据：

```json
{
  "eval_id": 0,
  "eval_name": "描述性名称",
  "prompt": "用户的测试 prompt",
  "assertions": []
}
```

### 第2步：展示对比结果时，同步起草断言

不要等用户反馈——展示对比结果的同时，为每个测试用例起草定量断言。

好的断言是可客观验证的，有描述性名称——让人一目了然每个断言在检查什么。主观 Skill（写作风格、设计品质）更适合定性评估——不要给需要人类判断的东西强制加断言。

展示给用户：
- 📋 测试 prompt
- ❌ 无 Skill 的输出
- ✅ 有 Skill 的输出
- 📊 断言草案（通过/失败 + 证据）

### 第3步：评分和汇总

用户审阅完成后：

1. **逐断言评分**：对每个测试用例的每个断言，判断 PASS/FAIL，提供证据
2. **汇总统计**：计算通过率和关键指标
3. **分析**：找出模式——哪些断言总是通过（可能太弱）、哪些总是失败（Skill 需要改进）、哪些不稳定

评分输出格式（参考 `references/schemas.md` 的 grading.json）：
```json
{
  "expectations": [
    {"text": "断言内容", "passed": true, "evidence": "在输出中找到的证据"}
  ],
  "summary": {"passed": 2, "failed": 1, "total": 3, "pass_rate": 0.67}
}
```

### 第4步：展示结果并等待反馈

用自然语言总结：「我整理了测试结果。这里每个测试用例的对比——左边是没 Skill 的输出，右边是用了 Skill 的输出。下面是定量评分。你觉得哪里需要改进？」

### 第5步：读取反馈并改进

用户反馈后，读取具体意见。空反馈 = 用户觉得没问题。聚焦有具体意见的测试用例。

## 改进 Skill

这是循环的核心。你已经跑了测试，用户审阅了结果，现在根据反馈改进。

### 改进思路

1. **从反馈中泛化**。我们在创建能用一百万次的 Skill——如果只适用于这 2-3 个测试用例，那就没用了。不要加过度拟合的修补或压迫性的 MUST，如果某个问题顽固，尝试用不同的比喻、推荐不同的工作模式。
2. **保持 prompt 精简**。去掉不产生价值的东西。读对比输出——如果 Skill 让模型浪费大量时间做无意义的事，删掉那部分指令。
3. **解释为什么**。努力解释每个要求背后的原因。今天的 LLM 很聪明，有良好的心智理论。如果你发现自己在大写"永远"或"绝对不要"，或用超级刚性的结构——那是黄牌警告。重新组织，解释推理，让模型理解为什么这件事重要。
4. **从测试用例中找重复工作**。如果所有测试用例中模型都独立写了类似的辅助步骤，那说明 Skill 应该内置那个步骤。

### 迭代循环

改进后：
1. 应用改进到 Skill
2. 重新跑所有测试用例（新的一轮对比）
3. 展示新结果
4. 等待用户审阅
5. 读取新反馈，再次改进

直到：
- 用户说满意
- 反馈全是空的（看起来都好了）
- 没有有意义的进展

## Description 优化

description 是 SKILL.md frontmatter 中决定模型是否触发 Skill 的核心字段。创建或改进 Skill 后，提供优化 description 的选项。

详细流程见 `references/description-optimization.md`。摘要：

1. **生成 20 个触发测试查询**：should-trigger 和 should-not-trigger 各半
2. **用户审查**：展示测试集确认
3. **对话中优化**：模拟触发测试 → 统计分数 → 分析失败案例 → 改进 → 重复 3-5 轮
4. **应用结果**：取最佳 description，更新 SKILL.md，展示 before/after

**触发机制要点**：模型只对"难以独立完成"的任务查阅 Skill。简单的单步查询即使 description 匹配也可能不触发。所以测试查询要足够实质性。

## Claude.ai / 纯文本环境专属指南

你正处于纯文本对话环境（如 Web 版韭菜盒子）。这恰好是 Claude.ai 模式——核心流程相同（草稿 → 测试 → 审阅 → 改进 → 重复），但因为没有子 Agent 和文件系统，机制有所变化：

**运行测试用例**：没有子 Agent = 没有并行执行。对每个测试用例，读取 SKILL.md 指令，然后在对话中执行测试 prompt。一次一个。跳过无 Skill 基线对比——直接用 Skill 完成任务。

**审阅结果**：无法打开浏览器。直接在对话中展示结果。对每个测试用例，展示 prompt 和输出。询问内联反馈：「这个看起来怎么样？有什么要改的吗？」

**量化评分**：跳过量化基准测试——它依赖基线对比，没有子 Agent 就没有意义。聚焦用户的定性反馈。

**迭代循环**：同上——改进 Skill，重新跑测试，请求反馈——只是中间没有浏览器审阅步骤。

**Description 优化**：原版需要 `claude` CLI 工具（`claude -p`），只有 Claude Code 中可用。在纯文本环境中，对话模拟替代（见 `references/description-optimization.md`）。

**打包**：纯文本环境中，直接输出最终 SKILL.md 的 markdown 代码块。用户复制后在 Skill 选择器中使用。

**更新已有 Skill**：
- **保留原始名称**。Skill 目录名和 `name` frontmatter 字段保持不变。
- 先复制到可写位置再编辑。
