# 通用转换协议

SKILL.md 阶段 1 未匹配任何已知类型时读本文件。兜底协议。

## 协议流程

### 第 1 步：分析内容特征

向用户展示分析结果：

```
🔍 内容特征分析

- 总字数：X 字
- 结构特征：[章节/段落/无结构/对话交替/...]
- 主要内容：[主题概括，200 字]
- 角色识别：[列出疑似角色名]
- 特殊元素：[世界观/设定/时间线/...]

A 分析准确，继续设计架构
B 有误，我补充信息
```

### 第 2 步：设计 Wiki 架构

基于特征分析，设计 Wiki 架构示意图：

```
⚠️ 此类型暂无标准模板，以下为根据内容特征设计的架构

📋 建议 Wiki 结构：

wiki/
├── {目录1}/
│   ├── _index.md
│   └── {文件名}.md     ← 内容说明
├── {目录2}/
│   └── ...
└── index.md

A 就按这个来，开始写
B 我想调整（说想加什么/减什么/改什么）
C 换个思路重新设计
```

### 第 3 步：讨论迭代

用户说「调整」→ 根据反馈修改架构 → 再展示 → 再确认。循环直到用户满意。

### 第 4 步：执行写入

确认后逐文件写入 `wiki/`，补双链，刷新 index。

## 架构设计原则

1. **角色、世界、剧情** 是最常用的 3 个顶层目录，优先考虑
2. 目录名用中文，文件名用核心概念
3. 每个目录下放 `_index.md` 作索引
4. 内容多就拆分，少就合并——不为了目录好看而建空壳
5. 所有 .md 文件必须补 `[[双链]]`

## 约束

- **必须有第 2 步的架构示意图**，不能跳过直接写
- **必须有第 3 步的讨论环节**，用户确认前不写入
- 如果用户在讨论中说「随便」「你定」，给 A/B 两个选项而不是自己定
