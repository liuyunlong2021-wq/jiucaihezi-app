# 架构扩展规范

> 定义架构扩展时怎么判断结构修改与语义迁移、怎么搭建、何时交给 `jc-raw-wiki`。

## 为什么会有这个需求

wiki 刚建好时的分类往往是初始猜测——小说建库时可能只想到"角色/世界/剧情"，没料到写到中途伏笔多到该单独建账本；开发项目建库时按"SDD/开发日志/学习笔记"分，做大了才发现同一个模块（比如"文件树"）的 SDD、日志、排障经验散落在三个不同顶层目录里，越查越乱。

wiki 架构不是一次性定死的，它该跟着项目认知一起长。这个模式就是负责"长" 的那部分操作。

## 触发路径

1. **用户主动提出**：直接说"想加个xx分类""这个模块内容太散该拆开"。
2. **巡检发现**：[[jc-jian-wiki]] 的报告可能出现分类过粗或同一关键词散落多处。这是建议，不是修复指令；模型先评估新结构是否真有长期价值，影响既有内容时再向用户确认。

## 第一步：判断轻量 vs 重量

**轻量（你自己做）**：只是新建一个目录 + 占位/待摄入壳文件 + 更新 index.md，没有现成内容要搬。比如"以后伏笔都记到这个新目录"，从这一刻起生效，之前的伏笔线索留在原处不动。

**语义迁移（转交 `jc-raw-wiki`）**：需要把已经写在别处的内容重新理解、拆分或重写归档。这类工作由 `jc-raw-wiki` 按项目 Reference 的 Raw 填充标准完成，不能由本 Skill 复制 Raw 或臆造新内容。

判断依据很简单：**新分类要不要吃掉已有内容？** 不要 → 轻量，你自己搭；要 → 重量，你搭完骨架后转交。

## 第二步：和用户对齐新结构

问清楚三件事：

1. **新分类叫什么**：目录名要和现有 wiki 目录命名风格一致（中文、无空格，参考现有的"角色/世界/剧情/道具/场景"）。
2. **放什么内容**：一句话说清楚这个目录的职责边界，避免和已有目录重叠（如果"伏笔"信息本来就零散记在"剧情/悬念.md"里，问用户是要独立成目录还是继续沿用现有文件——不是所有新需求都值得开新目录，粒度过细也会增加维护负担）。
3. **要不要迁移旧内容**：决定是轻量还是重量路径。

对于开发项目这种"按模块重新切分"的大重构（比如把开发历史.md 拆成"文件树模块/工作台模块/编辑区模块"三个子目录），先展示一个拆分方案让用户确认，不要直接动手——这种重构影响全库索引，改错了代价比新增一个空目录高得多。

## 第三步：搭建结构

轻量路径，用 `scripts/apply_fix.py scaffold` 建目录 + 占位文件 + 更新 index.md：

```bash
# 预览
python scripts/apply_fix.py scaffold --wiki wiki --category 伏笔 --desc "伏笔追踪账本"

# 确认后执行
python scripts/apply_fix.py scaffold --wiki wiki --category 伏笔 --desc "伏笔追踪账本" --apply
```

脚本会：
- 建 `wiki/伏笔/` 目录 + `_index.md`
- 在 `wiki/index.md` 里追加一条新分类的索引条目（如果 index.md 存在）
- 提示是否需要同步更新 `wiki/hot.md`（如果新分类是"续写前必须知道"的信息，比如伏笔、悬念，建议加进 hot.md；如果只是归档性质，如"参考资料"，不需要）

重量路径，脚本先建好空结构，然后明确交接：

```
结构已建好：wiki/伏笔/_index.md

内容迁移需要读懂原文语义再重新归档，这部分交给 jc-raw-wiki：
"请把角色档案、剧情大纲里提到的伏笔线索，摘出来整理进 wiki/伏笔/，
按埋设章节分文件，补 [[双链]]。"
```

## 第四步：记 log

无论轻量重量，都要追加 `wiki/log.md`：

```
## [2026-07-18] 架构扩展 | 新增分类「伏笔/」，用户提议，用于追踪伏笔埋设与回收
## [2026-07-20] 架构扩展 | 开发历史.md 按模块拆分为 文件树模块/工作台模块/编辑区模块 三个子目录，巡检官建议+用户确认
```

如果结构扩展后转交 `jc-raw-wiki` 进行语义迁移，等迁移完成后再由实际执行者记录结果。
