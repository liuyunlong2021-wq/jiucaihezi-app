# 巡检清单

## 输入

已有 Wiki、命中的项目语境、`来源索引.md`、需要检查的事实页和正文。Raw 只作为来源定位后的证据读取，不复制到 Wiki。

## 检查路由

| 实际结构 | 检查内容 |
|---|---|
| 任何 Wiki | 断链、孤儿页面、重复入口、索引与事实页的明显不一致 |
| 开发项目 | 架构、开发、运维、排障、巡检报告之间的链接与当前入口一致性 |
| 小说、漫剧、短故事、电影、电视剧 | 角色、世界、剧情、场景、道具、伏笔与正文/大纲的双链和跨页事实一致性 |
| 有 `映射表.md` 的创作项目 | 额外检查旧名残留、映射撞车、命名漂移；没有映射表时不启用 |
| 广告项目 | 品牌、产品、受众、创意、脚本、分镜之间的事实和链接一致性 |
| 通用资料 | 概念、主题和来源索引之间的可追溯性 |

项目语境没有定义的目录或字段不能凭空列为错误。

## 判定原则

- 机械问题：链接指向不存在、索引指向不存在、同一映射表中的明确撞车等。报告必须带具体文件与行或明确位置。
- 语义问题：角色设定、时间线、模块边界、规则或关系冲突。报告必须列出双方来源与冲突点，不替用户选择结论。
- 架构建议：分类过粗、导航不足等是建议，不计入错误总数，也不要求立刻修改。
- 页面头部标注“历史/已归档/已替代”或位于 `归档/` 时，机械问题归入“归档卫生”；仍可报告，但不计入现行错误总数和退出码。

## 输出标准

报告先分“现行风险”和“归档卫生”。现行风险再按 `❌ 必修`、`🔻 重要`、`⚠️ 建议`、`💡 架构建议` 分组。每项包含问题、证据位置、影响、可确定的下一步和不确定性。

- 项目语境已经规定巡检报告目录时，写入该目录；否则只向用户输出，避免另建平行目录。
- 巡检只查不改，不更新 `log.md`、`hot.md`、`CLAUDE.md` 或业务事实。
- 确定性修复交给 `jc-xiu-wiki`；需要理解 Raw、补充结论或迁移内容时交给 `jc-raw-wiki`。

## 脚本出口

`scan_vault.py` 只做可重复的机械扫描：`0` 无现行机械问题，`1` 发现现行问题，`2` 参数或路径错误，`3` 无法识别 Wiki 结构。归档卫生不改变退出码；脚本输出不是完整语义巡检的替代品。
