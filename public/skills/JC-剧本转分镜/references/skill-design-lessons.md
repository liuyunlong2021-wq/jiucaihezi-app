# Skill 设计经验沉淀

> 来自 `JC-剧本转分镜` 的完整优化过程。记录可复用的设计原则，用于升级其他创作类 Skill。

---

## 一、最大化挖掘 LLM 训练数据

### 核心洞察

LLM 的训练数据里最有价值的是**结构化领域知识**——电影评论、导演访谈、拉片笔记、技法分析。直接让 LLM「按王家卫风格写」只能触发表层关联。让它「回忆王家卫在《花样年华》里怎么处理对峙戏——几镜、什么节奏、什么构图」才能触发深层知识。

### 设计模式：分类档案法

**不是**：给一个笼统的风格标签，让模型自由发挥。

**而是**：让模型按分类维度，从训练数据中提取结构化的知识档案。

```
旧：「按王家卫风格写分镜」
新：「从王家卫《花样年华》中提取场景节奏档案——
     对峙戏怎么处理？爱情戏怎么切？
     独角戏跟不跟人？武打戏碎几镜？
     → 按戏型分类，一次生成全剧复用」
```

### 通用模板

```
锁定 [领域专家/作品] → 按 [分类维度] 提取 [结构化档案]
→ 用户输入匹配分类 → 对照档案出选项 → 执行
```

适用场景：
- 写作 Skill：锁定作家 → 按文体/情节类型提取叙事手法档案
- 设计 Skill：锁定设计师 → 按项目类型提取视觉语言档案
- 代码 Skill：锁定架构师 → 按系统模块提取设计模式档案

### 关键原则

1. **分类维度要「训数据里有」**：电影评论按戏型分类、小说分析按情节类型分类——选 LLM 训练语料中自然存在的分类方式
2. **档案一次生成全剧复用**：避免重复推理，最大化上下文利用率
3. **输出要引用具体参考**：「参考《花样年华》苏丽珍在酒店房间质问周慕云那场戏」比「王家卫式留白」强 10 倍

## 二、Skill 结构设计原则

### 渐进式披露三层架构

```
skill-name/
├── SKILL.md          ← 始终加载：理念+规则+流程（<200行目标）
└── references/       ← 按需加载：模板/示例/参考数据
```

| 层 | 内容 | 规则 |
|---|------|------|
| **元数据** (name+description) | 一句话身份+触发条件 | ~50字，不含防御性措辞 |
| **SKILL.md 正文** | 理念、规则、流程、阶段指令 | <200行，不含详细模板/示例 |
| **references/** | 格式模板、长示例、参考数据 | 每个文件有明确读取时机指引 |

### 拆分判断

一个内容块该留在 SKILL.md 还是拆到 references/：

- **留在 SKILL.md**：流程步骤、规则约束、简短示例（≤3行）
- **拆到 references/**：详细模板（>10行）、多段落示例、参考数据表

每个 reference 必须在 SKILL.md 中明确写出**什么时候读**：「阶段 1.5 时读取」「写分镜时读取」

### 命名与措辞

- **Skill 名**：功能导向，不用项目代号。`JC-剧本转分镜` > `JC-manju-daoreng-fenjing`
- **描述**：只说做什么，不说「不做什么」。去掉「零 JSON」「不含 XXX」等防御性措辞
- **触发词**：覆盖用户可能用的词，不保留历史遗留词
- **规则/约束**：只写正面行为指令，少写「不允许」「严禁」（除非真的是硬约束）

### 措辞对照

| ❌ 防御性 | ✅ 正面表述 |
|-----------|-----------|
| 零 JSON 输出 | （直接不提，输出格式本身就是文本） |
| 不允许抽象风格词 | 风格必须「导演 + 作品」 |
| 不是封闭清单 | 库只作格式锚点 |
| 严禁跨镜代词 | 每镜角色首次出现用全名 |

## 三、优化迭代流程

```
1. 审计现有 Skill
   ├── 功能：每个阶段是否聚焦核心能力？有没有跑题步骤？
   ├── 结构：SKILL.md 是否 <200行？详细内容是否在 references/？
   └── 措辞：有没有防御性/历史遗留措辞？

2. 升级核心机制
   ├── 能不能用「分类档案法」替代笼统的风格指令？
   └── 分类维度是不是 LLM 训练数据里有的？

3. 重构结构
   ├── 拆出详细模板/示例到 references/
   ├── 每个 reference 加「什么时候读」
   └── 精简 SKILL.md 到 <200行

4. 清理
   ├── 删除防御性措辞
   ├── 重命名为功能导向
   └── 删除历史遗留触发词/字段
```

## 四、像素参考法（分类档案法的进化）

> 来自 `JC-角色设计提示词` 的优化过程。当文字回忆不够精准时，直接用真实参考图像素。

### 核心洞察

LLM 回忆「吉卜力风格的女战士」是模糊的。但 Pinterest 上有无数真实参考图——搜到一张，用 `view_image` 分析，提取的视觉 DNA 比任何文字描述精准 10 倍。

### 设计模式

```
人物小传 → 提取英文搜索关键词 → Pinterest/ArtStation 搜参考图
    ↓
navigate_page(pinURL) 直接打开大图 → screenshot_page 截图
    ↓
view_image 分析参考图 → 提取视觉 DNA（面部/体型/服装/配件）
    ↓
参考图 DNA vs 人物小传差异 → 逐项融合 → 输出 JSON
```

### 搜索词设计

- 角色类型（英文）+ 风格方向 + 关键特征
- 必加限定词：`character turnaround sheet`（命中多角度设定图而非单张肖像）
- 备选源：ArtStation、DeviantArt
- Pinterest `click_element` 常被注册弹窗拦截 → 改用 `navigate_page(pinURL)`

### 与分类档案法的关系

| 场景 | 用什么 |
|------|--------|
| LLM 训练数据中有结构化领域知识（如导演的戏型处理） | 分类档案法 |
| 需要精确视觉参考、LLM 文字回忆不够精准 | 像素参考法 |
| 两者可结合：先搜参考图锁定视觉方向，再提取设计档案 |

## 五、输出格式设计原则

### JSON > Markdown（下游适配）

如果产出会被下游 AI 工具消费，用 JSON：
- 机器可解析、字段可验证
- 去掉散文性描述（材质准确性大段文字 → `material` + `detail` 字段）
- 增加下游需要的字段（`onImageText` 明确图片上要渲染的文字、`closeupRequired` 布尔值驱动特写生成）

### 工程参考图 vs 肖像照

| 维度 | 肖像照（旧） | 工程参考图（新） |
|------|-----------|----------------|
| 背景 | 环境+戏剧光影 | 纯色中性灰 |
| 视角 | 单一视角 | 5视角（正/左45/右45/背/头特） |
| 配件 | 散文描述 | 独立数组 + 位置/材质/尺寸 |
| 文字 | 无 | onImageText 明确图片上渲染什么 |
| 特写 | 无 | closeups[] 独立小框+标注 |

## 六、流程设计原则

### 阶段 0.5：全剧风格确认

多角色 Skill 必须**在所有搜索之前**确认一次全局风格，避免碎片化：

- 媒介：真人/动漫/漫剧
- 题材：悬疑/爱情/奇幻/...
- 视觉风格：偏写实/偏卡通/...
- 特殊设定：兽人/机甲/赛博朋克/...

确认后所有角色共用，搜索词自动拼接。

### Wiki 联动

与 JC-duanju-shijiemoxing / JC-xiaoshuo-shijiemoxing 共用 `wiki/角色/` 目录：
- 读 `wiki/角色/角色名.md`（世界模型产出）
- 写 `wiki/角色/角色名.design.json`（本 skill 产出）
- 更新 `CLAUDE.md ## [角色设计]` 区块
