# 整理规范（.raw/ → wiki/ 的提炼方法）

> 本文件是记忆压缩官的执行手册。核心是"把零散素材提炼成该长期记住的结论，归档进结构化档案"。

## 一、提炼，不是搬运

最重要的纪律：**记"不能忘的结论"，不搬"完整原文"**。

- ❌ 错：把和用户的整段对话原样复制进角色档案。
- ✅ 对：从对话里提炼出"主角第30章会黑化，触发点是妹妹死亡"这一条结论，写进档案。

原因：`hot.md` 和档案是给大模型"快速召回"用的，塞满原始对话会稀释信息密度、撑爆上下文。压缩的价值就在"去水留干"。

`.raw/` 里的原文保留不删——提炼是从它读取，不是把它搬空。错了能回查。

## 二、信息归到哪个档案

| 素材里出现的信息 | 归档去向 |
|------------------|----------|
| 角色新性格/新关系/新身份/外貌细节/压力转折 | `wiki/角色/对应角色.md` 的「成长轨迹」段 |
| 新世界观设定、规则、势力 | `wiki/世界观/` |
| 新发生的关键事件、转折 | `wiki/时间线/` 或对应事件档案 |
| 新埋的钩子 / 回收的钩子 | `wiki/伏笔账本.md`（标埋设章、回收章） |
| 写作决策（定了某个走向、否决了某个写法） | `wiki/创作笔记.md` |
| 换皮库的新映射 | 提醒用户补进 `wiki/映射表.md`（你不直接改映射表，那是 linmo 的事） |

## 三、归档时尊重已有结构

- 角色档案已有「成长轨迹」段就追加，没有就按写作 skill 的模板结构补。
- 换皮库里，角色一律用 `映射表.md` 的**新名**，别用回范本旧名。
- 同一条信息已经归过档就不要重复写，检查是否已存在。

## 四、织双链的尺度

- 只链**真正相关**的：角色档案提到另一个角色/地点/事件，链过去。
- 别硬塞：不相关的笔记之间硬加链接只会制造噪音，反而干扰检索。
- 链接用笔记的真实文件名（`[[李四]]` 要对应 `wiki/角色/李四.md`），避免制造断链——这点和 JC-yizhixing 的断链检查对应。

## 五、增量整理，别推倒重来

- 每次只整理"上次整理之后新增的素材"，不要每次都把全库重写一遍。
- 看 `hot.md` 上次更新到第几章，从那之后的素材开始整理。
- 增量整理快、稳、不易出错；全量重写慢且容易覆盖掉用户手动改过的内容。

## 六、拿不准就问

- 某条信息是不是定稿？某个走向用户是不是真的定了？**拿不准就问用户**，别自己脑补进档案。
- 档案是"事实源"，一旦写错会污染后续所有召回。宁可多问一句。
