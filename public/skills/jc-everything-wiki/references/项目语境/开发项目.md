# 开发项目

用于代码仓库、应用、网站、插件、服务、产品文档和技术迭代材料。新项目使用 `docs/wiki/`；已有 `docs/wiki/` 或旧 `wiki/` 时接管现有位置，不自动迁移。

## 标准骨架

```text
docs/wiki/
├── CLAUDE.md
├── hot.md
├── log.md
├── 来源索引.md
├── 架构/
├── 开发/
├── 运维/
├── 排障/
├── 学习/
├── 巡检报告/
└── 归档/
```

| 位置 | 作用 |
|---|---|
| `架构/` | 稳定系统设计、模块边界、数据流和对照表 |
| `开发/` | SDD、功能迭代、技术决策和开发历史 |
| `运维/` | 部署、服务、模型、环境与运行规则 |
| `排障/` | 根因、复现、修复、兼容性和回归证据 |
| `学习/` | 可复用方法、手册和长期经验 |
| `巡检报告/` | 测试、审计、体检和待修复风险 |
| `归档/` | 已替代但需保留的历史资料 |
| `CLAUDE.md` / `hot.md` / `log.md` / `来源索引.md` | 总入口 / 当前状态 / 追加流水 / 证据映射 |

脚本类型：`dev_project`。模型可在现有结构外建立长期有价值的新分类；涉及迁移、合并或删除既有内容时必须确认。

## 原始材料

Studio 创模式会话保留在 App 会话存储或用户明确提供的导出中，不自动生成项目内副本；文/武会话保留在 OpenCode `ses_*`；VS Code 会话保留在当前项目对应的 `chatSessions/`。原始材料不复制，`来源索引.md` 只记录它们与 Wiki 的映射。

## Raw 填充标准

`jc-raw-wiki` 填充开发项目时，以上骨架是唯一落位标准。

### 绝对边界

- 会话、代码、SDD、测试输出和外部资料都是原始证据；只读引用，不复制、不移动、不删除。
- 不创建 `对话记录/`、会话摘要页、聊天转录页，也不把完整会话、整份 SDD、整段代码或完整日志写进 Wiki。
- 只有已确认、后续可复用的架构事实、开发结论、排障根因、验证结果或运行规则才能进入 Wiki；讨论和猜测保留在 Raw。

### 落位标准

| 确认内容 | 写入位置 | 必须关联 |
|---|---|---|
| 稳定架构、模块边界、数据流、对照关系 | `架构/` | 相关开发、运维或排障记录 |
| 已决 SDD、功能迭代、技术决策、阶段结论 | `开发/` | 相关架构、测试或 Git 证据 |
| 部署、服务、模型、环境、发布和运行规则 | `运维/` | 相关架构、验证证据 |
| 已定位根因、复现条件、修复与回归结论 | `排障/` | 相关开发、测试和平台证据 |
| 可复用的方法、手册和长期经验 | `学习/` | 来源文档或实践证据 |
| 测试、审计、体检风险与待修复项 | `巡检报告/` | 对应测试、审计或代码证据 |
| 已替代且确定需要保留的历史资料 | `归档/` | 替代它的现行文档 |
| 当前稳定入口和长期导航 | `CLAUDE.md` | 对应 Wiki 页面 |
| 当前状态、风险和下一步 | `hot.md` | 对应 Wiki 页面 |
| 本轮已确认事实 | `log.md` | 必要时链接到对应页面 |

没有合适位置时，只在现有结构下建立长期有价值的新分类；迁移、合并、删除或归档已有内容必须先确认。

### 写入顺序

1. 读取本轮指定内容或尚未处理的 Raw 范围，先从 `来源索引.md` 确认已处理范围。
2. 判断内容是确认事实、未验证风险还是讨论；只写前两类，不把讨论当结论。
3. 写入唯一语义位置；架构、开发、运维、排障和巡检之间存在稳定关系时建立双链和回链。
4. 在 `来源索引.md` 登记来源路径、Git/测试证据和已处理范围；只写定位信息，不摘抄 Raw。
5. 当前状态变化时更新 `hot.md`，向 `log.md` 追加本轮已确认事实。
6. 自检：Wiki 中没有完整会话、原始 SDD 副本、代码副本、断链或重复结论。
